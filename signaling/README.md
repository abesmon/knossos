# VRWeb — сигнальный сервер (standalone)

> Тот же протокол встроен в монолит домашнего сервера ([../homeserver/](../homeserver/),
> эндпоинт `/signal`; см. [docs/home-server.md](../docs/home-server.md)). Этот standalone —
> для деплоя сигналинга отдельно, без идентичности.

Минимальный WebSocket-релей для WebRTC-handshake. Группирует подключения по комнатам
(комната = страница, ключ — `PageFetcher.seed_key(url)` на клиенте) и пересылает между
пирами одной комнаты offer / answer / ICE-кандидаты. После установки p2p-соединения
позиции и чат идут напрямую (mesh) — сервер в обмене данными не участвует.

## Запуск локально

```bash
cd signaling
pip install -r requirements.txt
python server.py
```

Сервер слушает `ws://0.0.0.0:8080`. Переменные окружения: `HOST`, `PORT`.

## Запуск в Docker

```bash
cd signaling
docker build -t vrweb-signaling .
docker run -p 8080:8080 vrweb-signaling
```

В клиенте (экран настроек VRWeb) укажите адрес, например `ws://localhost:8080` или
`ws://your-host:8080`.

## Протокол

JSON-сообщения поверх WebSocket.

Клиент → сервер:

| сообщение | смысл |
|---|---|
| `{"type":"join","room":<str>,"nick":<str>[,"access_token":<str>]}` | войти в комнату |
| `{"type":"offer"\|"answer"\|"candidate","to":<peer_id>,"data":<any>}` | переслать сигнал пиру |

Сервер → клиент:

| сообщение | смысл |
|---|---|
| `{"type":"welcome","id":<peer_id>}` | присвоенный id |
| `{"type":"peers","seq":<мой join seq>,"peers":[{"id","nick","seq"},...]}` | кто уже в комнате (новичку) |
| `{"type":"peer_join","id","nick","seq"}` | вошёл новый пир |
| `{"type":"peer_leave","id"}` | пир вышел/отключился |
| `{"type":"offer"\|"answer"\|"candidate","from":<peer_id>,"data":<any>}` | релей сигнала |

**`seq` — порядковый номер входа в комнату** (монотонный счётчик на сервере, штампуется
на каждый `join`). Меньший `seq` = вошёл раньше; по нему клиенты вычисляют авторитета
комнаты (см. [docs/authority.md](../docs/authority.md)). Старый сервер без `seq` — клиент
откатывается на сравнение по `peer_id`.

**Антиглар.** Offer создаёт пир с меньшим `id` — так ровно одна сторона инициирует
соединение, без коллизий.

### Опциональное расширение: аутентификация входа

Базовый релей пускает в любую комнату по любой строке `room` и **игнорирует**
`access_token` — этот standalone работает без идентичности. Поля ниже — **необязательное
расширение**, которое использует монолит домашнего сервера для приватных персональных
пространств («дверь открыта, пока хозяин дома», см.
[docs/personal-spaces.md](../docs/personal-spaces.md)); отдельная реализация вправе их не
поддерживать:

| элемент | направление | смысл |
|---|---|---|
| `access_token` в `join` | клиент → сервер | токен сессии домашнего сервера; сервер, где сигналинг и идентичность в одном процессе, привязывает по нему WS-сессию к федеративному адресу. Базовый релей игнорирует |
| `{"type":"join_denied","room":<str>,"reason":<str>}` | сервер → клиент | вход в комнату отклонён политикой пространства (напр. `reason:"space_closed"`). Базовый релей его не шлёт |

Клиент шлёт `access_token` **только своему** домашнему серверу (когда хост сигналинга
совпадает с хостом аккаунта) — чужому/standalone-серверу токен не уходит. Получив
`join_denied`, клиент остаётся онлайн, но вне комнаты. Контракт остаётся агностичным:
ядро релея не знает, что такое токен и пространство, — за него решают внешние хуки монолита
(`auth`/`join_check`, см. [../homeserver/app.py](../homeserver/app.py)).

## Федерация (точка расширения, пока не реализована)

`room` — произвольная строка, состояние комнат изолировано. Чтобы пиры с разных
сигнальных серверов попадали в одну комнату (как в Matrix), сервер можно научить
пересылать `join` и сигналы дружественным серверам из конфигурации. peer_id тогда
нужно сделать глобально уникальными (префикс сервера). Это позволит людям хостить
свои сигнальные серверы и обходить блокировки. См. `docs/multiplayer.md`.
