# Консоль пространства (`~`) и редактирование VRWML-слоя

> Интерактивный редактор пространства «как DevTools в браузере»: по `~` открывается панель
> с HTML-репрезентацией текущего пространства. Пользователь редактирует **сцену как единую
> сущность** — слитый блок `<vrwml>` (узлы страницы + эфемерные изменения поверх), не
> задумываясь, что запечено в страницу, а что — дельта. Наружу при сохранении уходят
> объектная дельта и allowlisted конфигурация инстанса через
> [эфемерный слой](../network/ephemeral-changes.md). Во flush попадает только объектная дельта;
> конфигурация инстанса остаётся временной.

## Модель: база + оверлей

Сцена = **база** (VRWML-слой страницы, детерминирован по URL) + **оверлей** (эфемерные
дельты). Править узлы страницы напрямую нельзя — единственный сетевой механизм мутаций
это эфемерный слой, поэтому любая правка **неминуемо выражается дельтой** в нём:

| Kind оверлея | Что описывает | props |
|---|---|---|
| `vrweb-patch` | правка/удаление **узла страницы** | `{ "set": {атрибут: raw-строка}, "removed": bool }` |
| `vrweb-node` | **добавленный** узел сцены | `{ "tag": "OmniLight3D", "attrs": {атрибут: raw} }` |

Отдельно от object map хранится config корня инстанса. V1 разрешает только `mode`; у config
нет object id/bindings/TTL, и он не участвует во flush.

- id патча **детерминированный**: `vpatch:<id узла>` — анти-хайджек протокола (`add` с
  занятым id отклоняется) даром даёт «узел страницы одновременно правит один пользователь»,
  а `bindings.creator` — что чужой патч не перепишешь.
- Значения атрибутов VRWML ходят **сырыми строками** (литералы Godot: `Transform3D(...)`,
  `Vector3(...)`) — слой их не интерпретирует, сравнение точное строковое.
- `vrweb-node` c детьми — каждый элемент отдельный объект, вложенность через `parent`
  (включая якорь `page:<id узла>` — монтаж под реальный узел страницы).
- Object-протокол сохранил готовые `add`/`update`/`remove`/`reparent`; config использует
  отдельный `update-config` в том же ordered event stream.

### Адресация узлов страницы

Каждому элементу VRWML — детерминированный id (`SceneHtml.build_page_index`):
авторский атрибут `id`, иначе структурный `n<путь>` (напр. `n0-2`). Страница детерминирована
по URL ⇒ id одинаковы у всех пиров без согласования. Суб-ресурсы (`<Resource id="M1">`)
адресуемы так же — патч `size` у BoxMesh живьём меняет все меши, которые его используют.

## Консоль

Два редактора (перетаскиваемый сплит), ничего не реконструируется из геометрии:

| Область | Источник истины | Редактируемость |
|---|---|---|
| Страница (процедурный HTML, без `<vrwml>`) | хранимое дерево `HtmlNode` (`main._current_doc`) | **read-only физически** — процедурную часть править бессмысленно |
| Сцена `<vrwml>` (слитый слой) | индекс VRWML-узлов + эфемерное состояние (`SceneHtml.serialize_scene`) | **редактируемая**: узлы страницы с применёнными патчами, добавленные узлы, мировые объекты (пузыри/штрихи) |

В слитом виде у каждого элемента виден его id — по нему дифф отличает узел страницы от
эфемерного объекта и от нового элемента. Новые элементы: **PascalCase-тег** → узел сцены
(`vrweb-node`, включая `<VRWebStroke>`), **lowercase-тег** → мировой kind (`bubble`, будущие).

## Дифф → дельта (`SceneHtml.diff_scene`)

- правка атрибута узла страницы → `vrweb-patch` c **минимальным** `set` (оверрайды,
  совпавшие с базой, уходят из патча сами; пустой патч снимается целиком);
- удалённый узел страницы → tombstone (`removed: true`; поддерево скрывает патч верхнего);
- новый элемент → `add` (`vrweb-node` или мировой kind; без id — адрес генерит
  `NetworkManager.new_object_id()`);
- правка/перенос/удаление эфемерных объектов — как раньше (update-патч, reparent, remove);
- смена корневого `mode` → `update-config` (только rank 0; совпадение с базой снимает override);
- **запреты** (честная ошибка, не тихий пропуск): перемещение/смена класса узла страницы,
  удаление его базового атрибута, правка атрибутов самого блока `<vrwml>`, кроме allowlisted
  `mode`, смена kind/ttl существующего объекта.

## Материализация (EphemeralView)

- `vrweb-patch`: применяется к **живому** узлу через реестр «id → объект» (провенанс
  `VrwebBuilder.build`: элемент → узел); оригиналы свойств запоминаются для отката
  (снятие патча возвращает страницу как была). `removed` — скрыть + `PROCESS_MODE_DISABLED`
  (вместе с физикой поддерева).
- `vrweb-node`: строится `VrwebBuilder.build_element` — тем же путём, что узлы страницы.
  ⚠️ Это **расширяет принятый ClassDB-риск** ([vrwml-tags.md](../space/vrwml-tags.md)): классы теперь
  инстанцируются не только из страницы, но и из действий **пиров**. Закрывается той же
  будущей песочницей/белым списком.
- Ключи, убранные из `attrs` при update `vrweb-node`, остаются с прежними значениями —
  принятое упрощение (полный откат только у патчей).

## Цикл сохранения

Как и раньше (ack-расширение транспорта, см. [ephemeral-changes.md](../network/ephemeral-changes.md)):
на время отправки документ блокируется; все действия приняты → перерисовка из актуального
состояния; отказ/таймаут → правка **не стирается**, панель обводится красным, «Отменить»
возвращает факт. Пока блок не редактировался — живое обновление по сигналам сцены.

## Персистенция (сценарий «сохранить в страницу»)

Реализовано кнопкой **«В страницу»** (видна на страницах с `persist`-атрибутом блока
`<vrwml>`): клиент шлёт **объекты дельты** (не слитый HTML — server-authoritative merge,
присланному документу сервер не доверяет) с авторизацией по [page-persistence.md](../network/page-persistence.md),
сервер отвечает per-object (applied/rejected/deferred) и переиздаёт страницу с новым `rev`;
принятые объекты помечаются `update` `persisted_rev`. `409` (база уехала) — статус
«обновите страницу и повторите». Заперсисченные объекты **не снимаются** с оверлея —
дедупликацию против новой базы делает материализация (иначе `remove` стёр бы контент у
пиров, ещё живущих со старой базой); подробно — раздел «Дедупликация» там же.
`SceneHtml.serialize_scene(index, objects, include_world=false)` (слитый `<vrwml>` целиком)
остаётся для консоли и отладки — в сетевом флаше она не участвует.

## Управление

- `~` — открыть/закрыть (перехват в `main._input`, раньше GUI). На русской раскладке
  клавиша печатает «ё»: если фокус в текстовом поле и нажатие даёт не бэктик/тильду —
  событие уходит в поле.
- Ctrl/Cmd+S — сохранить; клик по 3D возвращает браузинг и закрывает консоль (правки
  остаются до «Отменить»/успешного сохранения); офлайн-сохранение отклоняется сразу.

## Смена `combine` / `exclusive` из консоли

> Статус: **реализовано**. `SceneHtml.diff_scene` выделяет `mode` в instance config action;
> остальные корневые атрибуты остаются неизменяемыми.

### Модель продукта и граница права

Публичный инстанс не считается неизменяемым представлением авторского замысла страницы. Это
живая общая территория: первый вошедший становится authority и получает ранг 0, а затем может
дать ранг 0 другим участникам. Для стабильного авторского вида предназначен отдельный личный
или закрытый инстанс. Поэтому смена режима **не требует opt-in страницы**.

Правило v1: конфигурацию инстанса может менять любой участник с рангом не хуже
`INSTANCE_CONFIG_RANK = 0`. Текущий authority проверяет ранг отправителя и сериализует
изменение обычным `(epoch, seq)`. Authority имеет это право потому, что автоматически получает
ранг 0, но право не привязано исключительно к роли authority и может быть делегировано.

Это не создаёт нового обещания безопасности поверх существующей модели рангов: `user_id`
самозаявлен, публичные инстансы допускают хаотичное управление, а доверенный состав участников
обеспечивается выбором приватного инстанса. При этом обычный участник без ранга 0 менять
конфигурацию не может.

### Почему это не патч обычного узла

`mode` меняет не свойство одного VRWML-узла, а состав всего представления страницы:
`combine` включает процедурный HTML-мир, `exclusive` убирает его вместе с геометрией,
коллизиями, переходами и HTML-медиа. Поэтому применять `mode` через обычный
`vrweb-patch` нельзя:

- у корня `<vrwml>` нет адресуемого живого Godot-объекта;
- простое `visible = false` оставит физику и обработку процедурного слоя;
- полный `_rebuild_world` сейчас также пересоздаёт сетевые view, VRWML-слой и телепортирует
  игрока — это слишком широкая реакция на переключатель;
- обычные объекты защищает creator policy, а корневая конфигурация — состояние **инстанса**,
  которое должно переживать уход инициатора и переход authority.

Дополнительный атрибут наподобие `mode-edit` на `<vrwml>` не вводится. Корневые атрибуты
описывают поведение всего VRWML-блока, а не ACL отдельной консольной операции. Право находится
там, где оно действует: в политике рангов инстанса.

### Протокол и состояние

Корневую конфигурацию лучше сделать отдельным first-class состоянием сцены, а не маскировать
служебным `vrweb-node`/`vrweb-patch`. Это общий механизм для разрешённых к live-edit
атрибутов корня, а не отдельная сущность только для `mode`:

```text
action: { op: "update-config", set: { "mode": "exclusive" } }
event:  { op: "update-config", set: { "mode": "exclusive" }, by: "<user_id>", epoch, seq }
snapshot.config: { attrs: { "mode": "exclusive" } }
```

Причины: фиксированная единственная запись без захвата id, отдельная ACL, отсутствие ложного
bindings и невозможность случайно отправить конфигурацию во flush как обычный объект.
Поле `by` нужно для диагностики/UI, но не даёт полномочий над конфигурацией.

V1 содержит allowlist изменяемых корневых ключей только с `mode`; значение нормализуется и
валидируется как `combine|exclusive`. Изменение остальных атрибутов корня по-прежнему честно
отклоняется. Это оставляет протокол расширяемым: будущий корневой параметр добавляется в
схему конфигурации вместе со своей валидацией, без появления отдельного `*-edit` атрибута.

Эффективные атрибуты для консоли — базовые атрибуты страницы, поверх которых наложен
`snapshot.config.attrs`. Если отредактированный `mode` совпал с базовым, override удаляется.
Для единообразия консоль может всегда сериализовать нормализованный `mode`, даже если в
исходной странице он был опущен и неявно означал `combine`.

Authority проверяет `rank_of_peer(sender) <= INSTANCE_CONFIG_RANK` (для собственного действия —
`my_rank()`), схему ключей и значения. Получатели по-прежнему принимают event/snapshot только
от вычисленного authority и повторно валидируют структуру/enum перед материализацией, но не
пытаются заново авторизовать уже сериализованное событие по своей копии рангов.

Late join получает config в snapshot; новый authority наследует тёплую копию. После
распада/слияния mesh победивший authority рассылает свой снимок по существующему правилу
ресинка. Когда комната полностью опустела, override исчезает вместе с остальным эфемерным
состоянием. Конфигурация намеренно не связывается с `base_rev`: это состояние инстанса по URL,
а `mode` валиден для обеих версий страницы; существующий инстанс продолжает жить при обновлении
базы.

Изменение `mode` участвует в том же неатомарном сохранении, что и остальные правки консоли.
Частичный отказ показывается существующим UX. Будущий атомарный batch улучшит всю консоль, но
не является условием для смены режима.

### Материализация

Процедурный мир помещён в отдельный `HtmlLayer`; VRWML, `EphemeralView` и сетевые view живут
вне него. Переключение затрагивает только `HtmlLayer`:

- `combine → exclusive`: отменить streaming `WorldGenerator`, удалить процедурный слой из
  дерева (не только скрыть), тем самым снять физику/process/media; VRWML, оверлей, другие
  игроки, соединение с комнатой и текущая поза остаются;
- `exclusive → combine`: из сохранённых `TopologyBuilder`-данных и seed заново создать
  `HtmlLayer`, затем зарегистрировать появившиеся HTML video/transition consumers;
- не вызывать навигацию, повторный fetch, `join_room` или общий `_rebuild_world`;
- если исчезнувшая HTML-коллизия была опорой игрока, скрытого телепорта нет: дальнейшее
  падение — честное следствие нового мира;
- консоль сериализует эффективный `mode`, а остальные неизменяемые root attrs берёт из базы.

### Flush и границы полномочий

V1 flush продолжает принимать только объектную дельту и **не получает** instance config.
Следовательно, участник с рангом 0 может временно изменить режим комнаты, но не способен
записать его в страницу без отдельного права origin — как и требуется. Ранги комнаты не
участвуют в авторизации flush.

Если постоянная смена понадобится, это отдельное расширение persistence: capability-ответ
явно анонсирует `accepts_config: ["mode"]`, payload несёт отдельный `config_patch`, а сервер
сам проверяет подписанта, `base_rev` и policy. Нельзя молча преобразовывать instance override
в изменение корневого атрибута при существующем flush.

### Проверки

- запрос обычного пира, rank-0 пира, локальный запрос authority и делегирование ранга 0;
- отклонение незнакомого root key и неверного значения `mode` на authority и follower;
- late join, смена authority, reconnect, split-brain → merge;
- обновление базы при живом инстансе не снимает его config override;
- оба направления во время незавершённого streaming HTML, отсутствие оставшихся collision/
  process/video nodes;
- сохранение позы, VRWML/эфемерных объектов, сетевых игроков и membership комнаты;
- частичный отказ общего batch и независимый отказ flush.

Чистые round-trip/ACL/snapshot проверки находятся в `test_scene_html.gd` и
`test_scene_changes.gd`; `net_scene_test.gd` на двух реальных WebRTC-клиентах проверяет отказ
обычному пиру и успешное делегирование rank 0.

## Текущие ограничения

UI пока не восстанавливает tombstoned node; batch не атомарен; специальные VRWeb-узлы при
live patch не пересобираются автоматически. План устранения ограничений ведётся в
[едином roadmap](../roadmap.md#p2--space-console-и-live-editing).
