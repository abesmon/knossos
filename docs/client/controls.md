# Управление и именованные действия ввода

Весь ввод в клиенте завязан на **именованные действия** Godot (`InputMap`), а не на конкретные
клавиши/кнопки. В коде мы спрашиваем `Input.is_action_pressed("player_jump")` /
`event.is_action_pressed("player_interact")`, а физическая привязка (какая именно клавиша) живёт
в секции `[input]` файла `project.godot`. Это даёт одну точку правды, возможность переназначения
и читаемость кода.

## Почему так

Раньше по всему проекту были разбросаны явные проверки (`Input.is_physical_key_pressed(KEY_W)`,
`event.keycode == KEY_V`, `event.button_index == MOUSE_BUTTON_LEFT`). Чтобы поменять раскладку или
понять, «что делает эта клавиша», приходилось грепать константы по всем файлам. Теперь смысл
(«прыжок», «взаимодействие», «основное действие») отделён от привязки.

## Соглашение об именовании

Действия сгруппированы префиксами по домену:

| Префикс     | Домен                                            |
|-------------|--------------------------------------------------|
| `player_`   | перемещение и действия игрока в 3D-мире           |
| `tool_`     | слоты инструментов (см. [tools.md](tools.md))     |
| `debug_`    | отладочные оверлеи                                 |
| `voice_`    | голосовой чат                                      |
| `console_`  | консоль пространства                               |
| `view_`     | 2D-виды (граф топологии, вид сверху геометрии)     |
| `ui_`       | общий UI; плюс встроенные Godot `ui_*`             |

## Карта действий

### `player_*` — игрок ([actors/player/player.gd](../../actors/player/player.gd))

| Действие                   | Привязка по умолчанию | Назначение                                     |
|----------------------------|-----------------------|------------------------------------------------|
| `player_move_forward`      | W (физ.)              | вперёд                                          |
| `player_move_back`         | S (физ.)              | назад                                           |
| `player_strafe_left`       | A (физ.)              | влево                                           |
| `player_strafe_right`      | D (физ.)              | вправо                                          |
| `player_jump`              | Space (физ.)          | прыжок / вверх в полёте / «тап» пробелом        |
| `player_sprint`            | Shift (физ.)          | бег / ускорение полёта                          |
| `player_descend`           | Ctrl (физ.)           | вниз в полёте                                   |
| `player_interact`          | E                     | взаимодействие с объектом под прицелом          |
| `player_chat`              | Enter / KP Enter      | открыть чат                                     |
| `player_main_action`       | ЛКМ                   | основное действие: рисовать/подтвердить/взаимод.|
| `player_secondary_action`  | ПКМ                   | вторичное действие: отмена размещения           |
| `player_scale_up`          | Колесо вверх          | увеличить прицеленный/удерживаемый объект        |
| `player_scale_down`        | Колесо вниз           | уменьшить объект                                 |

Клавиши перемещения и прыжка/бега привязаны **физически** (по расположению на клавиатуре), чтобы
WASD работал на любой раскладке — как раньше `is_physical_key_pressed`.

### `tool_*` — слоты инструментов (см. [tools.md](tools.md))

(вложенный цикл режимов, тумблер). Нажатие чужого хоткея снимает текущий инструмент.

| Действие      | Привязка | Инструмент слота                              |
|---------------|----------|-----------------------------------------------|
| `tool_slot_2` | 2        | рисование: цикл нет → карандаш → ластик → нет |
| `tool_slot_3` | 3        | размещение картинки: тумблер прицеливания     |

## План VR: semantic actions поверх нескольких источников

`InputMap` остаётся desktop/OpenXR binding layer, но его недостаточно для hybrid controller + hand
tracking. Между raw bindings и игровыми системами появится per-hand `InputSourceArbiter`:

- controller и skeletal hand могут быть доступны одновременно;
- source выбирается отдельно для каждой руки и semantic action;
- hand detection не отключает hardware menu/trigger bindings;
- ручной override выше auto detection;
- press/release несут `hand`, `source` и `source_generation`, поэтому release от нового устройства
  не завершает action, начатый старым;
- `InteractionRouter` решает, получает действие UI, captured target, tool или world.

Причина такого разделения и известные failure modes VRChat описаны в
[vrchat-hybrid-lessons.md](vrchat-hybrid-lessons.md#7-ui-и-pointers-hand-identity-должна-быть-явной).

Текущая desktop карта действий не меняется до реализации этого слоя.

## Что осталось на явных кнопках (и почему)

Не всякая проверка кнопки — это «действие ввода». Указатель в 2D-видах
([topology_graph_view.gd](../../scenes/topology_graph_view.gd),
[geometry_top_view.gd](../../scenes/geometry_top_view.gd)) читает **ЛКМ вместе с позицией** для
пана/выделения — это позиционный указатель, а не переназначаемое логическое действие, поэтому
сравнение с `MOUSE_BUTTON_LEFT` там оставлено. Аналогично
[rich_panel.gd](../../actors/rich_panel/rich_panel.gd) не *читает* ввод, а **синтезирует**
`InputEventMouseButton` для проброса клика во вложенный `SubViewport` — это конструирование события,
привязка тут ни при чём.

## Как добавить/поменять действие

Правь секцию `[input]` в `project.godot` (удобнее — через *Project → Project Settings → Input Map*
в редакторе). В коде ссылайся только на имя действия. Физическую vs логическую привязку выбирают
полем события: `physical_keycode` (по расположению, для WASD и `` ` ``) или `keycode` (по символу,
для буквенных шорткатов вроде E/V/S).
