# Система инструментов

> Инструменты игрока (рисование, размещение картинок, пузырь) — единая система: каждый
> инструмент — отдельный класс с инкапсулированной логикой, активен максимум один. Артефакты
> инструментов (штрихи, картинки, пузыри) — обычные объекты [эфемерного слоя](../network/ephemeral-changes.md).

## Модель: инструмент ≠ артефакт

**Инструмент** — то, чем действует игрок (визуал «в руке», прицеливание, превью, обработка
ввода). **Артефакт** — то, что инструмент оставляет в мире: эфемерный объект (`kind`+`props`),
который сериализуется, ходит по сети и материализуется `EphemeralView`. Инструмент живёт у
игрока и по сети не ходит; артефакт — общий для всех в комнате.

| Инструмент | Класс | Слот | Артефакт (`kind`) | Актор |
|---|---|---|---|---|
| Рисование (карандаш/ластик) | [drawing_tool.gd](../../actors/tools/drawing_tool.gd) `DrawingTool` | **2** | `stroke` | `StrokeActor` |
| Размещение картинки | [image_placement_tool.gd](../../actors/tools/image_placement_tool.gd) `ImagePlacementTool` | **3** | `vrweb-node` (`<VRWebImage>`) | через `VrwebBuilder` |
| Пузырь «ушёл сюда» | [bubble_tool.gd](../../actors/tools/bubble_tool.gd) `BubbleTool` | системный, без слота | `bubble` | `Bubble` |

## Устройство

```
Player (хаб ввода: _unhandled_input, именованные действия)
└── ToolManager (actors/tools/tool_manager.gd) — держатель, роутинг, «один активный»
    ├── DrawingTool          ← tool_slot_2
    ├── ImagePlacementTool   ← tool_slot_3
    └── BubbleTool           ← программный вызов (main._drop_leave_bubble)
```

- **[player_tool.gd](../../actors/tools/player_tool.gd) `PlayerTool`** (база, `Node3D`) —
  жизненный цикл инструмента:
  - `activation_request() -> bool` — нажат хоткей слота; инструмент **сам** решает, что
    происходит (см. ниже). Возврат: активен / деактивировался.
  - `equip()` / `unequip()` — «достали»/«убрали»: базовая реализация вешает/снимает визуал
    `make_held_node()` под камерой (`HAND_OFFSET`) и включает/выключает `_physics_process`
    (пер-кадровая логика — только у активного).
  - `primary_pressed()` / `primary_released()` — основное действие (ЛКМ).
  - `secondary_pressed() -> bool` — второстепенное действие (ПКМ); true — поглощено.
  - `on_mouse_capture_changed(captured)` — Esc/потеря фокуса/файловый диалог: инструмент сам
    решает, что отменять (незавершённый штрих, прицеливание).
  - сигналы `hint_changed(text)` (подсказка в статус-строку) и `finished` (инструмент сам
    завершился — менеджер деактивирует).
- **`ToolManager`** — создаёт инструменты, держит `_slots` (input-действие → инструмент) и
  единственный `_active`. Нажатие **чужого** хоткея снимает текущий инструмент и отдаёт запрос
  новому. Ретранслирует `hint_changed` → `status_hint` (main пишет в статус-строку) и
  `pick_requested` → `image_pick_requested` (main открывает файловый диалог).
- **`Player`** остаётся хабом ввода: именованные действия (`tool_slot_2`, `tool_slot_3`,
  `player_main_action`, `player_secondary_action`, см. [controls.md](controls.md)) форвардятся
  в `ToolManager`. Приоритет ЛКМ: активный инструмент → иначе взаимодействие с порталами.

## Запрос активации: инструмент решает сам

Хоткей слота — не команда «включись», а **запрос**: инструмент интерпретирует его по-своему.

- **DrawingTool** — «вложенный» инструмент с внутренним циклом режимов: каждый запрос — шаг
  `НЕТ → КАРАНДАШ → ЛАСТИК → НЕТ` (см. [pencil-tool.md](pencil-tool.md)).
- **ImagePlacementTool** — тумблер: запрос включает прицеливание, повторный — отменяет.
  Состояния `IDLE → AIMING → AWAITING_FILE → IDLE`; в `AWAITING_FILE` инструмент ждёт
  `provide_file(ok, path)` от main (пикер — UI, живёт в main; спавн артефакта — в инструменте).
- **BubbleTool** — системный: запрос активации всегда отклоняет (слота нет), вызывается
  программно (`drop(target_url)` из `main._drop_leave_bubble` при состоявшемся переходе).

## Границы

- **Спавн артефактов** — единый путь у всех инструментов:
  `NetworkManager.request_scene_action({op:"add", id:new_object_id(), kind, parent, ttl, props})`.
  Офлайн-фоллбеки (локальный штрих/узел в мир) — внутри инструмента.
- **UI (файловый диалог, статус-строка)** — у `main`; инструменты общаются с ним сигналами.
- **Знание о навигации** (когда ронять пузырь, `_base_url` для офлайн-картинок) — у `main`;
  инструментам передаётся параметрами/провайдерами.
- **Смоук-тест контракта** — [tests/tool_system_test.gd](../../tests/tool_system_test.gd)
  (`godot --headless tests/tool_system_test.tscn`): слоты, цикл/тумблер, офлайн-штрих/ластик,
  переключение инструментов, отмена по потере захвата мыши.
- **Новый инструмент** = класс-наследник `PlayerTool` + регистрация в `ToolManager.setup`
  (+ слот в `project.godot`, если выбирается игроком) + новый `kind` артефакта с веткой в
  `EphemeralView._make_node` (клиентская половина рецепта из
  [ephemeral-changes.md → «Новые инструменты»](../network/ephemeral-changes.md#границы)).

## Будущее: инструмент как эфемерный объект (pickable)

Идея: инструмент — часть эфемерного слоя, спавнится «из игрока» и может оставаться на сцене.
Тогда другие пользователи видят чужие инструменты в мире (в том числе созданные другими
пользователями), могут подобрать их и сохранить к себе в инвентарь — всё, что попало в
эфемерный слой, уже сериализуемо и воспроизводимо.

Задел в базовом классе — `PlayerTool.descriptor() -> Dictionary`: сериализуемое описание
инструмента в стиле эфемерного слоя (`{"kind": "tool-pencil", "props": {"color": …}}`).
Сейчас дескриптор **никуда не отправляется**: inventory, pickup/ownership и материализация
`tool-*` объектов не определены. Эта работа ведётся в
[едином roadmap](../roadmap.md#p2--performance-streaming-и-ui).
