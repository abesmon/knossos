# Диплинки (собственные схемы приложения)

Как открываются ссылки с собственными схемами VRWeb — `vrwebresource://` (бандл-ресурс) и
`vrweblocal://` (файл ОС). Есть два независимых сценария.

## 1. Внутри приложения (работает)

Клик по ссылке с такой схемой — в HTML-странице или в чате — это **навигация внутри
приложения**, а не внешнее намерение для ОС.

Раньше `vrwebresource://`/`vrweblocal://` классифицировались как «другая схема» и уходили в
`OS.shell_open()` (как `mailto:`/`tel:`), который не находил обработчик — ссылка «не
открывалась». Теперь `TopologyBuilder.classify_href()`
([scripts/topology_builder.gd](../../scripts/topology_builder.gd)) для `PageFetcher.is_local(href)`
возвращает `{kind:"navigate"}`, и адрес открывается обычным `_navigate`.

Единая классификация используется и для ссылок страницы (`_activate_transition`), и для
кликабельных ссылок чата (`_on_chat_meta_clicked` в [scenes/main.gd](../../scenes/main.gd),
см. [multiplayer.md](../network/multiplayer.md) → текстовый чат).

## 2. Запуск приложения по диплинку извне (частично; на macOS не доведено)

Сценарий «кликнул ссылку в браузере/мессенджере → ОС подняла приложение и открыла адрес».

Что сделано:

- **Регистрация схем за приложением (macOS)** — `CFBundleURLTypes` в `Info.plist` через
  поле `application/additional_plist_content` пресета macOS в
  [export_presets.cfg](../../export_presets.cfg). Проверено: схемы `vrwebresource`/`vrweblocal`
  реально попадают в `Info.plist` при `--export-release`, и Launch Services после регистрации
  бандла привязывает их к приложению.
- **Разбор стартового URL** — [scripts/deeplink.gd](../../scripts/deeplink.gd) (`Deeplink.launch_url()`)
  берёт из аргументов запуска первый адрес, который `PageFetcher` умеет открыть как страницу.
  В [scenes/main.gd](../../scenes/main.gd) (`_ready`) он имеет приоритет над домашней страницей.

Известное ограничение (macOS):

- При открытии диплинка macOS поднимает приложение, но передаёт URL **не через `argv`**, а
  Apple Event'ом `application:openURLs:`. Godot 4.6 этот ивент в GDScript **не пробрасывает**
  (в `argv` URL отсутствует — проверено через `ps` у запущенного по диплинку процесса).
  Поэтому внешний диплинк на macOS пока стартует приложение, но открывает домашнюю страницу,
  а не целевой адрес. `Deeplink.launch_url()` сработает только там, где ОС кладёт URL в `argv`
  (например, Windows-обработчик схемы из реестра).

Чтобы довести внешний запуск до конца, нужен один из путей:

- нативный хук на `application:openURLs:` (GDExtension/патч macOS-слоя), отдающий URL в GDScript;
- отдельный лаунчер-шим, который ловит URL и перезапускает бинарь с ним в `argv`.

Регистрация схем под Windows (ключи `HKCU\Software\Classes\<схема>`) экспортом Godot **не
делается** — нужен установщик или разовая саморегистрация при первом запуске; пока не реализовано.

Maker Kit не зависит от этого незавершённого пути: его рекомендуемый **Build & Run** запускает
настроенный executable/`.app` и передаёт `vrweblocal://` обычным argv. Системный handler доступен
там только как явный fallback; на macOS UI рекомендует `executable`.
