# VRWeb — обзор проекта

> **Суть:** превратить интернет в навигируемое 3D-пространство — любую существующую
> HTML-страницу можно «пройти» как трёхмерную сцену, встречая там других людей. Никакой
> поддержки со стороны сайтов не требуется.

## Два имени: VRWeb и Knossos

Их легко перепутать, поэтому сразу разводим:

- **VRWeb** — это **набор технологий и стандарт** (аналогия — HTML/WebRTC). Он описывает, *как*
  договориться: правила трансляции HTML → 3D, синтаксис vrweb-тегов, сетевые протоколы,
  федеративная идентичность. По этим контрактам кто угодно может написать свою реализацию —
  как разные браузеры (Firefox, Safari) читают один и тот же HTML.
- **Knossos** — это **референсная реализация** VRWeb (аналогия — конкретный браузер). Один
  репозиторий, который одновременно и proof-of-concept, и рабочий продукт. Сюда входят два
  референса: **клиент** на Godot 4 (тот, по которому ходят) и **домашний сервер** на FastAPI
  (`homeserver/` — показывает, как реализовать сигналинг, идентичность и серверные фичи).

Дальше в документах «клиент», «домашний сервер», «сигнальный сервер» — это части Knossos;
когда речь про контракт, который обязан соблюдать любой участник экосистемы, — это VRWeb.
Мотивация и позиционирование целиком — в [корневом README](../README.md).

## Концепция

Клиент Knossos — это интерпретатор: он берёт произвольный HTML-документ и превращает его в
3D-сцену, по которой можно ходить. Аналогия — обычный браузер интерпретирует HTML в 2D-страницу;
Knossos делает то же самое, но в трёх измерениях.

Ключевые принципы:
- **Любой сайт без изменений** — страница не обязана знать о VRWeb
- **Клиент решает всё** — вся логика трансляции на стороне клиента
- **Слои независимы** — каждый следующий этап (мультиплеер, кастомные теги, персистентность) добавляется поверх, не ломая предыдущий
- **Открытый стандарт** — любой может написать альтернативный клиент (плагин для браузера, мобильное приложение и т.д.)

## Архитектура: слои как независимые строительные блоки

Главная идея, которую легко упустить за списком фич: **VRWeb — не монолит, а стек
самостоятельных блоков.** Каждый следующий слой строится поверх предыдущего, но в идеале
**не обязан знать о других** — ни о тех, что под ним, ни тем более над. Каждый блок достаточно
самодостаточен, чтобы его можно было **заменить, вынести в отдельный сервис/сервер или
делегировать**; а при грамотной оркестрации все блоки складываются в цельный продукт уровня
VRChat, где любую детальку можно поменять, не сломав остальное.

Прямое следствие этой развязки — **graceful degradation**: один и тот же стек предъявляется
целым спектром представлений — от полноценного VR-экспириенса с фуллбоди-трекингом до простого
голосового чата, как в Discord (все подключаются к одному инстансу). Не обязательно запускать
весь стек: осмысленно работает любое подмножество. Спектр представлений и как он ложится на слои —
[graceful-degradation.md](graceful-degradation.md).

Ниже — слои снизу вверх; у каждого важно не только «что делает», но и **чего он намеренно
не знает** (это и есть граница блока).

### Слой 1 — Визуализация (документ → 3D)

Превращает **документ** в навигируемое 3D-пространство по правилам маппинга. Здесь ещё
**нет онлайна вообще** — это чистый одиночный «3D-браузер».

Граница: слою **всё равно, откуда взялся документ.** HTML из сети, кастомные vrweb-теги или
сцена, досбранная из изменений эфемерного слоя (Слой 2), — для визуализации это **один документ,
который надо отрисовать**. О сети, идентичности и персистентности слой не знает ничего.

**Режимы работы:**
- `combine` — HTML рендерится в 3D, кастомные теги дополняют сцену
- `exclusive` — HTML игнорируется, используются только кастомные vrweb-теги (подробности — [space/vrweb-tags.md](space/vrweb-tags.md))

Поставка собственных классов и скриптов автора вместе со страницей проектируется через
namespaced `.vrmod`: trusted GDScript для разрешённых origin и отдельный capability sandbox для
недоверенного кода — [space/scripting-modules.md](space/scripting-modules.md).

### Слой 2 — Реалтайм (WebRTC)

Всё, что делает пространство **живым и общим**: голос, чат, эфемерные объекты (граффити,
пузыри…), сигнальный сервер для handshake, карточки пользователей по сети, ранги/права в комнате.

Внутри слоя проходит важная граница:
- **Голос, чат, карточки, присутствие** ничего не знают и **не должны знать про 3D**. На них
  одних можно собрать чисто голосового клиента, где люди — «призраки», как в Discord (это и есть
  graceful degradation, см. [graceful-degradation.md](graceful-degradation.md)).
- **Эфемерные объекты** — единственная часть слоя, которая **частично знает о документе**: её
  задача — описать, *как* передаются изменения документа между участниками. Результат отдаётся
  вниз, Слою 1, как часть «документа на отрисовку».

### Слой 3 — Идентичность и домашний сервер

Серверная сторона: идентичность (регистрация, верификация по публичному ключу — аналог
ActivityPub / W3C DID), друзья, персональные пространства, прочий функционал домашнего сервера.
Опционален: без регистрации пользователь — анонимная капсула.

Граница: слой **ничего не знает ни о реалтайме, ни о 3D.** Для него это просто серверные вещи —
аккаунты, друзья, пространства; сервер **не знает, зачем и как** его данные будут использованы.
Поэтому его можно вынести куда угодно и федерировать (как Matrix).

### Слой 4 — Персистентность (эфемерное → постоянное)

Протокол «флаша»: как временные изменения из Слоя 2 закрепляются в самом документе/на странице
навсегда. Клиент шлёт событие → сервер/сайт обрабатывает → обновляет страницу → при следующем
заходе изменение уже часть документа.

Граница: протокол **ничего не знает о 3D и о том, как шло эфемерное взаимодействие.** Его дело —
задать, *какие сообщения* сервер/сайт/страница может принять, как это работает и зачем нужно
концептуально. Это контракт **конечного клиента и конечного сайта** — федерация, домашние
серверы и сигналинг здесь ни при чём. Специфицирован в
[page-persistence.md](network/page-persistence.md).

## Правила трансляции HTML → 3D (Слой 1)
Этот раздел мог устареть! И носит скорее ознакомительный характер. Подробности в /docs/space

### Структура → геометрия

| HTML-элемент | 3D-аналог |
|---|---|
| `<body>` | корневое пространство |
| `<header>` | фасад / вход |
| `<main>`, `<article>`, `<section>` | отдельные комнаты / зоны |
| `<nav>` | коридор или хаб с указателями |
| `<aside>` | боковое ответвление |
| `<footer>` | подвал (буквально) |

Для сайтов без семантики (div-soup) — визуальные эвристики по размеру, позиции и плотности контента блоков.

### Контент → объекты

| Контент | 3D-аналог |
|---|---|
| `<h1>`–`<h6>` | крупные объекты, масштаб по уровню |
| `<p>` | текстовые панели / таблички |
| `<img>` | текстуры / картины на стенах |
| `<video>` | экраны / телевизоры |
| `<a href="...">` (внутренняя) | дверь в другую комнату |
| `<a href="...">` (внешняя) | портал в другой мир |
| `<button>`, `<input>`, `<form>` | интерактивные панели |
| `<ul>`, `<ol>` | кластеры объектов |

### CSS → визуальные свойства

| CSS | 3D-аналог |
|---|---|
| `background-color` | цвет стен / пола |
| `font-size` | масштаб объекта |
| `display: none` | не рендерится |
| `z-index` | глубина в пространстве |

### Граф ссылок → топология

- Слабосвязанные части сайта → острова
- Частые переходы между страницами → мостики
- Страницы с высоким in-degree → большие центральные хабы

## Кастомные теги (Слой 1, расширение)
Этот раздел мог устареть! И носит скорее ознакомительный характер. Подробности в /docs/space

Теги с неймспейсом `vr-*` — обычный браузер их игнорирует, VRWeb их читает:

```html
<!-- Вставить 3D-сцену -->
<vr-scene src="world.glb"/>

<!-- Режим замены HTML -->
<html vr-mode="override">

<!-- Декларативное событие -->
<vr-object vr-on="player_enter" vr-action="animate"/>
```

Контент сцены — GLTF/GLB (открытый стандарт, Godot поддерживает нативно).

## Мультиплеер (Слой 2)
Этот раздел мог устареть! И носит скорее ознакомительный характер. Подробности в /docs/network

- **Комната (WebRTC-room)** — группа пиров по URL как непрозрачному ключу; про документ/3D не знает (полная связка URL↔3D↔HTML — это «инстанс», см. [multiplayer.md](network/multiplayer.md))
- **Транспорт:** WebRTC (голос, чат, позиции, эфемерные объекты)
- **Сигнальный сервер:** нужен только для handshake; после установки соединения — p2p напрямую
- **Федерация серверов:** сигнальные серверы пересылают сообщения друг другу (аналог Matrix); пользователи с разных серверов попадают в одну комнату
- **Масштабирование:** при 50+ людях в комнате mesh WebRTC не справляется — нужен SFU (Selective Forwarding Unit)

Простой вариант для прототипа: один сигнальный сервер на комнату, клиент выбирает через метатег на странице (или дефолтный публичный сервер).

## Технический стек

**Клиент:** Godot 4

**Ключевые проблемы реализации в Godot:**
- Парсинг HTML — свой снисходительный парсер на GDScript (`HtmlParser` → `HtmlNode`),
  т.к. нативного HTML-парсера в Godot нет (`XMLParser` требует валидный XML). GDExtension/libxml2
  не понадобился
- JS-рендеринг SPA — в первой версии не поддерживается: ожидаем что HTML уже готов после загрузки
- WebRTC — официальный модуль `godot-webrtc`
- 3D-контент — GLTF/GLB нативно поддерживается Godot

**Получение HTML:**
- Прямой `HTTPRequest` (без исполнения JS)
- Позже: прокси или headless Chromium для JS-рендеринга

## Этапы разработки (Roadmap)

### Phase 1 — Singleplayer 3D Browser
**Цель:** ходить по любой странице в 3D, кликать по ссылкам для навигации

- [x] HTTPRequest HTML из Godot (`PageFetcher`)
- [x] Парсинг HTML → дерево узлов (`HtmlParser`, свой снисходительный парсер)
- [x] Маппинг: `section` → комната, `h1` → объект, `a` → портал/дверь (`TopologyBuilder`)
- [x] Базовая навигация от первого лица (`actors/player`)
- [x] CSS `background-color` → цвет окружения (`hints.css` → `WorldGenerator`)
- [x] Кликабельные `<a>` — телепорт/переход на новую страницу (`actors/portal`)

**Deliverable:** можно ввести URL и погулять по странице в 3D. ✅ Реализовано —
см. [implementation.md](implementation.md).

---

### Phase 2 — Multiplayer
**Цель:** встречать людей на любой странице

- [x] Сигнальный сервер (WebSocket, Python) — `signaling/`
- [x] WebRTC handshake через Godot-модуль (mesh, p2p)
- [x] Капсулы других игроков в сцене (`actors/remote_player`)
- [x] Синхронизация позиций (`NetworkManager`, `RemotePlayersView`)
- [x] Текстовый чат
- [x] Голосовой чат — Opus поверх data-каналов mesh (аддон twovoip), VAD + джиттер-буфер,
  пространственный звук от капсулы — см. [voice-chat.md](client/voice-chat.md)

**Deliverable:** несколько человек одновременно на одной странице видят и слышат друг друга.
✅ Реализовано — см. [multiplayer.md](network/multiplayer.md), [voice-chat.md](client/voice-chat.md).

---

### Phase 3 — Identity + Custom Worlds
**Цель:** аватары, персистентность, расширенная разметка

- [x] `<vrweb>` теги — парсинг и рендеринг сцены узлами Godot (`VrwebBuilder`),
  режимы `combine`/`exclusive` — см. [vrweb-tags.md](space/vrweb-tags.md)
- [x] Видео-плеер (как в VRChat): логический плеер → текстура → поверхности, синхронизация
  воспроизведения по сети (`<VRWebVideoPlayer>`/`<VRWebVideoScreen>`) — см. [video-player.md](client/video-player.md)
- [x] Инструментарий редактора: сборка сцены в Godot → экспорт в HTML, авторинг
  внешних ресурсов по URL, дебаг-превью (`addons/vrweb_tools/`) — см. [vrweb-export.md](vrweb-export.md)
- [x] Добавить в `test_pages` открываемое из общего index package-demo: self-contained
  переключатель, готовый `.vrmod` с GDScript/сценой/ассетами и воспроизводимая команда его
  пересборки встроенным exporter. Автоматический round-trip и пользовательская демка готовы —
  см. [scripting-modules.md](space/scripting-modules.md).
- [x] GLTF/GLB-сцены в HTML — через `<ExtScene src="ExtResource:::…">` + `<ExtResource>`
  с URL на `.glb`; скачивается и вставляется в сцену (`VrwebResourceLoader`), см.
  [vrweb-tags.md](space/vrweb-tags.md). Base64-инлайн (без внешнего URL) — не сделано.
- [x] Home server — pubkey-идентичность (сертификаты `nick@domain`), см. [home-server.md](network/home-server.md)
- [x] Анонимный режим (капсула без аккаунта) — самозаявленный `user_id`, см. [ranks.md](network/ranks.md)
- [x] Персистентность: флаш эфемерного слоя в страницу + персональные пространства
  домашнего сервера, см. [page-persistence.md](network/page-persistence.md), [personal-spaces.md](network/personal-spaces.md)
- [ ] Федерация server-to-server: сигнальные серверы пересылают сообщения друг другу, чтобы
  пользователи с разных сигналингов попадали в один меш (см. [multiplayer.md](network/multiplayer.md)
  → «Федерация»); фундамент для будущих фич вроде друзей

**Deliverable:** кастомные персистентные 3D-миры поверх обычных HTML-страниц.

- ~~Как именно парсить HTML в Godot?~~ — решено: свой парсер `HtmlParser` на GDScript.
- Fallback для div-soup сайтов: какие именно визуальные эвристики?
- Формат описания правил трансляции (чтобы сообщество могло добавлять свои)

  (`<ExtScene>`, `<VRWebVideoPlayer>`, …), см. [vrweb-tags.md](space/vrweb-tags.md); неймспейс
  `vr-*` из ранних заметок не используется.
- Безопасность: scripting modules уже проходят integrity-проверку и exact-hash trust preflight,
  но trusted GDScript всё ещё получает права процесса. Декларативные Godot-классы document и
  live-peer путей сведены в единую audit policy, однако enforce-правила пока пусты. План —
  пользовательские уровни безопасности и opt-in safe profile. Статус, карта рисков и модель —
  в [security.md](security.md).
- Стандартизация приватных инстансов: query-параметр (рабочий вариант `vrweb-instance=<код>`),
  чтобы клиенты и серверы понимали его одинаково — см.
  [multiplayer.md](network/multiplayer.md) → «Приватные инстансы».
