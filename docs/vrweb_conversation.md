# VRWeb — лог переписки

---

**Юзер написал:**
Идея: сделать UGC 3D/VR платформу поверх HTTP. Смысл — каждая страница в интернете расценивается как узел/инстанс. Пользователи соединяются через WebRTC и попадают в комнату с названием страницы. Клиент рендерит HTML в 3D по определённым правилам и постепенно создаёт свой слой разметки. Авторизация — через распределённую систему, каждый может поднять свой сервер или пользоваться публичным. Без авторизации можно просто зайти анонимной капсулой. Аватар берётся с home server'а. Также планируется возможность менять миры в реальном времени через протокольные методы на странице.

**Я ответил:**
Разобрал архитектуру по блокам:
- WebRTC + STUN/TURN: кастомные STUN/TURN задаются через RTCConfiguration. Публичные STUN от Google бесплатны, TURN требует своего сервера. Сигнальный сервер всё равно нужен для handshake — это единственная централизованная точка.
- HTML → 3D: самая сложная часть. Варианты — парсить DOM и строить сцену по правилам (секции → комнаты, ссылки → двери, картинки → текстуры). A-Frame уже делает похожее. Кастомный слой разметки — логичное развитие.
- Авторизация: описанная модель похожа на ActivityPub/Fediverse или DID (W3C стандарт). Анонимный режим снижает барьер входа.
- Live editing: DOM mutation через MutationObserver. Вопрос в том кто имеет право менять DOM. Event-based протокол поверх WebRTC data channels.
- Главные риски: TURN сервера стоят денег, сигнальный сервер — точка централизации, производительность WebRTC mesh при 50+ людях (нужен SFU).

Аналогия с браузером правильная — вы делаете рантайм/клиент, а не платформу. Зафиксировал архитектуру:
- Слой 1: эвристики для HTML → 3D + кастомные теги + режимы (hybrid, override, full replace)

3/4) Да, каждый слой не зависит от другого. Все могут существовать сами по себе.

**Я ответил:**
- Godot — хороший выбор. Есть официальный модуль godot-webrtc. Главный вопрос — парсинг HTML внутри Godot (либо самостоятельно, либо через GDExtension + libxml2 или аналог).
- Федерация сигнальных серверов: WebRTC само по себе не знает про сигнальный сервер. Если Алиса на сервере A, а Боб на сервере B — серверам нужно пересылать сигнальные сообщения друг другу. Это именно что делает Matrix. Предложенная модель: комната = URL страницы, живёт на одном сервере, пользователи с других серверов участвуют через server-to-server протокол. После handshake — WebRTC напрямую. Более простой вариант для начала: один сигнальный сервер на комнату, клиент выбирает к какому подключиться (через метатег на странице).
- Независимость слоёв подтвердил — архитектурно сильная сторона.

**Юзер написал:**
Страница никак не объявляет поддержку протокола. Любая страница должна поддерживаться — планирую парсинг ЛЮБОГО HTML документа в 3D пространство. Просто по спец тегам можно будет понять что есть кастомная надстройка.

**Я ответил:**
Клиент сам решает как рендерить — без участия сервера. Это интерпретатор HTML → 3D сцена. Набросал систему правил:
- Структура → геометрия: body → корневое пространство, header → фасад/вход, section/article/main → комнаты/зоны, nav → коридор/указатели, footer → подвал буквально.
- Контент → объекты: h1-h6 → крупные объекты по уровню, p → текстовые панели, img → текстуры, video → экраны, a → порталы, button/input → интерактивные объекты, ul/ol → кластеры.
- CSS → визуальные свойства: background-color → цвет стен/пола, font-size → масштаб объекта, display:none → не рендерится, z-index → глубина.
Кастомные теги как опциональный слой — клиент проверяет наличие тегов с нашим неймспейсом. Атрибут vr-mode="override" говорит игнорировать HTML и рендерить только vr-теги.
Вопрос: как получать HTML внутри Godot? HTTPRequest (без JS execution), GDExtension обёртка над парсером, или прокси-сервис. SPA без JS execution — отдельная большая проблема.

**Юзер написал:**
Если использовать HTML + Godot, насколько можно объединять gdscene и файлы Godot внутри HTML? Кажется что структура документов Godot уже похожа на XML. По рендеру и JS — в первое время не интересует. Если сайт рендерит себя через JS, ожидаем что документ будет конечным после вызова. Но есть другая хотелка: нужно создавать динамику внутри 3D сцен и описывать скрипты. Хотим поддерживать как существующий скриптинг, так и скриптинг Godot. Не знаю как это решать.

**Я ответил:**
- По объединению Godot файлов и HTML: TSCN — текстовый формат похожий на INI, ESCN/GLTF — JSON. Технически можно через `<vr-scene src="world.glb"/>` или инлайн GLTF как data. GLTF лучший выбор — открытый стандарт, Godot поддерживает нативно.
- По скриптингу — четыре варианта:
  - A) GDScript инлайн в HTML: Godot умеет компилировать GDScript в рантайме. Но огромная проблема безопасности — исполнение произвольного кода с любого сайта.
  - B) JavaScript через встроенный движок (V8 или QuickJS через GDExtension): люди его знают, но большая зависимость.
  - C) Декларативные события (vr-on event="player_enter" action="animate"): безопасно, предсказуемо, но ограничено.
  - D) WebAssembly: любой язык компилируется в WASM, хорошая изоляция по природе sandbox.
- Рекомендация: для первого этапа — вариант C как базовый минимум, параллельно думать про вариант D как путь к полноценному скриптингу без дыр в безопасности. GDScript инлайн — слишком опасно для публичного клиента.
