# Домашний сервер (Слой 3 — федеративная идентичность)

> Реализация «Слоя 3» из [vrweb-overview.md](vrweb-overview.md) и «центров авторизации»,
> обещанных в [ranks.md](ranks.md). Код — в [../homeserver/](../homeserver/).

## Что такое домашний сервер

**Домашний сервер** — сервер, на котором пользователь регистрируется и который отвечает за
его идентичность. Адрес пользователя глобален и федеративен, как в федиверсе:

```
nickname@homeserverdomain
```

Система домашних серверов проектируется как федерация: каждый может поднять свой сервер,
пользователи разных серверов встречаются в одних мирах, а идентичность подтверждается
криптографически без центрального реестра.

Роли домашнего сервера (сейчас и в перспективе):

1. **Авторизация и валидация личности** — главная функция v1. Сервер подтверждает, что
   владелец ключа действительно `nickname@domain` (см. «Сертификаты идентичности»).
2. **Анонс функционала (features)** — сервер публикует список возможностей, которые он
   поддерживает. Клиент сверяет со своим списком: пересечение — то, что реально работает.
3. **Анонс конфигурации** — сигнальный сервер, домашняя страница, прочие настройки, которые
   клиент подхватывает после логина.
4. **Веб-морда** — регистрация, логин, просмотр базовой информации об аккаунте; дальше —
   всё, что потребует растущий функционал.
5. *(будущее)* — друзья, хранение аватара/профиля, персистентные изменения миров (Слой 4),
   server-to-server федерация сигналинга.

## Домашний сервер и сигналинг: разные сущности, один процесс

Концептуально **домашний сервер** (идентичность) и **сигнальный сервер** (WebRTC-handshake,
см. [multiplayer.md](multiplayer.md)) — независимые сущности: клиент может логиниться на
одном домене, а сигналиться через другой (адрес сигналинга — часть анонсируемой конфигурации).

В нашей реализации они **слиты в один монолит** — один процесс, один порт:

```
                    ┌──────────────── homeserver (FastAPI, один процесс) ───────────────┐
                    │                                                                    │
 [браузер] --HTTP-->│  веб-морда (/)           ┐                                         │
 [Godot]  --HTTP--> │  REST API (/api/v1/...)  ├─ accounts / identity / keys / db       │
 [любой]  --HTTP--> │  /.well-known/vrweb      ┘   (модули домашнего сервера)           │
 [Godot]  --WS----> │  /signal  ──────────────────  SignalingHub (модуль сигналинга)    │
                    │                                                                    │
                    └────────────────────────── SQLite + ключ сервера в data dir ───────┘
```

Граница между сущностями держится на уровне модулей: `signaling.py` — самодостаточный
transport-agnostic hub, не знающий об аккаунтах; модули идентичности не знают о сигналинге.
Разнести их по разным процессам — вопрос деплоя, не переписывания.

Старый standalone-сервер [../signaling/](../signaling/) остаётся рабочим (тот же протокол);
монолит — его надмножество.

## Discovery: `/.well-known/vrweb`

Точка входа федерации. Любой (клиент или чужой сервер), зная только домен, получает всё
нужное одним GET-запросом:

```json
{
  "server":  {"software": "knossos-homeserver", "version": "0.1.0",
              "domain": "example.com", "name": "My VRWeb Home"},
  "features": ["identity.v1", "signaling.v1"],
  "config":  {"signaling_url": "wss://example.com/signal",
              "homepage": "https://example.com/"},
  "signing_keys": [
    {"key_id": "rsa:1", "algorithm": "rsa-sha256", "public_key": "<base64 DER SPKI>"}
  ]
}
```

- **`features`** — версионированные строки-капабилити. Клиент поддерживает свой набор;
  работает пересечение. Сервер может не анонсировать функционал, который клиент умеет
  (и наоборот) — обе стороны обязаны это переживать. Новые фичи добавляются новыми
  строками (`friends.v1`, …), несовместимые изменения — новой версией (`identity.v2`).
- **`config`** — рекомендации сервера клиенту (подхватываются после логина, но доступны
  и анонимно). Клиент вправе переопределить локально (адрес сигналинга в настройках
  уже так работает).
- **`signing_keys`** — публичные ключи, которыми сервер подписывает сертификаты
  идентичности. Именно через этот эндпоинт чужие участники проверяют подписи —
  это и есть база федерации: доверие к `nick@example.com` сводится к HTTPS-доверию
  к домену `example.com`.

### Криптография: почему RSA-SHA256, а не Ed25519

Подписи проверяет в том числе Godot-клиент, а его `Crypto`/`CryptoKey` (mbedTLS)
умеют только RSA, и `HashingContext` — максимум SHA-256 (Ed25519 требует SHA-512).
Поэтому вся система работает на **RSA-2048, подпись PKCS#1 v1.5 + SHA-256**
(`algorithm: "rsa-sha256"`), ключи ходят как base64 от DER SubjectPublicKeyInfo —
это формат и ключей сервера в `signing_keys`, и ключей клиентов в сертификатах.
Поле `algorithm` версионирует выбор: появление Ed25519 в клиенте — это новый
`algorithm`/`key_id`, а не слом формата.

## Сертификаты идентичности

Закрывает компромисс из [ranks.md](ranks.md) («user_id самозаявлен и не подписан»).

Модель (аналог ActivityPub/DID, упрощённо):

1. Клиент локально генерирует **свой** RSA-ключ (в клиенте — `user://identity_key.pem`;
   `user_id` пока остаётся отдельным случайным id, см. оговорку ниже).
2. Клиент логинится на домашнем сервере и просит подписать свой публичный ключ:
   `POST /api/v1/identity/certify`.
3. Сервер выдаёт **сертификат** — канонический JSON, подписанный ключом сервера:

   ```json
   {"v": 1, "address": "alice@example.com",
    "public_key": "<base64 DER SPKI клиента>",
    "issued_at": 1780000000, "expires_at": 1782600000,
    "key_id": "rsa:1"}
   ```

   Подпись — над канонической сериализацией (JSON c sorted keys, компактные разделители,
   UTF-8). Ответ несёт `certificate` (объект — для отображения), **`certificate_json`**
   (та самая каноническая строка) и `signature` (base64). Клиенты хранят и пересылают
   именно строку: проверяющему не нужно воспроизводить каноническую сериализацию —
   он верифицирует подпись над сырыми байтами и парсит поля из них же.

4. В комнате пир предъявляет сертификат и **доказывает владение приватным ключом**
   (см. «Проверка в комнате» ниже). Итог: `nickname@domain` подтверждён, подделать нельзя.
5. Пользователь **без** домашнего сервера остаётся анонимной капсулой с самозаявленным
   `user_id` — как сейчас. При выдаче прав такому id клиент должен предупреждать
   (см. [ranks.md](ranks.md)).

Клиентская часть реализована (см. «Клиент» ниже): autoload `HomeServer`
([../scripts/home_server.gd](../scripts/home_server.gd)) держит ключ/токен/сертификат и
проверяет чужие сертификаты, а challenge–response в меше делает `NetworkManager`.

### Три пары ключей — кто чем владеет

Чтобы флоу читался, надо держать в голове, что в системе **три разные** пары ключей, и
приватная половина каждой никогда не покидает своего владельца:

| Пара ключей | Приватная часть у | Публичная часть где видна | Зачем |
|---|---|---|---|
| **Ключ сервера** (`signing_key.pem`) | на диске home server'а | в `/.well-known/vrweb` → `signing_keys` | сервер подписывает **сертификаты** |
| **Ключ клиента А** | только на устройстве А | внутри сертификата (`public_key`) | А доказывает, что сертификат — его |
| **Ключ клиента Б** | только на устройстве Б | внутри сертификата Б | то же самое для Б |

Ключевая мысль: **приватный ключ клиента (`PRIV_A`) никогда никуда не отправляется** — ни на
сервер, ни в комнату, ни в сертификат. В сертификат идёт только **публичный** `PUB_A`. Сервер
подписывает связку «адрес ↔ `PUB_A`», но самого `PRIV_A` он не знает и знать не должен.

### Полный флоу выдачи сертификата

```
устройство А                          home server (example.com)
────────────                          ─────────────────────────
генерирует пару (PRIV_A, PUB_A)
  — один раз, PRIV_A остаётся тут
логин (nick + пароль) ───────────────▶ проверяет пароль, выдаёт access_token
                       ◀─────────────── access_token
POST /api/v1/identity/certify
  Bearer access_token
  body: { public_key: PUB_A }  ───────▶ знает из сессии: это alice
                                        собирает certificate =
                                          { address: "alice@example.com",
                                            public_key: PUB_A, expires_at, key_id }
                                        signature = sign(PRIV_сервера,
                                                         canonical_json(certificate))
                       ◀─────────────── { certificate, signature, key_id }
кладёт (certificate, signature) себе;
предъявляет их в комнатах
```

То есть А получает сертификат **заранее**, залогинившись на своём home server'е паролем.
Дальше сервер в общении А и Б **не участвует** — сертификат самодостаточен (в нём уже есть
подпись сервера). Это важно для p2p-меша: проверить А можно, даже если его home server сейчас
недоступен, лишь бы однажды удалось скачать `signing_keys` домена.

### Проверка в комнате: **две независимые подписи**

Когда А предъявляет сертификат пиру Б, Б выполняет **два** отдельных шага — и оба обязательны:

**Шаг 1. Подпись сервера на сертификате** — «серверу можно верить, что `PUB_A` = alice».

- Из адреса в сертификате (`alice@example.com`) Б берёт домен `example.com` и запрашивает
  `https://example.com/.well-known/vrweb` (кэшируется), достаёт оттуда публичный ключ сервера
  по `key_id`.
- Проверяет `signature` над `canonical_json(certificate)` этим ключом. Прошло → сервер
  действительно выдал этот сертификат, и в нём не переставлены поля (иначе каноническая
  сериализация не совпадёт и подпись развалится).
- **Почему это доверяют:** ключ сервера берётся с самого домена по HTTPS. Доверие к
  `alice@example.com` сводится к TLS-доверию к домену `example.com` — тому же, на котором
  держится обычный веб. Подделать `signing_keys` может лишь тот, кто уже владеет доменом.

> **Строгая проверка источника ключей (критично!).** Всё держится на том, что ключи для
> домена `example.com` берутся именно с `https://example.com`. Поэтому клиент считает ключи
> **авторитетными для домена только из канонического источника**: схема `https://` И хост URL
> совпадает с доменом. Из этого следуют два правила:
>
> 1. Ключи, полученные при discovery СВОЕГО домашнего сервера, кэшируются под его доменом,
>    только если заявленный сервером `domain` совпадает с хостом, куда клиент реально ходил.
>    Иначе сервер (даже свой — если в конфиге `domain=example.com`, а живёт он на `localhost`,
>    — или скомпрометированный, анонсирующий чужой `domain`) присвоил бы себе чужой домен и
>    **отравил кэш**: `example.com → ключ_localhost`. Строгий режим такие ключи не кэширует.
> 2. Кэш на диске (`known_servers.cfg`) помнит origin каждой записи; неканоничные записи в
>    строгом режиме игнорируются и перезапрашиваются канонично (чтобы запись, добавленная в
>    insecure-режиме, не «пережила» до истечения TTL).
>
> Из-за этого **`domain` в конфиге сервера обязан быть реальным HTTPS-хостом, где сервер
> отдаётся**. Для локальной разработки (self-host по http, `domain` = адрес вида
> `127.0.0.1:8080`) строгая проверка отклонит не-https источник — на этот случай есть
> **insecure-режим**: cmdline `--insecure-identity` или env `VRWEB_INSECURE_IDENTITY=1`
> ([HomeServer.insecure_identity](../scripts/home_server.gd)). Он не отключает проверку
> подписи — лишь снимает требование канонического источника (неканоничный принимается с
> предупреждением «подозрительно» вместо отказа). По умолчанию ВЫКЛ; **в проде не включать**.

**Шаг 2. Доказательство владения приватным ключом (challenge–response)** — «предъявитель
прямо сейчас и есть владелец `PUB_A`, а не тот, кто скопировал чужой сертификат».

```
Б → А:  nonce = 32 случайных байта (свежие, одноразовые)
А → Б:  proof = sign(PRIV_A, nonce)          // подпись именно этого челленджа
Б:      verify(PUB_A_из_сертификата, proof, nonce) ?
```

`PUB_A` Б берёт **из уже проверенного на шаге 1 сертификата**. Подпись сходится → предъявитель
владеет `PRIV_A`. Итог обоих шагов: `alice@example.com` подтверждён криптографически.

### Почему Б не может украсть личность А, увидев его сертификат

Ровно тот вопрос, ради которого всё это. Сертификат **не секретный** — его видно всем в
комнате, и это нормально. Пусть Б скопировал сертификат А и заявляет «я alice»:

- Шаг 1 у Б пройдёт: подпись сервера настоящая, сертификат неизменён. ✅
- Шаг 2 — провалится. Проверяющий В шлёт Б **свежий случайный** `nonce` и требует
  `sign(PRIV_A, nonce)`. Но `PRIV_A` у Б **нет** — он никогда не покидал устройство А. Б либо
  не ответит, либо подпишет своим `PRIV_B`, и проверка через `PUB_A` из сертификата не сойдётся.
  ❌ Самозванец разоблачён.

**Почему нельзя переиграть старую подпись (replay).** А что если Б подсмотрит не только
сертификат, но и `proof`, которым А однажды ответил кому-то? Не поможет: `nonce` **случайный и
одноразовый**. Перехваченный `proof` сделан над старым `nonce = X`; когда В проверяет Б, он
генерирует **новый** `nonce = Y`, а подпись над `X` для `Y` не подходит. Подписать `Y` заново
может только владелец `PRIV_A`. Поэтому челлендж обязан быть свежим на каждую проверку. В нашей
реализации подписываются не голые байты nonce, а
`"vrweb-identity-proof.v1:<peer_id доказывающего>:<peer_id проверяющего>:" + nonce` —
привязка к паре пиров закрывает и relay-атаку (переслать чужой челлендж настоящей Алисе и
вернуть её подпись как свою не выйдет: peer_id в payload не совпадут).

> **Аналогия.** Сертификат — паспорт с печатью МВД: «этот человек — Алиса». Ксерокопию сделать
> легко (шаг 1 — печать настоящая). Но на проходной просят вживую **повернуться по команде**
> (шаг 2 — свежий челлендж) — и владелец ксерокопии это повторить не может, потому что «лицо»
> (`PRIV_A`) есть только у настоящей Алисы. Одной печати недостаточно — нужны оба шага.

## Клиент (Godot)

Клиентская сторона живёт в двух местах:

- **`HomeServer`** (autoload, [../scripts/home_server.gd](../scripts/home_server.gd)) —
  всё «между клиентом и серверами»: discovery своего сервера, логин/регистрация/логаут
  (Bearer-токен), генерация RSA-ключа, получение/продление сертификата, проверка подписи
  сервера на чужих сертификатах и кэш `signing_keys` доменов.
- **`NetworkManager`** — всё «между пирами»: предъявление сертификата при открытии
  p2p-канала (вместе с карточкой идентичности), челлендж владения ключом, сигнал
  `identity_verified(id, address)` и `verified_address_of(peer_id)` для UI.

Конфигурация адресов (приоритет сверху вниз):

| Значение | Домашний сервер | Сигналинг |
|---|---|---|
| задал пользователь (вкладки «Аккаунт»/«Сеть») | `Settings.home_server_url` | `Settings.signaling_url` |
| анонс домашнего сервера (`config` из discovery) | — | `HomeServer.announced_signaling_url` |
| дефолт сборки (`config/build.private.cfg`) | `BuildConfig.home_server_url` | `BuildConfig.signaling_url` |

Резолвят `Settings.effective_home_server_url()` / `effective_signaling_url()`; пустое поле
в настройках = «авто» (плейсхолдер показывает фактический адрес).

Файлы в `user://` (все — через `Sandbox.resolve`, см. docs/multiplayer.md → Проверка):

- `identity_key.pem` — приватный RSA-ключ клиента (никогда не покидает устройство);
- `homeserver.cfg` — токен сессии, адрес аккаунта, сертификат (каноническая строка + подпись);
- `known_servers.cfg` — кэш `signing_keys` доменов (сутки свежий; протухший — fallback,
  чтобы проверять сертификаты и при недоступном сервере пира). Каждая запись помнит `origin`
  — строгая проверка источника ключей (см. врезку выше) доверяет только каноническим.

Локальная разработка: клиент против self-host по http (напр. `home_server_url=http://127.0.0.1:8080`)
в строгом режиме НЕ сможет верифицировать сертификаты (источник не https) — запускать с
`--insecure-identity` (или `VRWEB_INSECURE_IDENTITY=1`). Вкладка «Аккаунт» показывает
предупреждение, когда этот режим включён.

Флоу в UI: вкладка **«Аккаунт»** в настройках — адрес сервера (пусто = дефолт сборки),
статус discovery (имя, домен, features), вход/регистрация/выход, срок сертификата.
Логин автоматически запрашивает сертификат; при старте `HomeServer.refresh()` валидирует
токен и продлевает сертификат, если тот истекает в ближайшие 7 дней. В списке
«Пользователи» подтверждённые пиры получают иконку-галочку и адрес `nick@domain`.

### Две идентичности в UI: display_name и подтверждённый адрес

У пира две разные «личности», и UI показывает обе:

- **`display_name`** — имя, которое пир **сам заявил** о себе (ник карточки, `nick_of`). Ничем
  не подтверждено — так было всегда.
- **подтверждённый адрес `nick@domain`** — криптографически проверенная федеративная
  идентичность (`verified_address_of`, обе подписи сошлись). Есть только у пиров с домашним
  сервером.

**Неймплейт-бабл над головой** ([../actors/nameplate/](../actors/nameplate/), класс `Nameplate`) —
«view-bubble»: крупно `display_name`, а под ним, **при наличии**, вторая строка `nick@domain`
с иконкой-галочкой статуса. Технически это UI-плашка (PanelContainer) через `SubViewport` на
billboard-`Sprite3D` — как речевой бабл, но обновляется лениво (`UPDATE_ONCE` на изменение).
`RemotePlayersView` подхватывает адрес из сигнала `identity_verified` и из
`verified_address_of` при создании капсулы (личность могла подтвердиться раньше).

**Иконки статуса** ([../resources/status-icons/](../resources/status-icons/)) — единый набор
из трёх (`checkmark`/`warning`/`err`), общий для 3D и 2D через хелпер `StatusIcons`
([../actors/nameplate/status_icons.gd](../actors/nameplate/status_icons.gd), enum
`Status` + `texture(status)` + `color(status)`). Иконки монохромные — цвет-семафор
накладывается через modulate: `VERIFIED` зелёный, `WARNING` жёлтый, `ERROR` красный.
Где используются: галочка `VERIFIED` — на неймплейте и в списке «Пользователи»; `warning`/`err` —
легитимность аватара над неймплейтом (см. [avatars.md](avatars.md)).

Оговорка переходного периода: `user_id` (ключ таблицы рангов, docs/ranks.md) пока остаётся
самозаявленным случайным id и живёт параллельно с сертификатом — подтверждённый адрес
показывается рядом, но ранги к нему ещё не привязаны. Привязка рангов к федеративному
адресу — следующий шаг (см. дорожную карту).

## Аккаунты и сессии

- **Регистрация**: `nickname` (`[a-z0-9._-]{3,32}`, нормализуется в lowercase) + пароль.
  Может быть закрыта конфигом (`registration_open = false`) — тогда только владелец сервера
  добавляет аккаунты (пока — руками в БД; инвайты — будущее).
- **Пароли** — scrypt (stdlib hashlib), соль на пользователя.
- **Сессии** — случайный токен (256 бит); в БД хранится только его SHA-256. Один и тот же
  механизм для REST (`Authorization: Bearer <token>`) и веб-морды (HttpOnly cookie).
  TTL настраивается, продлевается при использовании.

## REST API (v1)

Все ответы — JSON; ошибки — `{"error": {"code": ..., "message": ...}}`.

| Метод и путь | Auth | Что делает |
|---|---|---|
| `GET /.well-known/vrweb` | — | discovery: сервер, features, config, ключи |
| `POST /api/v1/register` | — | `{nickname, password}` → `{address, access_token}` |
| `POST /api/v1/login` | — | `{nickname, password}` → `{address, access_token}` |
| `POST /api/v1/logout` | Bearer | отозвать текущий токен |
| `GET /api/v1/account` | Bearer | `{address, nickname, created_at}` |
| `POST /api/v1/identity/certify` | Bearer | `{public_key}` → `{certificate, certificate_json, signature, key_id}` |

`address` всегда полный (`nick@domain`) — клиенту не нужно самому склеивать.

## Веб-морда

Серверный рендеринг (Jinja2), без JS-фреймворков: `/` (инфо о сервере и его features),
`/register`, `/login`, `/account` (адрес, дата регистрации, выданные ключи, logout).
Стили — один встроенный CSS. Это осознанно минимальная база: новые страницы добавляются
по мере роста функционала.

## Хранение и конфигурация

- **SQLite** в `data/homeserver.db` (таблицы `users`, `sessions`, `user_keys`), схема
  создаётся при старте. Ключ сервера — `data/signing_key.pem` (генерируется при первом
  запуске; потеря ключа = невалидность старых сертификатов, бэкапить вместе с БД).
- **Конфиг** — `homeserver.cfg` (INI, как `config/build.private.cfg` в клиенте) +
  переопределение переменными окружения (`VRWEB_DOMAIN`, `PORT`, …). Обязателен только
  `domain`. Пример — [../homeserver/homeserver.example.cfg](../homeserver/homeserver.example.cfg).

## Протокол сигналинга в монолите

Эндпоинт `GET /signal` (WebSocket) — протокол в точности как у standalone-сервера:
`join` / `offer` / `answer` / `candidate`, см. [../signaling/README.md](../signaling/README.md).
Клиенту достаточно сменить адрес на `wss://domain/signal`. Аутентификация на сигналинге
пока не требуется (комнаты эфемерны) — но т.к. это модуль монолита, привязка сессий —
дешёвая точка расширения.
