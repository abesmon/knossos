# SpaceLayout — единый генератор пространства (фаза F)

**Алгоритм формы и раскладки комнат** — `SpaceLayout`. Это **единый продакшн-генератор**
фазы геометрии F: даёт комнатам **нерегулярную форму из пентамино** и раскладывает их
**примыканием от корня**. Одна и та же раскладка кормит **двух потребителей**:

- **реальное 3D-пространство**, по которому ходит игрок — из клеток `SpaceLayout`
  [world_generator.gd](../scripts/world_generator.gd) строит полы, стены с проёмами-дверьми,
  полы-коридоры и объекты (см. «Перенос в 3D» ниже);
- **отладочный вид сверху** — площадка [geometry_debug.tscn](../scenes/geometry_debug.tscn)
  рисует ту же раскладку 2D-проекцией (щупать алгоритм, не запуская весь мир).

Вход — артефакт топологии (§1 [html-to-3d-topology.md](html-to-3d-topology.md)), без единой
3D-координаты. Всё пространство сочиняется здесь, детерминированно от `seed`.

Файлы:
- [scripts/space_layout.gd](../scripts/space_layout.gd) — чистый алгоритм (`SpaceLayout`),
  топология+seed → раскладка на сетке. Без рендера, тестируется отдельно.
- [scripts/world_generator.gd](../scripts/world_generator.gd) — **3D-потребитель**:
  раскладку `SpaceLayout` → геометрия комнат (полы по клеткам, стены по периметру футпринта,
  проёмы на дверях, объекты вдоль внутренних дорожек). Свою раскладку НЕ считает.
- [scenes/geometry_top_view.gd](../scenes/geometry_top_view.gd) — 2D-вьюер
  (`GeometryTopView`): сетка, футпринты комнат, описывающие прямоугольники, дорожки;
  панорама/зум/клик. Чисто отладочная проекция.
- [scenes/geometry_debug.gd](../scenes/geometry_debug.gd) + `.tscn` — площадка: адрес → HTML →
  топология → `SpaceLayout` → вьюер. Кнопка `⟳` — новый seed на той же топологии.

Площадка `geometry_debug` не подключена в навигацию — запускается вручную (Run Specific
Scene / F6), как [topology_debug.tscn](../scenes/topology_debug.tscn).

### Параметры (ползунки в площадке, поля `SpaceLayout`)

- **`alcove_fill_chance`** [0..1] — вероятность закрыть найденный алков (пустую клетку-«зубец»
  внутри формы) частью комнаты (§3.5).
- **`pull_to_parent_chance`** [0..1] — вероятность подтянуть комнату (с поддеревом) к родителю
  по свободному месту после раскладки (§6).
- **`route_base_excess`** [×1..3] — базовая «избыточность» внутреннего маршрута сквозь комнату (§7).
- **`route_deadend_mult`** [×1..4] — множитель длины маршрута для тупиков-листьев (§7).
- **`route_pentomino_mult`** [×0..1.5] — вклад пентамино: множитель = `max(1, pent_count × это)` (§7).

Ползунки меняют раскладку **на том же seed** — виден чистый эффект параметра.

## Алгоритм

### 1. Потребность комнаты

`need = |объекты комнаты| + запас_на_проход`. Запас (`_passage_slots`) — пока **заглушка,
всегда 1**; позже это функция от плотности/связности (сколько нужно свободного места, чтобы
ходить между объектами). Соединитель без наполнения → `need = 1`.

### 2. Форма комнаты

- `need ≤ 4` → один **прямоугольник** из `{2×2, 3×2}` (случайно). Прямоугольник свободно
  вращается (packer кладёт случайную ориентацию), поэтому «ширина/высота» не важны.
- `need > 4` → берём `max(2, ceil(need / 5))` **случайных пентамино** (1 пентамино = 5 клеток),
  то есть **не меньше 2**. Примеры: `5 объектов + 1 = 6` слотов → 1 пентамино не влезает (5 < 6)
  → **2** пентамино; `10 + 1 = 11` → `ceil(11/5) =` **3** пентамино; одиночная маленькая
  пентамино-комната (`need=5`) тоже получает **2** (минимум).

12 свободных пентамино (F I L N P T U V W X Y Z) заданы в `_build_pentominoes`; для каждого
генерируются все уникальные ориентации (4 поворота × 2 отражения, дедуп по ключу множества
клеток).

### 3. Упаковка → описывающий прямоугольник

Детали пакуются **максимально компактно** в один полиомино, чтобы минимизировать «хвост»
(пустые клетки описывающего прямоугольника). Жадный алгоритм «снизу-влево с приращением»
(`_pack_pieces`):

1. первая деталь — в начало координат (случайная ориентация);
2. каждая следующая перебирается по всем ориентациям и позициям в окрестности уложенного,
   должна **примыкать гранью** к уже уложенному (связность ⇒ компактность) и не пересекаться;
3. из допустимых выбирается та, что даёт **наименьшую площадь описывающего прямоугольника**
   (с небольшим бонусом за квадратность: `score = area*2 + |w−h|`).

**Описывающий прямоугольник упаковки = площадь, необходимая комнате** (`dims`). На реальных
наборах 3 пентамино обычно паркуются в ~94% заполнения (15 клеток в 4×4).

### 3.5. Закрытие алковов (`alcove_fill_chance`)

После упаковки в форме остаются **алковы** — вогнутости с **симметричными краями**. Алков
(`_find_alcove_runs`) — это непрерывный ряд пустых клеток (по строке или столбцу), у которого:
**на обоих концах стенки** (фланги) и **сплошная задняя стенка** с одной перпендикулярной
стороны. Каждый такой ряд с вероятностью `alcove_fill_chance` достраивается частью комнаты
(`_close_alcoves`); клетки приписываются к соседней детали (раскраска), `cells` растёт, `dims`
не меняется. Заполнение **итеративное**: закрытый ряд открывает доступ к лежащему за ним, так
закрываются и глубокие симметричные ниши. Решение принимается по разу на каждый отдельный ряд.

```
x o o x        x o o o x      — заполняются целиком (края ровные/симметричные)
x x x x        x x x x x

x o o o o      — заполнится только нижний ровный ряд (x o o o x),
x o o o x        верхний останется торчать хвостиком: правый край неровный (не симметричен)
x x x x x
```

### 4. Раскладка комнат (от корня, с резервом улиц под дорожки)

Обход дерева от корня; корень в `(0,0)`. Дети ставятся в **случайном** порядке. Раскладка
работает на **реальных клетках** (футпринтах), не на описывающих прямоугольниках — комнаты
могут входить «зубцами». Для каждого ребёнка (`_place_rooms` / `_place_and_connect`):

- **дверь к родителю** (`_place_adjacent`): перебираются **все ориентации** ребёнка (4 поворота
  × 2 отражения футпринта — «вращаем комнату») и позиции у родителя; кандидат годится, если его
  клетки **граничат гранью с родителем** (`_adjoins`), **не пересекаются** ни с чем
  (`_overlaps_cells`) и **НЕ касаются чужих комнат** (`_touches_other`) — между комнатой и
  не-родителями всегда остаётся улица ≥1 клетки. Так **по построению резервируется место под
  дорожки**: коридоры пойдут по этим улицам в обход комнат. Из годных берётся случайная.
- **запасной путь с маршрутом** (`_fallback_route`): если дверью примкнуть негде, ищем ближайшее
  **изолированное** место (вокруг — улица) и проверяем BFS, что до родителя есть **маршрут в
  обход комнат**. Кандидаты перебираются по возрастанию расстояния ⇒ берётся первый
  маршрутизуемый = **коридор минимальной длины**. Это и есть «менять layout, чтобы освободить
  место под дорожку»: комната отодвигается ровно настолько, чтобы прошёл коридор.

Выбранная ориентация мутирует комнату (`pos`/`pieces`/`cells`/`dims`).

### 5. Дорожки (пути по центрам клеток, в обход комнат)

- **дверь** (примкнувшая связь): путь = две граничащие клетки родителя и ребёнка
  (`_door_path`); лежит на их общем стыке, третью комнату не пересекает.
- **коридор** (запасная связь): кратчайший путь BFS по **свободным** клеткам
  (`_route_corridor`) — все комнаты, кроме точек входа, считаются препятствиями, поэтому коридор
  **гарантированно не пересекает комнаты**. Проложенные клетки коридора резервируются в `occ`
  маркером `CORR`, чтобы их не заняла комната, размещённая позже (это и устраняло остаточные
  пересечения); новые коридоры могут проходить по уже проложенным.
- если маршрута нет совсем (родитель замурован своими комнатами) — связь помечается `unrouted`:
  вьюер рисует **красные кольца** у обеих комнат **без линии**, чтобы НЕ изображать пересечение.

Вьюер рисует пути **ярко** по центрам клеток (`+0.5`): розовый — дверь/коридор примкнувшей
связи, **оранжевый** — запасной коридор, **красный разрыв** — `unrouted`.

Инвариант проверен прогоном на 40 сидах плотного дерева: **0 пересечений дорожками комнат**,
0 неразведённых связей.

### 6. Подтягивание к родителю (`pull_to_parent_chance`)

После раскладки и дорожек остаётся свободное место — комнату можно подвинуть ближе к родителю.
Проход сверху вниз по дереву; для каждой не-корневой комнаты с вероятностью
`pull_to_parent_chance` запускается `_pull_subtree`:

- двигаем комнату **вместе со всем её поддеревом** (дети едут с родителем) жёстко-телесно
  единичными шагами к центру родителя (`_pull_steps`), пока не упрёмся;
- чужие комнаты и чужие коридоры — **препятствия** (`_collides`), внутренние коридоры поддерева
  едут вместе с ним ⇒ пути не ломаются и комнаты не накладываются;
- после сдвига связь с родителем **переразводится** (дверь или коридор BFS); если переразвести
  не удалось — сдвиг откатывается целиком.

Порядок сверху вниз: сперва подтягивается родитель, затем его дети — к новому положению. Прогон
на 40 сидах: при 100% площадь bbox раскладки падает на ~13%, пересечений/наложений по-прежнему 0.

### 7. Внутренние маршруты движения сквозь комнату (зелёные)

Для каждой комнаты считаются маршруты перемещения по её клеткам (`_compute_routes`, кладутся в
`rooms[id].routes` — **список путей**, рисуется **зелёным** по центрам клеток). **Вход** комнаты —
её клетка на связи от родителя (у корня родителя нет — берётся центр комнаты); **выходы** — её
клетки на связях к детям.

- **проходная** (есть дети): **N путей**, N = число детей — по одному **из входа к каждому
  выходу** (к ребёнку). Пути к разным детям **независимы** и **могут пересекаться между собой**;
  самопересечений внутри одного пути нет.
- **лист** (нет детей): один путь — **петля вход→…→вход** («зайти, походить, вернуться»);
  избыточность сразу ×`route_deadend_mult`.

Каждый путь — модель «верёвки». Длина:
```
target ≈ реальная_длина × route_base_excess × (route_deadend_mult если лист) × пентамино_фактор
пентамино_фактор = max(1, pent_count × route_pentomino_mult)   (иначе 1)
реальная_длина = кратчайший путь вход→выход (для листа 2 × эксцентриситет входа)
```
Натянутая верёвка = кратчайший путь; с ростом `target` она **извивается по комнате, не пересекая
себя** (self-avoiding walk), вплоть до заполнения комнаты — дальше длину не нарастить, **предел =
площадь комнаты**. Прокладка (`_simple_path` / `_simple_cycle`) — рандомизированный DFS с
отсечением ходов, после которых выход/старт становится недостижимым (чтобы верёвка не «замуровала»
себе путь), длина подбирается как можно ближе к `target`; запас — кратчайший BFS. Прогон на 40
сидах: **0 самопересечений**, все пути внутри клеток комнаты и связны; число путей = числу детей
(лист — один путь-петля).

## Перенос в 3D (world_generator.gd)

`SpaceLayout` — единый генератор: его раскладку потребляет реальное пространство.
[world_generator.gd](../scripts/world_generator.gd) переводит клетки в геометрию
(`_consume_layout` + `_build_room`):

- **сетка → метры**: клетка `SpaceLayout` = `GRID` (3 м); мировая позиция клетки —
  `_cell_world`, корень сдвинут к началу координат (`_shift`).
- **полы**: по GRID-квадрату на каждую клетку футпринта (и на каждую клетку коридора).
- **стены**: по периметру футпринта (`_wall_runs`) — граничные рёбра клеток склеиваются в
  прямые пробеги; на месте **дверей** (рёбра, через которые проходит связь, `_door_edges`)
  пробег рвётся → **проём**. Стена сдвинута внутрь на полтолщины, как в старой раскладке.
- **двери/коридоры**: концы пути связи (`corridors[*].path`) лежат в комнатах — там проёмы;
  середина пути — свободные клетки → полы-коридоры. `adjacent`-связь = прямая дверь между
  примкнувшими комнатами; иначе коридор по улицам. Коридор **огорожен стенами** по открытым
  краям клеток (`_add_corr_wall`): стена не ставится там, где улица продолжается (соседняя
  коридорная клетка) или примыкает комната (у неё своя стена с дверью) — так проходы открыты,
  а дорожка не висит без стен.
- **объекты — вдоль внутренних дорожек** (`routes` комнаты). Каждый объект занимает 1 клетку,
  **притянут к её краю** (к стене / прочь от дороги) и смотрит на центр клетки = на дорогу.
  Слоты считаются заранее (`_object_slots`): обход маршрутов от входа по часовой стрелке, для
  каждой клетки дорожки берётся клетка **по левую руку** (рядом с дорогой, не на ней) — объект
  встаёт в неё, притянутый к дальнему краю; если её нет/занята — на саму клетку дорожки,
  притянутый к левой стене (там обычно стена ⇒ «висит на стене, смотрит на дорогу», не загораживая
  центр). Объект **никогда не перекрывает проход** (`_wall_clear`): не ставится на клетку-дверь и
  не притягивается к стене с дверью (включая ±1 клетку вдоль неё — чтобы даже широкая панель не
  нависала над проёмом). Объекты раскидываются по N слотам; если объектов больше — лишние идут **стопкой друг над
  другом** в передних слотах (слот i получает `base+(1 если i<rem)`, `base=⌊M/N⌋`, `rem=M%N`;
  первый в группе — сверху). Спавн — «у первого объекта, лицом к нему».
- **вход + вывеска-заголовок**: проём от родителя (`_room_entrance`) несёт ленту-заголовок
  секции, читаемую с обеих сторон; спавн игрока — «у первого объекта» или «через титульный проём».

Проверка инвариантов геометрии: [tests/test_space_geometry.gd](../tests/test_space_geometry.gd).

## Что дальше (нереализовано)

- **Запас на проход** — настоящая функция вместо заглушки `1` (плотность объектов, связность).
  В 3D сейчас комната сайзится по числу объектов, а не по их метрическим габаритам.
- **Замурованный родитель** (`unrouted`): сейчас связь просто помечается разрывом; можно
  раздвигать уже стоящие комнаты, чтобы пробить улицу (полноценное «менять layout»).
- **Ширина коридора** > 1 клетки.
- **Внутренние маршруты в 3D**: `routes` (зелёные пути сквозь комнату) пока только в виде сверху;
  в мире не материализуются (пол сплошной — игрок ходит свободно).
- **Минимизация длины коридора** сейчас приближённая (берётся ближайшее маршрутизуемое место по
  расстоянию, не по фактической длине BFS) — можно сравнивать реальные длины маршрутов.
