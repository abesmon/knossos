# HTML → 3D: топология пространства

Алгоритм трансляции произвольного HTML-документа в навигируемое 3D-пространство.
Документ описывает **логический уровень** (как из HTML получается топология) и контракт,
по которому эту топологию забирает генератор геометрии.

---

## 0. Главный принцип: топология ≠ геометрия

Трансляция разбита на два независимых уровня, между которыми лежит явный
сериализуемый артефакт.

1. **Топология (этот документ).** Из HTML строится *функциональное* описание
   пространства: какие есть комнаты, что в них стоит, как они соединены, какие
   у объектов функции. Ни одной 3D-координаты на этом уровне нет.
2. **Геометрия (отдельная фаза, отдельный документ).** Генератор берёт топологию +
   `seed` и сочиняет конкретное пространство: координаты, размеры, материалы,
   раскладку дверей по стенам, стиль.

Дерево DOM определяет **функцию** пространства, а не его форму. Геометрию определяет
генератор. Топология может передавать генератору необязательные *хинты* (вес комнаты,
исходные цвета, степень соединителя, семантический тег), но генератор волен любой хинт
проигнорировать.

**Практическое следствие.** Генератор пространства — сменный. Одна и та же топология
может быть отрендерена «поместьем», «островами», «станцией» разными генераторами.
`seed` выбирает не только раскладку, но потенциально и сам стиль генерации.

## 1. Контракт: промежуточный артефакт

Выход топологической фазы — сериализуемое описание **без единой 3D-координаты**.
Это точка тестирования (отлаживается как обычный JSON, без 3D) и граница, за которую
геометрия не протекает.

Концептуальная схема (формат финализируется при реализации):

```
Space {
  rooms: Room[]
  labels: { [anchorId]: roomRef | objectRef }   // цели якорей #id
  typography: { base_px }         // базовый кегль текста страницы (см. §13)
  document: { bg?, bg_image?, fg? } // визуальный паспорт <body> -> небо/земля/палитра (world-visualization.md)
}

Room {
  id
  kind: "room" | "connector"      // connector = бывший общий предок ≥2 комнат
  objects: Object[]               // наполнение + декор
  children: roomRef[]             // вложенные/смежные комнаты (что во что собралось)
  entry: roomRef                  // точка входа в это поддерево (см. §8 — открыто)
  hints: {                        // всё опционально, генератор может игнорировать
    weight: int                   // число листьев-объектов в кластере
    degree: int                   // для connector: число дочерних комнат
    doc_order: int                // очерёдность кластера в документе (порядок чтения; меньше = выше на странице)
    semanticTag: string?          // main/section/article/nav/...
    css: { bg?, border?, ... }    // визуальные признаки исходного контейнера
  }
}

Object {
  id
  type: "image" | "text" | "heading" | "input" | "button" | "list" | "table" | "media"
      | "form" | "code" | "quote" | "figure" | ...   // code=<pre>, quote=<blockquote>
  function: Transition | null     // capability перехода, навешивается поверх type
  content: { ... }                // src, текст, уровень заголовка и т.п.
                                  //   heading: level (1..6)
                                  //   image:   width_px?, height_px? (абсолютные из HTML, §13)
  hints: { ... }
}

Transition =
  | { kind: "navigate", href: url }      // другая страница: fetch + swap пространства
  | { kind: "external", uri: uri }       // нестандартная схема: отдаём ОС (OS.shell_open)
  | { kind: "teleport", target: anchorId } // якорь на этой же странице: метка
  | { kind: "back" }                       // из стека истории навигации
```

**`<video>` → видео-плеер.** Стандартный HTML-тег `<video>` классифицируется как объект
`media` с пометкой `media_tag: "video"` и `src` (из `<video src>` или первого вложенного
`<source src>`, по стандарту HTML; плюс `autoplay`/`loop`/размеры). Геометрия строит из него
**тот же `VrwebVideoScreen`**, что и стандартный тег VRWML `<VRWebVideoScreen>` — то есть HTML-видео
проигрывается реальным плеером приложения (FFmpeg-декод в текстуру, прогрессивная докачка,
синхронизация по сети). Без аддона FFmpeg или без `src` — деградирует до статичной заглушки
`▷`. Прочие медиа (`audio`/`iframe`/`canvas`/`embed`) пока остаются заглушками. Подробно —
[video-player.md](../client/video-player.md).

**Классификация `href` по схеме** (`_transition_for` → `_scheme_of`, RFC 3986):
`#…` → `teleport`; `javascript:` → игнор (`null`); `http`/`https`/относительные (без
схемы, включая protocol-relative `//host`) → `navigate`; **любая другая схема**
(`mailto:`, `tel:`, `sms:`, `magnet:`, кастомные app-схемы) → `external`. Это не
веб-страница, а намерение для ОС: `main._activate_transition` отдаёт `uri` в
`OS.shell_open`, который сам решает, что запустить (почтовик, телефон, торрент-клиент,
стороннее приложение). Внешние порталы — фиолетовые (`⮺`), inline-ссылки тоже.

## 2. Базовая модель

Три сущности на логическом уровне:

- **Объект** — атомарный контент, стоит *внутри* пространства: изображение, текстовый
  блок, заголовок, поле ввода, кнопка, список, видео.
- **Комната** — пространство, образованное кластером объектов.
- **Соединитель** — пространство, возникающее там, где встречаются два независимых
  кластера. Соединитель — это **тоже комната** и может нести декор (см. §5).

Ключевая переформулировка: **комната — это не узел дерева, а множество листьев-объектов.**
Какой именно `<div>` в цепочке «является» комнатой — неважно: цепочка одиночных
контейнеров всё равно схлопывается. Комнату определяет её набор листьев; контейнеры —
это строительные леса, которые сносятся.

## 3. Классификация узлов

Классификация идёт **по содержимому**, а не по тегу. У `<a>` две ортогональные вещи:

**Что это как объект** — берётся из содержимого по общему правилу:
- пустой / чисто текстовый `<a>` → объект-текст (вырожденно: «голая дверь», если своего
  визуального наполнения нет, и тогда сам переход и есть представление объекта);
- `<a><img></a>` → объект-картина;
- `<a>` внутри списка → интерактивный текст / кнопка.

**Что это как поведение** — наличие `href` навешивает на получившийся объект
*функцию перехода* (`function: Transition`).

Поэтому **«портальность» — это не класс узла, а capability, навешиваемая на любой объект.**
Классификатору не нужна отдельная ветка под ссылки: он классифицирует `<a>` по
содержимому как любой узел, а `href` добавляет функцию. Это автоматически чинит
гомогенный повтор (§7): список ссылок — коллекция объектов, у каждого своя функция
перехода; свёртка в объект-список ничего не ломает, функция едет с каждым элементом.

Невизуальное (`script`, `style`, `meta`, `link`) и **невидимое для глаза** на этой фазе
выкидывается (`_is_hidden` в [topology_builder.gd](../../scripts/topology_builder.gd)).
Основной путь — вердикт **мини-каскада** (`computed.hidden`, [css-cascade.md](css-cascade.md)):
он учитывает внешние таблицы и классы, поэтому `.sr-only`/`.visually-hidden` и скрытые
меню теперь ловятся. Layout-пасса по-прежнему нет, поэтому признаки — те же надёжные
CSS-декларации (по каскаду, а без резолвера — только инлайновые):

- `display:none`, `visibility:hidden`;
- `opacity` ≈ 0 (нулевая альфа);
- схлопнутый бокс `width:0` / `height:0`;
- унос за экран: `position:absolute|fixed` + `left`/`top` ≤ −999px, либо `text-indent` ≤ −999px;
- обрезка в ноль: `clip-path:inset(100%)`, старый `clip:rect(0,0,0,0)`;
- атрибуты `hidden`, `aria-hidden="true"`, `<input type="hidden">`.

Относительные единицы (`%`, `em`) в офф-скрин-проверке намеренно игнорируются — их не
перевести в метры, а ложно скрыть видимое (`min-width:0`, `left` без `position`) дороже,
чем пропустить редкий невидимый элемент.

## 4. Контракция: схлопывание «с конца» (bottom-up)

> **АМЕНДМЕНТ (заменено).** Эта секция описывает первую модель, где комнаты рождались из
> **вложенности DOM-контейнеров**. Актуальная модель построения комнат — **кластеризация**
> ([clustering.md](clustering.md)): комнаты из дерева кластеров по разным сигналам, внутри
> кластера дробители режут RichText. (Промежуточная модель «только по заголовкам» —
> [content-sectioning.md](content-sectioning.md) — тоже заменена ею.) В актуальной реализации
> ([topology_builder.gd](../../scripts/topology_builder.gd)) DOM-контракция удалена целиком:
> комнату создаёт **только заголовок** (явный `<h1>..<h6>` или визуальный — эвристика
> крупного/жирного/отдельно стоящего блока), вложенность комнат — из **рангов заголовков**,
> а внутри секции лежит **плоский список** терминальных объектов. Текст ниже сохранён как
> история проектирования (что не сработало и почему — плоские заголовки в вики ломали
> контракцию). Инварианты объектной модели (§3, §7) и масштаб (§13) остались в силе.

Это **главный алгоритм** — насколько умно схлопывается div-soup, ровно настолько хорошо
ощущается любой сайт. Подход — агломеративная кластеризация по дереву, снизу вверх.

1. Берём листья-объекты (картинки, кнопки, текст, …).
2. Над ними должна быть как минимум одна комната.
3. Поднимаемся вверх по дереву, сливая объекты в кластер, пока не встречаем границу.
4. Цепочка контейнеров с одним значимым ребёнком — это путь; схлопывается (контейнер не
   «является» комнатой, он строительные леса).
5. Несколько объектов под общим предком можно объединить в одну комнату (с порогом-`threshold`,
   подбирается на реальных сайтах).

**Правило остановки слияния.** Если у узла оказывается ≥2 уже сформированных комнаты-ребёнка
— этот узел становится **соединителем**, и кластеры вверх дальше **не сливаются**. Так
«соседняя комната никогда не объединяется» получается само собой: два кластера, пришедшие
с разных концов дерева, встречаются в общем предке, и предок их соединяет, а не сливает.

**Блокировка схлопывания по CSS.** Контейнер с фоном / рамкой / заметным паддингом несёт
визуальную информацию (стены, пол) даже если структурно «пустой». Такие признаки уезжают
в `hints.css` и могут блокировать схлопывание контейнера в ничто.

### 4.1 Неветвящийся узел не плодит сущностей

Главный источник «лишних» комнат — контейнеры-обёртки вокруг одного-единственного
элемента. Цепочка `div > div > div > <ul>` не должна давать четыре вложенные комнаты, где
в каждой по одному объекту. Узел, у которого **нет ветвления** (≤1 дочернее пространство),
лежит на *пути*, а не на границе комнаты, поэтому новой сущности не создаёт. Два частных
правила (оба — следствие §2 «комната = множество листьев, контейнеры = леса»):

- **Правило A — одиночный лист не кристаллизует комнату.** Контейнер без дочерних комнат и
  с ровно одним свободным листом комнатой не становится — лист всплывает наверх как loose и
  кластеризуется с соседями на ближайшем выжившем пространстве. Так распускаются «леса»
  вокруг одиночного списка / картинки / текста (`div.vector-menu-content > ul`), и заодно
  **тонкие сиблинги** (по одному листу в каждом, типичная навигация `nav > div > a`,
  `div > a`, …) собираются в **одну** комнату, а не в россыпь комнат-одиночек.
- **Правило B — линейная цепочка вливается, а не вкладывается.** Узел с ровно одной дочерней
  комнатой и собственными свободными листьями вливает эти листья *в* ту комнату и
  схлопывается, вместо того чтобы оборачивать комнату в комнату.

**Стоп-условия для A и B** (узел сохраняется как отдельное пространство): визуальный CSS-box
(стена/карточка) на узле или на дочерней комнате; семантический тег (`section`, `article`,
`nav`, … — автор явно разметил пространство); ребёнок-**соединитель** (реальная точка
ветвления — в неё контент сверху не вливаем). Якоря (`id`) схлопнутых контейнеров не
теряются: переносятся на выжившую комнату/объект (`labels`).

**Почему это не ломает инвариант «соседи не сливаются».** Правила A/B срабатывают только на
неветвящихся узлах (≤1 дочерняя комната). Где есть ветвление (≥2 комнаты) — узел становится
соединителем, и независимые кластеры соседей по-прежнему не сливаются (правило остановки
выше). Кластеризуются только *тонкие* листья-одиночки, а не самостоятельные пространства.

## 5. Свободные листья при ветвлении

> **АМЕНДМЕНТ (заменено).** Вся логика «заголовок — декор, не образующий комнату» и
> костыли вокруг `mw-heading` относятся к старой DOM-модели и **сняты**: в актуальной
> модели заголовок, наоборот, — **единственный** источник комнаты и её ранга
> ([content-sectioning.md](content-sectioning.md) §3–§4, §6). Текст ниже — история.

Узел стал соединителем (≥2 комнаты), но кроме комнат под ним болтаются ещё и свободные
листья напрямую (`<h1>`, `<img>` рядом с двумя `<section>`).

**Правило:** свободные листья прилипают к самому соединителю и **декорируют его**. h1
становится табличкой над проходом, img — картиной на его стене. Соединитель — это
полноценное пространство, оно может нести наполнение. Без этого правила объекты повисают
при контракции.

Общий инвариант контракции: **каждый объект после свёртки принадлежит ближайшему
выжившему пространству-предку** (комнате или соединителю).

**Заголовок-обёртка не образует комнату.** Заголовок (`<h1>`–`<h6>`) по семантике — это
подпись к контенту, который идёт *после* него, а не самостоятельное наполнение, «якорящее»
собственное пространство. Поэтому контейнер, у которого нет дочерних комнат, а все
свободные листья — только заголовки (типичный случай: `<div class="mw-heading"><h3>…</h3></div>`
в вики-разметке), **комнатой не становится**. Его заголовки всплывают наверх как свободные
листья и прилипают к ближайшему выжившему пространству-предку — там, где собирается контент,
который этот заголовок вводит. Иначе обёртка вокруг заголовка боксится в отдельную комнату на
уровень *ниже* соседних абзацев (которые как inline-контент всплывают выше), и заголовок
оказывается вложенным под текст вместо того, чтобы стоять с ним рядом первым объектом. Как и
любое схлопывание, это правило блокируется визуальным box'ом обёртки (§4): фон/рамка на
контейнере-заголовке сохраняют его как пространство.

## 6. Соединитель и его степень

Соединитель — единый логический тип. На топологическом уровне степень узла (сколько
комнат он соединяет) ничего не ломает: коридор с пятью дверьми (как в поместье, где вдоль
прохода идут спальни) — это нормально. Веб — не человеческое жилище, и пространства могут
выглядеть нечеловечески; это не проблема.

Степень (`hints.degree`) передаётся вниз **как подсказка генератору геометрии**, а не как
тип: degree-2 естественно тянет на прямой проход, degree-5 — на проход с дверьми вдоль
стены или на зал. Генератор волен это использовать или нет.

## 7. Гомогенный повтор → объект-коллекция

Некоторые контейнеры должны автоматически становиться **цельным объектом**, а не
множеством комнат. Список (`<ul>`/`<ol>`) не может быть комнатой — это объект-список,
размещённый *в* комнате.

Детектор гомогенного повтора: контейнер с множеством структурно-похожих детей сворачивается
в один объект-коллекцию, а не в N комнат. Содержимое каждого элемента **линеаризуется в
прогоны (`runs`)** — ровно как абзац (§3): любая `<a href>` внутри элемента становится
отдельным кликабельным прогоном со своей `Transition`, обычный текст склеивается. Так двери
не теряются, и работают даже несколько ссылок в одном пункте: элемент списка несёт
`{ text, runs }` (`text` — плоский текст для дешёвого рендера без ссылок, `runs` — прогоны).
В геометрии список со ссылками рендерится тем же кликабельным актором, что и абзац (RichPanel);
список без ссылок остаётся дешёвой текстовой табличкой.

**Прогоны несут не только текст.** `run` бывает двух видов: текстовый
`{ text, function? }` (по умолчанию, без `type`) и **картиночный**
`{ type:"image", src, alt, width_px?, height_px?, function? }`. Картинка внутри ячейки/пункта
(`<td><img>`, `<li><img>`, `<td><a><img></a>`) раньше **молча терялась** — линеаризатор знал
только текст и ссылки, а `<img>` давал пустой прогон. Теперь `<img>` (в т.ч. завёрнутый в
`<a href>` — тогда несёт ещё и `function`) превращается в картиночный прогон, как объект-image
(§13). В геометрии список/таблица с картинкой-прогоном уходит в RichPanel (а не в плоскую
табличку), даже без ссылок. **Растровый инлайн реализован:** RichPanel вставляет настоящую
текстуру прямо в поток текста через `RichTextLabel.add_image` — серой заглушкой нужного размера,
пока идёт докачка, а по приходе текстуры **перезаполняет лейбл целиком** из текущего состояния
(`_populate_label` дёргается заново по `_on_image_result`; результаты кэшируются в `_img_tex`
по url, запрос на url — один). Текстуру тянет тот же `ImageLoader`, что и объекты-картинки;
`setup` получает `image_loader`/`base_url`. Размер берётся из `width_px`/`height_px` HTML,
недостающее измерение — из пропорций реальной текстуры, ширина капится под колонку текста.
После прихода картинок панель пересчитывает высоту (`_refit`), так что контент не обрезается.

**Картинка-ссылка кликается прямо по изображению.** У инлайн-`[img]` нет своего meta-канала,
но клик игрока в RichPanel — это синтетический клик мышью во вьюпорте (см. `interact_at`/
`_push_click`), а `meta_clicked` срабатывает на любой контент внутри meta-региона. Поэтому
картинку-ссылку оборачиваем в `push_meta(idx, META_UNDERLINE_NEVER)` / `add_image` / `pop()` —
и переход идёт по самому изображению, без отдельной подписи-ссылки рядом. При наведении строка
статуса показывает `alt` картинки **и** куда ведёт ссылка (`aim_hint_at` склеивает `_probe_alt`
с `TransitionText.describe`). Если лоадера/`src` нет **или докачка не удалась** (`_img_tex[url] == null`),
картинка деградирует в текстовый токен `🖼 alt` (кликабельный, если это ссылка) — `alt` служит и
подписью «битой» картинки, и текстом статуса.

**Таблица — тоже цельный объект.** `<table>` (с её `<tr>`/`<td>`/`<th>`) — это единый
объект-таблица, стоящий *в* комнате, а не россыпь комнат из строк и ячеек. Иначе `<table>`
становилась бы соединителем, а каждый `<tr>` — отдельным пространством, что дробит логически
неделимую сетку. Строки собираются сквозь обёртки `<thead>`/`<tbody>`/`<tfoot>`; ячейка несёт
`{ text, runs, header }` — содержимое ячейки линеаризуется в прогоны так же, как элемент
списка, поэтому `<a>` внутри ячейки остаётся кликабельной ссылкой в мире; `<caption>` →
подпись таблицы.

## 8. ОТКРЫТЫЙ ВОПРОС: точка входа в составное поддерево

> **АМЕНДМЕНТ (закрыто).** В модели сегментации по заголовкам вопрос снят: секция *и есть*
> свой вход — заходишь через её заголовок-комнату, спускаешься в подсекции по рангам
> ([content-sectioning.md](content-sectioning.md) §4.1). `entry = сама секция`.

Единственный незакрытый пункт топологической модели.

Когда соединитель `C` пристыковывает проход к поддереву `S`, а `S` — это уже не одна
комната, а сам по себе соединитель с пятью комнатами, — **в какой именно узел `S`
открывается проход?** Топологически нужно выбрать вершину-порт. Иначе граф недоопределён:
«проход ведёт в подпространство» — это не адрес.

Рабочая гипотеза в логике «свёртки с конца»: каждое поддерево экспонирует наверх свой
корневой узел (верхний соединитель или верхнюю комнату) как `entry` — точку входа.
Тогда стыковка всегда идёт `C → S.entry`. Это требует подтверждения на реальных кейсах:
не всегда верхний узел поддерева — осмысленный вход (например, поддерево, где верх — это
тонкий соединитель, а смысловой центр — большой зал внутри).

Поле `entry` в контракте (§1) зарезервировано под это решение.

## 9. Переходы

Все переходы — это **объект + функция**, одна машинерия с разными функциями:

| Что в HTML | type объекта | function |
|---|---|---|
| `<a href="other.html">` | по содержимому (§3) | `navigate` → fetch + перегенерация + swap пространства |
| `<a href="#section">` | по содержимому | `teleport` → перемещение игрока к метке `section` |
| `<a href="mailto:…">` (любая не-http схема) | по содержимому | `external` → `OS.shell_open(uri)`, ОС запускает внешнее приложение |
| кнопка «назад» (из истории) | служебный объект | `back` → предыдущее пространство из стека |

**Якоря — это метки, без графа.** Цель `id="section"` — просто метка на той комнате/объекте,
в которую свёрнут соответствующий узел (`labels` в контракте). Разрешение происходит в
момент взаимодействия по таблице меток. Держать ради этого ребро/граф не нужно.

**Никакого глобального графа сайта не материализуется.** Клиент держит одно пространство
за раз, как браузер — одну страницу. Кросс-страничная ссылка = физический объект с функцией
`navigate`; взаимодействие = загрузка нового URL, перегенерация, свап. Из браузерной
аналогии естественно следует «дверь назад» (`back`) — портал в предыдущее посещённое
пространство из стека истории.

## 10. Вес комнаты (без рендер-пасса)

Размер комнаты — это **хинт**, не часть топологии. Считается тривиально при той же
bottom-up-свёртке:

```
weight(room) = |листья-объекты кластера|  (+ опц. вклад вложенных подкомнат)
```

Никакого `getBoundingClientRect` / Puppeteer для первой версии не нужно — это ложится на
раннее ограничение «HTTPRequest без JS execution». Генератор геометрии может взять `weight`
как косвенный признак размера, или проигнорировать и раскладывать размеры от `seed`, или
комбинировать.

## 11. Фазы

| Фаза | Что делает | Выход |
|---|---|---|
| **A. Нормализация** | парсим HTML в дерево, выкидываем невизуальное (`script`, `style`, `meta`) и невидимое для глаза — `_is_hidden`: вердикт мини-каскада `computed.hidden` ([css-cascade.md](css-cascade.md), включая скрытие классами) + инлайновые признаки и атрибуты (`display:none`, `visibility:hidden`, `opacity≈0`, нулевой бокс, офф-скрин, clip, `hidden`/`aria-hidden`); layout-пасса нет (см. §3) | очищенное дерево |
| **B. Классификация** | каждому узлу — type по содержимому; `href` → функция перехода поверх | размеченные узлы |
| **C. Сегментация по заголовкам** *(заменила контракцию)* | линеаризация контента в поток заголовков-с-рангами и объектов; стек по рангам строит дерево секций-комнат; внутри секции — плоский список объектов ([content-sectioning.md](content-sectioning.md)) | комнаты по оглавлению |
| **D. Декор соединителей** *(растворено в C)* | заголовок секции — первый объект её комнаты; контейнерный CSS-box едет в хинты ближайшей секции | пространства с наполнением |
| **E. Сборка топологии** | комнаты + соединители + вложенность + метки якорей + хинты | **артефакт §1** |
| **F. Геометрия** *(отдельный документ)* | `seed = hash(GLOBAL_SEED \| host \| topology_signature)` (не полный URL — топологически одинаковые страницы одного хоста дают тот же мир, см. [geometry-lab.md](geometry-lab.md)); единый генератор `SpaceLayout` (формы комнат из пентамино, примыкание от корня, коридоры по улицам) даёт клеточную раскладку; `WorldGenerator` строит из клеток полы, стены по периметру с проёмами-дверьми, полы-коридоры и объекты по стенам ([geometry-lab.md](geometry-lab.md)) | 3D-сцена |

Фазы A–E — топология. Фаза F — геометрия, потребляет только артефакт §1. Геометрия строится
не за один кадр, а порциями (тайм-слайс), чтобы тяжёлая страница не вешала главный поток —
см. [performance-streaming.md](../client/performance-streaming.md).

## 12. Что прототипировать первым

**Фаза C (контракция).** Это и есть весь продукт; A/B/D/E — относительно прямолинейны.
Гонять на десятке реальных сайтов разной природы: статичный блог, новостной портал,
SPA-дашборд (финальный DOM после JS), лендинг, документация. Там сразу всплывут пороги
(`threshold` слияния, правила схлопывания), которые в теории не угадать.

Отлаживать на чистом артефакте §1 (JSON), **без всякого 3D** — топология проверяется как
обычное дерево.

## 13. Масштаб объектов: единая метрика 1 м = 128 px

Весь мир размеряется **одной метрикой `PX_PER_METER = 128`**: 1 метр = 128 CSS-пикселей
страницы. От неё считается всё — кегль шрифтов, ширины панелей, размеры картинок. Это
абсолютный, а не относительный к тексту масштаб: «14px шрифт» — это всегда `14/128 ≈ 0.11 м`
на любой странице, картинка `width=300` — всегда `300/128 ≈ 2.34 м`. Поскольку у текста и
картинок **общая линейка**, картинки больше не нужно «подгонять» под текст: их размеры просто
сопоставимы по построению.

**Топология (фаза E)** измеряет страницу в её собственных пикселях, без 3D:
- `typography.base_px` — средневзвешенный (по длине текста) кегль основного текста.
  Текст без явного `font-size` считается базовым (16px). База нужна **только** для детекции
  заголовков (ранг по относительному размеру) и расчёта реального css-кегля элементов через
  `em`-множители — **не** для перевода px→м.
- `content.width_px` / `height_px` у картинок — абсолютные размеры из `width`/`height`
  или inline-`style` (`px` или «голое» число). Относительные/процентные (`em`, `%`)
  игнорируются — в артефакт попадает только то, что переводимо в метры.

**Геометрия (фаза F)** берёт `m_per_px = 1 / PX_PER_METER` и размеряет им всё:
- кегль текста: `css_px / 128` (Label3D через `font_size × pixel_size`, RichPanel через
  `font_px / PIXEL_PER_METER` — обе линейки сведены к одному `128`);
- заголовок — `base_px × em(level)` (дефолты браузера h1..h6: 2.0 … 0.67), затем `/128`;
- картинка — `width_px`/`height_px` из HTML `/128`; **без размеров** сначала получает
  fallback-ширину `ImagePanel.BASE_WIDTH`, а после загрузки текстуры уточняется по собственному
  разрешению `/128` с cap'ом (`ImagePanel.NATURAL_MAX_WIDTH`/`MAX_HEIGHT`). Это позднее
  уточнение эмитит `ImagePanel.size_changed`, и `WorldGenerator` локально перепаковывает объекты
  комнаты;
- видео — стартует из HTML `width`/`height` или 16:9 fallback, затем `VrwebVideoScreen`
  уточняет высоту/ширину по первому кадру и тоже эмитит `size_changed` для локального reflow;
- inline-картинки в тексте (RichPanel) — тоже через `128`, поэтому совпадают по размеру с
  такой же отдельной картинкой;
- ширина панели RichPanel **пружинит под контент**, а не фиксирована: текст — не шире
  `TEXT_MAX_M = 1.7 м` (короткий ужимается ещё, до фактической ширины, но не уже `MIN_W_M`),
  а картинка пошире расширяет панель под свою ширину (приоритет самой широкой картинки внутри).
  Меряется по факту раскладки через `get_content_width` (см. `RichPanel._refit`); футпринт для
  раскладки мира берёт ту же верхнюю границу (`estimate_width_m`). После `_refit` панель эмитит
  `size_changed`, поэтому фактическая высота/ширина попадает в перепаковку комнаты.

Заголовок `h1` по-прежнему заметно больше `h6`, картинка — настолько крупнее строки,
насколько крупнее в пикселях страницы. Один коэффициент `128` — это **единственная ручка
масштаба всего мира**: меняешь его — пропорционально пересчитывается всё. У Label3D растровый
кегль выводится отдельной `LABEL_RASTER_SCALE`, поэтому качество текста от выбора масштаба не
зависит.

> Три константы должны совпадать: `WorldGenerator.PX_PER_METER`,
> `ImagePanel.PX_PER_METER`, `RichPanel.PIXEL_PER_METER` (= 128).

**Картинки и панели двусторонни без зеркала.** К объекту можно подойти с любой стороны, и
с изнанки он показывает ту же **незеркальную** картинку, что и спереди. Единая двусторонняя
грань (`CULL_DISABLED`) для этого не годится: задняя грань рендерит UV отражённо — текст и
изображение читаются зеркально. Поэтому и `ImagePanel`, и `RichPanel` держат **два квада**:
лицевой и изнаночный (`_mesh_back`), развёрнутый на 180° вокруг Y, с общими мешем и
материалом. Материал односторонний (`CULL_BACK`), так что с каждой стороны виден ровно один
квад в нормальной ориентации. Меш/материал общие, так что логика загрузки текстур не меняется.
Клик луча у всех плоских UI/медиа-поверхностей мапится через общий `WorldUiSurface.world_to_ui_uv`;
со стороны изнанки содержимое горизонтально зеркально лицевому, поэтому при попадании сзади
(`local.z < 0` — ближняя грань бокса-коллайдера) координата `u` отражается (`1 - u`), чтобы
клик попал в тот же элемент, что игрок видит с изнанки.

## Сводка решений

- Топология и геометрия разведены; между ними — сериализуемый артефакт без координат.
- **Иерархия комнат — по оглавлению заголовков** (явных и визуальных), а не по вложенности
  DOM. Комнату создаёт только заголовок; внутри секции — плоский список объектов. DOM-
  контракция снята. Детали и эвристика: [content-sectioning.md](content-sectioning.md).
- ~~Комната = множество листьев, а не узел дерева.~~ *(старая модель)*
- ~~Контракция снизу вверх; остановка слияния на ≥2 комнатах рождает соединитель.~~
- Соединитель — секция-развилка (≥2 подкомнаты, нет своего наполнения); степень — хинт.
- Степень соединителя топологию не ломает — это хинт для геометрии.
- Портал — это функция (capability) поверх объекта, а не тип узла.
- Список → объект-коллекция внутри комнаты.
- Якоря — метки на пространствах; кросс-страничные ссылки — функция `navigate` + swap;
  глобального графа сайта нет.
- Вес комнаты = число листьев; без рендер-пасса.
- Размеры всех объектов (текст, картинки, панели) — по единой абсолютной метрике
  `1 м = 128 px` страницы; картинки больше не подгоняются под текст (§13).
- ~~**Открыто:** точка входа (`entry`)~~ — закрыто моделью сегментации: секция = свой вход (§8).
