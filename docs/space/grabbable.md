# Grabbable-предметы: нормативный контракт

> **Статус:** норматив стандарта VRWeb. Здесь — всё, что обязана воспроизвести ЛЮБАЯ
> реализация (движок/язык не важен): тег, машина hold-состояния, wire-конвенции, точки
> крепления и события. Как это устроено в Knossos — отдельно, в
> [docs/client/grabbable.md](../client/grabbable.md).

Grabbable — предмет сцены, который участник может взять в руку, носить с собой и положить.
Референс — pickup-система VRChat (`VRC_Pickup` + `VRC_ObjectSync`, см.
[vrchat-developer-api.md](vrchat-developer-api.md)), но с иным сетевым каноном:

- **гонки захвата** решает единая сериализация команд авторитетом (у VRChat — гонки ownership
  transfer);
- **держимый предмет не синхронизируется по transform**: по сети идёт только факт «держит P
  рукой H с хватом G», позу выводит каждый клиент сам (у VRChat — continuous sync с дрожанием
  и отставанием от руки);
- **физика всегда локальна и косметична**, канон — только поза покоя (у VRChat — сетевая
  синхронизация rigidbody с расхождением траекторий).

> **Subject Bindings:** назначенный держатель хранится в общем slot `holder`; `grab`/`release`
> атомарно меняют binding и hold-state. См.
> [subject-bindings.md](../network/subject-bindings.md).

## Слои-предпосылки

Контракт использует два уже специфицированных слоя и не добавляет собственных механизмов:

1. [Replicated State](../network/replicated-state.md) — COMMAND/DELTA/SNAPSHOT/ACK,
   авторитетная валидация, late join.
2. [Контракт аватара](../client/avatars.md) — шина параметров и (ниже) точки крепления.

## Тег `<VRWebGrabbable>`

Обёртка: её содержимое (произвольное VRWML-поддерево) становится предметом.

```xml
<VRWebGrabbable id="ball" theft="allow" grip="Transform3D(...)"
                transform="Transform3D(...)">
  <MeshInstance3D mesh="SubResource:::BallMesh"/>
</VRWebGrabbable>
```

| Атрибут | Значение | Семантика |
|---|---|---|
| `id` | строка | Стабильный id элемента. **Обязателен** для сетевого состояния: без него предмет не синхронизируется, клиент вправе сделать его неинтерактивным |
| `theft` | `allow` (default) \| `deny` | Можно ли перехватить предмет из чужой руки |
| `mode` | `fixed` (default) \| `adjustable` | Как предмет ложится в руку и можно ли его подстраивать (см. ниже) |
| `grip` | `Transform3D`-литерал, default identity | Авторский хват режима `fixed`: поза предмета относительно якоря руки. В `adjustable` игнорируется |
| прочие | — | Обычные свойства узла-обёртки (transform и т.п.), как у всех тегов VRWML |

### Режимы удержания

Прямая параллель с `Orientation` у VRChat `VRC_Pickup`:

| Режим | Хват в момент захвата | Подстройка держателем | Аналог VRChat |
|---|---|---|---|
| `fixed` | Снап в авторский `grip` — предмет всегда лежит в руке одинаково | запрещена (`adjust` отклоняется) | `Orientation: Grip/Gun` + `ExactGrip`/`ExactGun` |
| `adjustable` | **Фактическая поза в момент захвата** — предмет не «прыгает», сохраняется естественный офсет | разрешена держателю | `Orientation: None` + `Allow Manipulation When Equipped` |

`fixed` нужен, когда поза предмета в руке — часть замысла (инструмент, оружие, кружка).
`adjustable` — когда предмет двигают и разглядывают (реквизит, деталь, мебель).

Требования к реализации:

- Предмет должен быть **прицеливаемым**: клиент предоставляет способ навести взаимодействие
  на обёртку (область попадания по умолчанию — ограничивающий объём содержимого; авторская
  геометрия коллизии, если платформа позволяет её объявить, уточняет область). Конкретный
  механизм (луч, касание, VR-рука) — презентация клиента.
- **Поведение содержимого при удержании** описывается поведенчески: на время удержания
  собственное движение содержимого (физика, анимационные смещения корня) приостанавливается,
  позу ведёт крепление; при release управление возвращается содержимому. Каким механизмом
  движок это делает (freeze, kinematic-флаг) — деталь реализации.
- Тег может приходить и слоем [эфемерных изменений](../network/ephemeral-changes.md)
  (`kind="vrweb-node"`) — контракт не меняется.

## Сетевой адрес

`object_id` hold-состояния выводится **детерминированно**: `"grab:" + id` элемента в
пределах комнаты/страницы. Реализации в одной комнате обязаны выводить одинаковый адрес —
это условие интероперабельности гетерогенных клиентов.

## Hold-состояние: schema `vrweb.grabbable.hold` v1

Канон предмета — binding и четыре поля. Мировая поза держимого предмета **канонической не является**
(см. «Точки крепления»).

| Поле | Тип | Смысл |
|---|---|---|
| `bindings.holder` | string | Кто держит; отсутствие slot — предмет свободен |
| `hand` | string | Рука держателя; открытый enum, см. ниже |
| `grip` | float[7] | Поза предмета относительно якоря руки (зафиксирована в момент захвата) |
| `rest` | float[7] | Поза покоя свободного предмета **в пространстве родителя точки декларации** |
| `takeover_allowed` | bool | Разрешён ли перехват; фиксируется из атрибута `theft`, командой не меняется |
| `adjustable` | bool | Режим из атрибута `mode`; тоже фиксируется при регистрации. В состоянии он нужен, чтобы reducer валидировал `adjust` независимо от того, построен ли узел у авторитета |

Начальное значение `rest` каждый клиент регистрирует из декларированной позы узла — она
детерминирована страницей и одинакова у всех; канон начинает действовать с первого
delta/snapshot авторитета.

### Wire-конвенции поз

- Поза = `[px, py, pz, qx, qy, qz, qw]`: метры + кватернион в порядке **x, y, z, w**.
- Оси — конвенция значений VRWML (правая тройка, Y вверх, −Z «вперёд» — та же, что у
  `Transform3D`-литералов документа).
- Кватернион нормируется при коммите; вырожденный (|q| ≤ 0.1) — невалиден.
- **Масштаб в позу не входит**: предмет сохраняет собственный scale.
- `rest` задаётся в пространстве родителя точки декларации предмета — предмет, объявленный
  под движущимся узлом, покоится согласованно у всех клиентов.

### Команды и машина состояний

Reducer исполняет каждый клиент, оказавшийся авторитетом, поэтому таблица переходов
**нормативна** — решения обязаны совпадать у разных реализаций.

`grab(hand, grip)`:

| Условие | Исход |
|---|---|
| `user_id` отправителя пуст | **отказ** (аноним держать не может: `holder == ""` значит «свободен») |
| `hand` пуст или длиннее 32 байт | **отказ** |
| `grip` не валидная поза | **отказ** |
| предмет держит другой и `takeover_allowed == false` | **отказ** |
| предмет свободен / держит сам actor / takeover разрешён | **принять**: `bindings.holder := actor_user_id`, `hand`, `grip` одной revision |

`release(rest)`:

| Условие | Исход |
|---|---|
| предмет свободен | **отказ** |
| отправитель не держатель И не авторитет | **отказ** |
| `rest` не валидная поза | **отказ** |
| иначе | **принять**: снять `bindings.holder`, поставить `hand := ""`, `rest` одной revision |

`adjust(grip)` — держатель подстраивает хват (только `adjustable`):

| Условие | Исход |
|---|---|
| `adjustable == false` | **отказ** (у `fixed` хват принадлежит автору) |
| предмет свободен | **отказ** |
| отправитель не держатель | **отказ** — в том числе **авторитет**: чужим хватом не управляют |
| `grip` не валидная поза | **отказ** |
| иначе | **принять**: `grip` (держатель и рука не меняются) |

**`rest` при отпускании — это поза, в которой предмет находился в руке.** Реализация не
должна переносить предмет в точку прицела или к ногам: отпускание не телепортирует. Дальше
свободный предмет живёт по правилам своего содержимого (см. «Физика содержимого»).

Право авторитета на принудительный release — основа **авто-release**: если держатель перестал
быть участником инстанса, авторитет ОБЯЗАН освободить предмет (поза покоя — где рука
держателя была видна в последний раз). Это аналог TTL-обязанности авторитета в эфемерном слое.

### `hand` — открытый enum

Стандартные значения v1: `"left"`, `"right"`. Неизвестное непустое значение **не
отвергается**: состояние принимается, а рендер уходит в fallback-якорь клиента. Это дешёвая
страховка расширяемости (двуручный захват, `mouth`, кастомные точки) без смены версии схемы.

Клиенту СЛЕДУЕТ держать не более одного предмета в одной руке; протокол это не форсирует
(состояние пообъектное), нарушение безвредно.

## Точки крепления аватара

Мировая поза держимого предмета — **производная презентация**:

```
поза предмета = поза якоря руки держателя × grip
```

- Аватар МОЖЕТ объявлять именованные точки крепления; стандартные имена — `hand.left`,
  `hand.right`. Механизм объявления — дело реализации аватарного слоя (узел-маркер, bone
  reference…).
- Значение `hand` отображается в имя точки как `"hand." + hand`.
- Нет точки (или `hand` неизвестен) — клиент использует **синтетический fallback-якорь**
  (примерное положение руки тела/камеры).
- Поскольку поза производная, небольшие расхождения якорей между клиентами легальны — как
  интерполяция аватаров. Трафика во время удержания нет вообще.

**Подстройка и трафик.** В `adjustable` держатель меняет `grip`, и это единственный случай,
когда во время удержания идут сообщения. Реализация ОБЯЗАНА ограничивать их частоту: держатель
применяет свою подстройку локально-оптимистично, а в канон отправляет `adjust` разреженно и
обязательно фиксирует итог перед `release`. Поток пер-кадровых `adjust` — нарушение контракта:
остальные участники получают подстройку редкими дельтами, а не потоком.

Рядом с этим — два optional-параметра шины аватара (`HoldLeft`, `HoldRight`, Bool,
default `false`): рука держит предмет, аватар может сжать кисть.

## События для скриптов страницы

Через обычный механизм подписки на события материализованного объекта (`handle.on`, см.
[scripting-api.md](scripting-api.md)):

| Событие | Где эмитится | Payload |
|---|---|---|
| `grab` | на **всех** клиентах, из канонических дельт | `{user_id, hand}` |
| `drop` | на **всех** клиентах, из канонических дельт (включая перехват и авто-release) | `{user_id, hand}` — кто держал |
| `use` | **только на клиенте держателя**, transient (нажатие основного действия) | `{user_id, hand}` |
| `use_end` | **только на клиенте держателя**, transient (отпускание; парно к `use`) | `{user_id, hand}` |

Фазы `use`/`use_end` — аналог `OnPickupUseDown`/`OnPickupUseUp` VRChat; непрерывное действие
между ними (ведение штриха, прицельная струя) автор реализует опросом
[`document.player.aim()`](scripting-api.md#прицел-vrwebaim1) в `document.on_update`.

Кроме событий, предмет принимает типизированные операции держателя/автора — `release`,
`holder`, `held_hand`, `set_enabled`/`is_enabled` (нормативная семантика: `release` действует
только на клиенте держателя; `set_enabled(false)` блокирует новые захваты, не выбивая предмет
из руки). Биндинг — [scripting-api.md](scripting-api.md#grabbable-предметы-vrwebgrabbable1).

`use` — намерение, а не состояние (доктрина «событие ≠ состояние»): его результат автор
страницы при необходимости сам делает каноном (`document.state`) или адресно рассылает
(`document.remote`). Реализация НЕ должна реплицировать `use` автоматически.

## Физика содержимого

Grabbable-система с физикой не взаимодействует, кроме приостановки на время удержания:

- свободный предмет живёт по правилам своего содержимого (rigidbody падает локальной
  симуляцией, статичный узел стоит);
- канон свободного предмета — только `rest` момента release; локальные траектории после
  него — косметика и могут расходиться;
- расширение (не v1): последний держатель, увидев успокоение своего rigidbody-содержимого,
  МОЖЕТ прислать повторный `release` с осевшей позой — это чтение позы, не вмешательство в
  физику, и оно сходится у всех через обычную дельту.

## Совместимость и расширения

- Двуручный захват, инвентарь, бросок с импульсом — совместимые добавления (новые
  optional-поля/команды по правилам эволюции схем Replicated State), не v1.
- Ссылки VRChat-параллелей: `OnPickup`/`OnDrop`/`OnPickupUseDown` → `grab`/`drop`/`use`;
  `DisallowTheft` → `theft="deny"`; pickup ownership → сериализация авторитетом.
