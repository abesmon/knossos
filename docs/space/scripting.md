# VRWeb scripting v1

> Статус: реализованный MVP и целевой формат первой версии. Скриптинг не является частью
> Godot/VRWML object model и не использует прикреплённые GDScript, кодовые компоненты или пакеты.

VRWeb использует интерпретируемый Luau как переносимый язык поведения страницы. WASM не требуется
для этой версии и позже может появиться как отдельный, более тяжёлый runtime profile.

## Объявление в HTML

Скрипты выполняются в порядке документа. `id` обязателен и уникален в пределах страницы.

```html
<script type="application/vrweb+luau" id="room.lights">
local lamp = document.query("#lamp")
if lamp then
  lamp.on("activate", function(event)
    lamp.set("visible", not lamp.get("visible"))
  end, "Switch light")
end
</script>

<script type="application/vrweb+luau" id="room.chat"
        src="./chat.luau"
        integrity="sha256-BASE64"></script>
```

Ровно одна форма обязательна:

- inline: исходный текст внутри элемента;
- linked: `src`, разрешаемый относительно URL документа.

Опциональный атрибут `realm` задаёт **явную identity page realm** — namespace wire-адресов
`document.state`/`document.remote` и адрес страницы для remote calls. Все теги, указавшие
`realm`, обязаны указать одно и то же значение (у страницы ровно один realm); без атрибута
identity остаётся `id` первого валидного script tag. Явный `realm` устойчив к переименованию
или перестановке скриптов: сетевые адреса страницы не меняются от рефакторинга её тегов.

Рабочий linked-пример можно открыть по адресу `vrwebresource://examples/external_script.html`. В нём
[external_script.html](../../addons/vrweb_tools/examples/external_script.html) последовательно выполняет три
скрипта в одном page realm:

1. [external_model.luau](../../addons/vrweb_tools/examples/external_model.luau) объявляет глобальный класс
   `SpeedModel`;
2. [external_tiny.luau](../../addons/vrweb_tools/examples/external_tiny.luau) использует `SpeedModel` и объявляет
   класс `SpinnerView`;
3. inline-контроллер создаёт экземпляры обоих классов и связывает их с кнопкой.

Как у обычных browser classic scripts, глобальные объявления предыдущего тега доступны следующим
тегам, а `local` остаётся локальным для своего chunk. Поэтому linked-файл можно использовать как
обычную библиотеку или utility-класс без RPC и специального module loader. Ссылка на демо есть на
`vrwebresource://index.html` под названием «Luau: скрипт из соседнего файла».

`src` и непустое тело одновременно являются ошибкой. Клиент получает и интерпретирует исходник;
страница не поставляет bytecode. Профиль семантики runtime: `vrweb-luau/1` — это отдельный от
capability names namespace: профиль описывает язык и семантику VM целиком, а capability
(`vrweb/<домен>/<версия>`) — обнаруживаемые возможности `document` внутри профиля.

Knossos загружает linked source с таймаутом 15 секунд и не более чем через 8 redirects. HTTP(S)
документ не может ссылкой прочитать `vrweblocal://` или `vrwebresource://`: локальная файловая
схема доступна только локальному документу, а bundle-схема — документу из того же bundle.
Политика downgrade и проверка каждого redirect hop остаются отдельным transport-hardening слоем.

## Integrity

Поведение совпадает с opt-in моделью HTML Subresource Integrity:

- нет `integrity` — проверка не запрошена, скрипт разрешено загрузить;
- `integrity` есть — клиент обязан проверить поддерживаемый digest до исполнения;
- digest не совпал или алгоритм не поддержан — этот скрипт не исполняется.

MVP поддерживает `sha256-BASE64`. Integrity подтверждает соответствие полученных байтов ожиданию
автора документа, но не выдаёт скрипту полномочий и не доказывает безопасность кода.

## Realm и жизненный цикл

Страница получает один sandboxed Luau realm, общий для всех её inline и linked scripts. Теги
выполняются в порядке документа и видят одну writable global table и один `document.session`.
В VM нет
Godot objects, файловой системы, ОС, сетевых классов/сокетов, загрузки нативных библиотек и
произвольного доступа к объектам клиента. Разрешённые HTTP(S)-данные доступны только через
ограниченную capability `document.assets`: она применяет URL policy, лимиты и выдаёт ресурсы как
opaque handles. Доступны безопасные стандартные библиотеки Luau и единственный host object
`document`.

Изоляция отдельных тегов не является поведением по умолчанию и не входит в v1. Если позже
появится реальный сценарий для недоверенного виджета или плагина внутри страницы, отдельный realm
может быть добавлен только как явный opt-in автора, не меняющий browser-like семантику обычных
`<script>`.

Page lifecycle v1 имеет явные границы:

1. `fetch` — документ и linked sources загружаются, код не исполняется;
2. `materialize` — декларативная сцена строится и добавляется в scene tree;
3. `scene-ready` — прошёл первый physics frame новой сцены; только теперь скрипты запускаются
   в порядке документа и могут регистрировать handlers;
4. `active` — успешные staged side effects опубликованы, callbacks могут выполняться;
5. `dispose` — навигация закрывает handlers, timers, handles и realm.

Все скрипты профиля `vrweb-luau/1` запускаются на `scene-ready`. Атрибут выбора фазы намеренно
не вводится в v1: почти любому поведению нужна полностью материализованная сцена, а ранний запуск
создаёт две семантики `document.query()` и гонки с физикой. Готовность сетевой комнаты не блокирует
`scene-ready`: офлайн-клиент должен работать сразу, а `document.state` самостоятельно переживает
room reset и authority transition.

Навигация уничтожает page realm. Ошибка загрузки или top-level исполнения одного тега отмечается
на его `id`, после чего следующие теги продолжают выполняться в том же порядке, как classic
scripts в браузере. Callback вызывается через контролируемый вход VM с interrupt deadline;
фатальная ошибка callback или превышение CPU budget закрывает page realm вместе с его handlers,
timers и owned objects, не обрушая клиент и декларативную сцену.

### Фазовый контракт host API

Поведение каждого вызова до `active` описывается ровно одной из трёх колонок; вызов вне своей
фазы отказывает стандартным кодом `lifecycle`
([контракт host-вызовов](scripting-api.md#контракт-host-вызовов)):

| Поверхность | Staged top-level (`scene-ready` → commit) | После commit (`active`) |
|---|---|---|
| `query`/`get`/`values`/`clock`-чтения, `players.*`, `player.*`, `scene.object(s)`, `state.read/bindings/revision`, `assets.resolve`, `remote.expose`, `features`, `log` | доступно сразу | доступно |
| `create`, `set`, `on`, таймеры, `on_update`, `state.define/ensure/command/on`, `scene.add/update/remove`, `remote.call`, `assets.fetch/load`, `resource.apply` | **стейджится**: side effect публикуется атомарно на commit | исполняется сразу |
| `handle.call` (транспорт video/grabbable и методы узлов), `files.pick`, `assets.decode` | `lifecycle` | доступно |
| `physics.bind` | доступно (только здесь) | `lifecycle` |

Правило границы: чтения и декларации доступны всегда, отложимые side effects стейджатся,
неотложимые операции (транспорт, системный диалог, немедленный декод) ждут `active`, а
структурная регистрация физики принадлежит только staged top-level.

## Realtime replacement

Клиент может заменить source по `id`, передав hash активной базовой ревизии. Поскольку последующие
теги могли сохранить ссылки на объявленные им globals, клиент пересобирает весь page realm в
порядке документа. Только успешная полная ревизия атомарно заменяет handlers/timers/owned objects;
при ошибке старая продолжает работать. Общий сериализуемый `document.session` переносится в новую
VM; состояние сцены и distributed state живут вне VM.

Это позволяет редактировать поведение во время realtime-сессии без перезагрузки пространства.
Транспорт и право публикации ревизии не входят в scripting v1.

## Слои безопасности

Модель намеренно разделена на три пересекающихся слоя:

1. стандартный безопасный capability API VRWeb;
2. фактически реализованный и ограниченный клиентом набор;
3. правила поведения и доступа конкретного instance, написанные автором мира.

Capability в этом контексте означает обнаруживаемую возможность стандарта, а не браузерный
permission prompt. Core API, включая безопасное управление локальной locomotion, доступен
странице без отдельного per-site согласия, если клиент его реализует. Как в HTML/Web, безопасность
должна следовать из ограничений самого API и корректности браузера, а не из перекладывания каждой
операции на пользователя. Настройки клиента могут отключать возможности, но это продуктовая
политика и feature compatibility, а не основная security boundary.

Это та же модель, что применяется к поддержке VRWML-тегов: документ использует стандартную
возможность, а реализация объявляет реально доступное безопасное подмножество. Instance rules
решают игровые вопросы «кто кого может телепортировать», но не расширяют host API.

CPU deadlines, memory ceilings, лимиты handles/timers/calls и допустимые engine classes — локальная
policy реализации, а не переносимое значение стандарта. Клиент вправе ограничивать циклы,
отключать код или классы и восстанавливаться после зависания; это не означает, что стандарт должен
оценивать эффективность authored content. Knossos применяет собственные консервативные лимиты;
жёсткий allocator ceiling может быть добавлен в GDExtension отдельно, не меняя HTML или
`document` API.

Общий для всего стека принцип авторской свободы, отделяющий security containment от оценки
эффективности authored content, задан в [архитектурном обзоре](../README.md#авторская-свобода-и-граница-безопасности).

## Совместимость реализаций

Стандарт фиксирует Luau-синтаксис, профиль библиотек, форму объявления, `document` API и наблюдаемую
семантику. Godot-типы не передаются напрямую. Knossos отображает opaque handles на Godot nodes;
другой клиент отображает те же операции на собственную сцену. Новые возможности добавляются как
versioned capability names, не расширением неограниченного engine bridge.

API описан в [scripting-api.md](scripting-api.md).
