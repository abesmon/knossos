# Мини-каскад CSS: вычисленные стили элементов

Клиент получает **вычисленные CSS-свойства** каждого элемента страницы — цвета, кегли,
видимость — включая внешние таблицы (`<link rel="stylesheet">`), каскад со специфичностью,
наследование и относительные единицы. Реализовано на чистом GDScript, без движка рендера.

Что это даёт пайплайну: палитра комнат и паспорт документа работают на реальных сайтах
(цвета из классов, а не только инлайн `style=""`); `typography.base_px` и эвристика
«визуальных заголовков» видят настоящие кегли; скрытие через классы (`.sr-only`,
мобильные меню) перестаёт попадать в мир.

**Не-цель:** layout (координаты и размеры боксов). Для него мини-каскада принципиально
недостаточно — это задача будущего GDExtension с настоящим веб-движком, который заполнит
тот же контракт `computed` точнее и добавит боксы. Мини-каскад — осознанный промежуточный
шаг: автономный, офлайновый, «клиент решает всё».

Связано: [world-visualization.md](world-visualization.md) (потребитель цветов),
[html-to-3d-topology.md](html-to-3d-topology.md) §3/§11 (скрытие, фазы),
[content-sectioning.md](content-sectioning.md) (визуальные заголовки).

## 1. Архитектура

```
HTML ──HtmlParser──> дерево HtmlNode
  │
  ├─ StyleResolver.collect_sheet_refs: <style> и <link rel=stylesheet> в порядке документа
  │
  ├─ CssFetcher.fetch_all (main.gd): качает внешние таблицы параллельно,
  │    разворачивает @import, кэширует по URL; дедлайн CSS_DEADLINE_SEC (4 с) —
  │    дальше едем с тем, что пришло
  │
  ├─ StyleResolver.resolve(doc, css_texts): парсинг правил (CssParser),
  │    каскад + наследование за один DFS ──> node.computed у каждого элемента
  │
  └─ TopologyBuilder / HeadingDetector читают node.computed
       (фолбэк на старые инлайн-проверки, если резолвер не бежал)
```

Разделение обязанностей:

| Файл | Роль |
|---|---|
| `scripts/css_parser.gd` | `CssParser` — таблица → плоский список правил; вайтлист свойств, @media/@supports/@import, цвета, токенизация с учётом вложенности |
| `scripts/style_resolver.gd` | `StyleResolver` — индекс правил, матчинг селекторов, каскад, наследование, вычисление значений → `node.computed` |
| `scripts/css_fetcher.gd` | `CssFetcher` — пул HTTPRequest + кэш (постоянный узел в main, переживает навигацию), разворачивание @import |
| `addons/vrweb_tools/html_node.gd` | поле `computed: Dictionary` (пусто = каскад не бежал) |

`TopologyBuilder` и `StyleResolver` остаются чистыми (без I/O): вся сеть — в `CssFetcher`
до фазы топологии. Контракт артефакта `Space` **не менялся**: `hints.css {bg, fg, border}`,
`document {bg, bg_image, fg}`, `typography.base_px` — те же строки/float, только данные
стали полными (цвета сериализуются как `#rrggbbaa`). `WorldGenerator` не тронут.

## 2. Поддержанное подмножество

**Свойства (вайтлист; остальное отбрасывается при парсинге):**

| Группа | Свойства |
|---|---|
| Цвета | `color`, `background-color`, `background-image`, `background` (шортхенд → цвет + url), `border*` (→ булев хинт «есть рамка»; `none`/`0` — нет) |
| Шрифты | `font-size`, `font-weight`, `font` (шортхенд → кегль), `text-align` |
| Видимость | `display`, `visibility`, `opacity`, `width`, `height`, `position`, `left`, `top`, `text-indent`, `clip`, `clip-path` |
| Переменные | `--*` (каскадируются и наследуются; `var(--x, fallback)` подставляется при вычислении, глубина ≤ 4) |

**Селекторы:** `tag`, `*`, `.class`, `#id`, `[attr]` (по наличию), `:root` (→ `html`);
комбинаторы потомок и `>`. Неподдержанное (`:hover`, `::before`, `+`, `~`, `[attr=v]`,
`:is()`, …) выбрасывает **этот селектор**, остальной comma-список правила живёт.
Специфичность (a, b, c) + порядок источника; тиры каскада: презентационные атрибуты
(`bgcolor`, `text`, `<font size>`, `align`) < author < инлайн `style` < author `!important`
< инлайн `!important`.

**@-правила:** `@media` — против предполагаемого viewport **1280×800 screen**
(`min/max-width|height`, не-`print/speech`, `prefers-color-scheme: light`; неизвестные фичи
считаются истинными); `@supports`/`@layer` — спускаемся внутрь (порядок слоёв не
моделируем); `@import` — разворачивается CssFetcher'ом (глубина ≤ 2, циклы отсекаются);
`@font-face`/`@keyframes`/`@page` — пропуск блоком.

**Единицы font-size:** `px`, `pt`, голое число — абсолют; `em`/`%` — от родителя;
`rem` — от `<html>` (страница-фрагмент без `<html>` получает виртуальный корень, чтобы
`:root`-переменные и rem-база не терялись); ключевые слова (`large`, …) — таблица.
Наследуются: `color`, `font-size`, `font-weight`, `visibility`, `text-align`, `--*`.

**Цвета:** `#hex` (3/4/6/8), `rgb()/rgba()` (включая современный синтаксис с `/`),
`hsl()/hsla()` (конверсия в RGB), имена X11, `currentColor` (→ вычисленный `color` узла);
`transparent`/альфа ≈ 0 → цвета нет.

## 3. Контракт `node.computed`

Заполняется на каждом элементе (пустой словарь = резолвер не бежал — фолбэк-пути
в потребителях читают инлайн `style` как раньше):

```gdscript
{
  "color": Color,             # унаследованная цепочка; присутствует, если где-то объявлен
  "color-own": bool,          # у узла СВОЯ победившая декларация (не наследство) —
                              # только такие дают fg-хинт комнаты
  "background-color": Color,  # только собственный; отсутствует при transparent
  "background-image": String, # сырой url-токен (не резолвлен в абсолютный)
  "border": bool,
  "font-size": float,         # абсолютные px; есть всегда (наследование учтено)
  "font-weight": int,         # 100..900; есть всегда
  "text-align": String,       # наследуется
  "display": String, "visibility": String, "opacity": float,   # когда объявлены
  "hidden": bool,             # готовый вердикт «невидим для глаза» (та же эвристика,
                              # что _is_hidden, но по каскадному набору)
}
```

Будущий движок-GDExtension заполнит этот же словарь точнее и добавит layout-боксы —
потребители не изменятся.

**Точки потребления:** `TopologyBuilder._css_hints` / `_is_hidden` / `_accumulate_font_px`
/ `_document_style`; `HeadingDetector._own_font_px` / `_self_bold` / `_is_centered`.
Везде паттерн «computed сначала, инлайн-фолбэк».

## 4. Производительность и капы

Главный рычаг — **whitelist-and-drop**: декларации фильтруются вайтлистом при парсинге,
правила с опустевшими декларациями выбрасываются. Реальные замеры (Godot 4.6, M-серия):
42 КБ (Wikipedia vector) → 197 правил, ~18 мс; 227 КБ (Bootstrap min) → 991 правило,
~87 мс. Разово на сайт (кэш CssFetcher), в бюджет укладывается; если встретится сильно
хуже — `CssParser.parse` чистый и готов к выносу в `WorkerThreadPool`.

Матчинг — через индекс по правому compound-селектору (`#id`/`.class`/`tag`/universal):
кандидаты на узел — объединение его бакетов, а не вся таблица.

Капы: `MAX_SHEET_BYTES` 1.5 МБ на таблицу (обрезка), `MAX_RULES` 20 000, тело HTTP
≤ 1.5 МБ, таймаут запроса 4 с, дедлайн всей загрузки 4 с, кэш ≤ 8 МБ (грубый сброс),
`@import` глубина ≤ 2.

Гонки: `CssFetcher._generation` отсекает поздние завершения старого `fetch_all`
(тексты всё равно оседают в кэше); `main._nav_id` отсекает продолжение `_finish_page`,
если за время загрузки стилей началась новая навигация; `_loading` держится до
`_finish_page` — повторный go! в это окно заблокирован.

## 5. Известные приближения (осознанные)

- **Viewport фиксированный** (1280×800): мобильные `@media`-ветки не применяются — и не
  должны: мир строим по десктопной вёрстке.
- **Псевдоклассы/элементы** не матчатся: `:hover`-стили и `::before`-контент невидимы
  для топологии (это состояния взаимодействия, не статика страницы).
- **`@layer` без порядка слоёв**: правила слоёв конкурируют по общей специфичности; на
  практике для наших свойств расхождения редки.
- **`background-image: linear-gradient(...)`** не берётся (только `url(...)`) — градиентное
  небо в бэклоге [world-visualization.md](world-visualization.md) §5.
- **Комментарии внутри строк** (`content: "/*"`) могут сбить вырезалку комментариев —
  экзотика, принято.
- **`width`/`height` в %-единицах** не участвуют в вердикте `hidden` (в метры не
  перевести) — как и раньше в инлайн-проверках.
