# Переносимые инструменты: анализ и точки роста

> **Статус:** направление принято; **база реализована** и **инструменты клиента перенесены
> на item-модель** (июль 2026): `document.scene` (`vrweb/scene-objects/1`), прицел
> (`vrweb/aim/1`) и фазы `use`/`use_end`, `document.files.pick` (`vrweb/files/1`),
> поверхность grabbable (`vrweb/grabbable/1`), item-runtime (`kind="vrweb-item"`), а
> карандаш/ластик/размещение картинок — item-документы (vrweb/builtin/items/*), клиентские
> `DrawingTool`/`ImagePlacementTool` удалены (тонкий ItemToolbelt —
> [docs/client/tools.md](../client/tools.md)). Смоуки — tests/test_portable_base.tscn и
> tests/test_item_tools.tscn. Следующий этап — руководство для авторов инструментов. Цель — уйти от инструментов,
> вшитых в клиент ([docs/client/tools.md](../client/tools.md)), к **переносимым предметам**:
> блок «разметка + скрипт + ассеты», который можно добавить в клиент и убрать из него без
> обновления самого клиента. База — grabbable ([grabbable.md](grabbable.md)) как физический
> носитель и scripting ([scripting.md](scripting.md)) как поведение.

## Принцип: модель браузера, а не модель SDK

Стандарт VRWeb **не регулирует качество и намерения кода**. Если API выдан в рамках
стандарта — всё, что через него выразимо, легально по построению: мы не добавляем правил
«насколько код эффективный» или «не зловредный ли он». Граница ровно одна — **песочница**:
скрипт не может сделать то, на что пользователь не соглашался (движок, ОС, файлы, произвольная
сеть); а явное действие пользователя (выбрал файл, взял предмет, вошёл в инстанс) — это и есть
согласие. Чистота инстанса, спам предметами, «читерские» инструменты — политика пользователей
и владельцев инстанса (ранги, instance ACL, client policy), не стандарта. Это та же философия,
что уже зафиксирована в [security.md](../security.md): транспорт и API дают возможности и
факты, политику выбирает клиент/пользователь. Как веб-браузер: он исполняет любой JS в рамках
Web API и не спрашивает, «хороший» ли это сайт.

Следствие для дизайна ниже: **никаких дополнительных permission-слоёв для скриптов предметов
не проектируем.** Скрипт предмета получает тот же стандартный capability pool, что скрипт
страницы, в собственном realm со стандартными лимитами. Вопрос «доверять ли предмету» не
существует на уровне стандарта — существуют только границы песочницы и политика инстанса.

## Что уже достаточно сегодня

Два столпа на месте, и класс «интерактивные предметы с поведением» реализуем без правок клиента:

| Потребность инструмента | Чем закрыто |
|---|---|
| Взять/носить/видно другим/руки L/R/VR-задел | grabbable: hold-состояние, attachment-модель, `hand`/`grip`, `fixed`/`adjustable` |
| Основное действие в руке | событие `use` (transient у держателя) |
| Канон, late join, права, ACK | `document.state` (Replicated State) |
| Адресные эффекты на выбранном клиенте | `document.remote` |
| Ресурсы: текстуры, звук, меши, шейдеры | `document.assets`, `document.shaders` |
| Общая шкала времени | `document.clock.authority_time`, `AUTHORITY_TIME` |

Реализуемо уже сейчас: пульты, ключи, реквизит, «волшебные предметы» (use → state/remote),
головоломочные предметы, adjustable-декор.

## Лакмус: текущие три инструмента

- **Карандаш** — не переносим: нужен прицел (точка луча каждый кадр), фазы нажатия
  (use press/release) и артефакт в эфемерном слое (скриптам недоступен).
- **Размещение картинки** — не переносим: прицел с нормалью, файловый пикер, эфемерный слой.
- **Пузырь** — останется клиентским: он живёт на границе навигации, это системная механика.

Разрывы общие, не точечные — они и есть точки роста.

## Точки роста (ранжированы)

Каждая открывает **класс** инструментов, а не конкретный инструмент. Пер-инструментные API
(`document.pencil`...) — анти-цель.

### 1. Item-пакет: доставка поведения без обновления клиента (ключевая)

Сейчас скрипты — только `<script>`-теги страницы, собранные на scene-ready; эфемерный
`vrweb-node` приносит разметку, но не поведение. Пока это так, инструмент неотделим от
страницы, где объявлен. Нужен формат **item**: VRWML-фрагмент + Luau + ассеты,
контент-адресуемый (blob-слой уже даёт hash-адресацию и p2p-докачку), с жизненным циклом
«смонтировать realm / размонтировать». Модель доверия по принципу выше проста: **стандартный
capability pool в собственном realm со стандартными лимитами**, никакой отдельной модели
разрешений. Открывает все классы разом: инструмент становится вещью, которую приносят в мир,
поднимают и уносят (взрослая версия «инструмент как pickable» из tools.md).

### 2. Эфемерный слой как capability

Артефакт-оставляющие инструменты (рисование, метки, размещение, временные порталы) — это
`add/update/remove` в [эфемерном слое](../network/ephemeral-changes.md), где уже есть права,
TTL, каскад, персистенция и консоль. Осталось выдать его скриптам (условно `document.scene`).
Авторитет и так валидирует действия, content-policy и так фильтрует материализацию — протокол
не меняется вообще. Самая дешёвая из крупных точек.

### 3. Прицел и фазы ввода как capability

Общая ось всего «наводимого» (рисование, измерение, покраска, указки, стройка): точка луча +
нормаль + handle цели (клиент уже считает это каждый кадр для курсора) и `use` с фазами
press/release (VRChat: `OnPickupUseDown/Up`; у нас пока только press). В VR та же ось станет
лучами рук — контракт один.

### 4. Пользовательский выбор файла

Web-модель `<input type="file">`: пользователь сам выбирает файл — это и есть согласие; скрипт
получает байты/блоб. Открывает класс инструментов импорта и стыкуется с
[realtime-ресурсами](../network/realtime-resources.md) (файл → `vrwebblob://`).

### 5. Скриптовая поверхность grabbable

Программный `drop`, включить/выключить взятие, прочитать держателя — `handle.call`/`get`
поверх готовых команд hold-схемы (у VRChat — `Drop()`, `pickupable`). Маленькая общая ручка.

### 6. Хранилище предметов: локальное — база, home server — опция

Инвентарь и перенос инструмента между страницами = сериализованный item-дескриптор (п. 1) +
хранилище. Принципиально: **базовое хранилище — на клиенте** (локальные данные пользователя,
`user://` через Sandbox); **home server — необязательная возможность** синхронизации между
клиентами в рамках аккаунта, для тех, кто его подключил. Клиент без home server — полноценный
сценарий (приватность), инвентарь обязан работать локально. Это отличает нас от VRChat
PlayerData, где persistence существует только на серверах платформы.

## Чего не делать

- Не расширять `PlayerTool`/`ToolManager` — уходящая натура; клиентскими остаются только
  системные механики (пузырь, OS-диалоги, статус-строка).
- Не делать пер-инструментные API — только общие capabilities.
- Не тащить артефакты в Replicated State — им место в эфемерном слое (граница из
  [replicated-state.md](../network/replicated-state.md) верна).
- Не строить permission-модель для item-скриптов — см. принцип браузера выше.

## Порядок

Каждый шаг открывает класс и проверяется реальным инструментом:

1. ~~**П. 2** (эфемерный слой скриптам)~~ — готово: `document.scene`, офлайн standalone.
2. ~~**П. 3** (прицел + фазы use)~~ — готово: `document.player.aim()`, `use`/`use_end`.
3. ~~**П. 4** (выбор файла)~~ — готово: `document.files.pick` → BlobStore → `vrwebblob://`.
4. ~~**П. 5** (поверхность grabbable)~~ — готово: `release`/`holder`/`held_hand`/`set_enabled`.
5. ~~**П. 1** (item-runtime)~~ — готово в минимальной форме: `kind="vrweb-item"` с `props.src`;
   item-документ = обычный HTML/VRWML со скриптами, никакого отдельного формата пакета —
   доставка контент-адресуемых бандлов (blob-паки) остаётся расширением.
6. ~~Перенос карандаша/ластика/размещения картинок на item-модель~~ — готово: item-документы
   в vrweb/builtin/items/, тонкий `ItemToolbelt` вместо `DrawingTool`/`ImagePlacementTool`
   (живое требование ластика дало общее расширение `document.scene.objects(kind)`, а
   рисование в воздухе — `origin`/`direction` в `aim()`).
7. ~~Руководство для авторов инструментов~~ — готово:
   [tool-authoring.md](tool-authoring.md).
8. **Дальше:** **п. 6** (локальный инвентарь) — по мере надобности; объявленные расширения
   ниже — по мере появления сценариев.

## Объявленные расширения (не сейчас)

- **Процедурные артефакты.** Штрих стандартизирован как специальный VRWML-тег
  `<VRWebStroke points="...">` и переносится универсальным `vrweb-node`; отдельной ветки
  материализации по `kind` больше нет. Если понадобятся произвольные «толстые» артефакты,
  остаётся отдельная точка расширения — общий capability процедурных мешей.
- **Контент-адресуемые бандлы предметов** (item + ассеты одним blob-паком) вместо документа
  по URL: нужно, когда предметы начнут ходить между мирами и храниться в инвентаре.

## Как это устроено (реализация, кратко)

- Item = эфемерный объект `kind="vrweb-item"` (`props.src`), см.
  [ephemeral-changes.md](../network/ephemeral-changes.md#переносимые-предметы-kindvrweb-item).
  Каждый клиент скачивает документ и исполняет его сам: тот же parser → content policy →
  builder → `VrwebLuauRuntime`, что у страниц. Никакого нового формата: item-документ — это
  маленькая страница.
- Realm предмета получает **стандартный** capability pool со стандартными лимитами
  (принцип браузера выше). Namespace скриптовых и grabbable-адресов — id объекта слоя:
  экземпляры не сталкиваются.
- Ресурсный потолок клиента: 32 одновременных item-realm (лимит песочницы, не политика).
- API базы специфицированы в [scripting-api.md](scripting-api.md): `document.scene`,
  `document.player.aim`, `document.files.pick`, методы grabbable.
