# Производительность — стриминг геометрии и ресурсов

> **Проблема:** при переходе на «тяжёлую» страницу (сотни элементов) клиент фризил —
> весь мир строился за один кадр. Скачивание ресурсов уже было асинхронным; вешала
> **синхронная генерация узлов**. Решение этой итерации — тайм-слайс билда (очередь по
> кадрам) + шеринг ресурсов + вынос локального декода за кадр. Радиус-стриминг (грузить
> только в радиусе от игрока) пока **не** делали — см. «Дальше».

## Где была блокировка

`WorldGenerator._build` за один вызов делал всё: раскладку (фазы 1–4) и создание **всех**
узлов (фаза 5) — полы/стены/объекты/коридоры/атмосферу. На странице с сотнями элементов это
тысячи `Node3D`/`MeshInstance3D`/`StaticBody3D` за кадр → фриз. Скачивание картинок
([image_loader.gd](../scripts/image_loader.gd)) и ext-ресурсов
([vrweb_resource_loader.gd](../scripts/vrweb_resource_loader.gd)) уже шло через пул
`HTTPRequest` с очередью — её не трогали.

## Что сделано

### 1. Тайм-слайс билда (очередь по кадрам)

`WorldGenerator._build` разбит на **синхронную** и **стримящуюся** части:

- **Синхронно** (до возврата из `generate()`): раскладка, позиции якорей (`_resolve_labels`),
  атмосфера (свет/небо — чтобы мир был освещён сразу) и **комната спавна** (чтобы `spawn_point`
  был готов и игроку было куда встать). Комната спавна = `_object_room[_first_obj_id]`.
- **Порциями по кадрам** (`_stream_remaining`): все остальные комнаты и полы-коридоров. Строим
  узлы, пока не выйдет бюджет кадра `BUILD_BUDGET_MS` (6 мс), затем `await tree.process_frame`
  и продолжаем со следующего кадра.

По завершении стриминга — `build_complete = true` и сигнал `build_finished`.

Маленькие страницы укладываются в один бюджет и достраиваются синхронно (тогда
`build_complete` уже `true` к возврату из `generate()`).

#### Тонкости реализации (грабли)

- **Стриминг запускается отдельной корутиной из `generate()`** (`g._stream_remaining()`,
  fire-and-forget), а `_build` остаётся синхронным. Вложенный `await` корутины внутри другой
  fire-and-forget-корутины (`_build` делал `await _stream_remaining()`) **не возобновляется** —
  одноуровневый fire-and-forget работает, вложенный ломается. Это проверено пробами.
- **Ждём кадр через `Engine.get_main_loop().process_frame`, а не `node.get_tree()`.**
  `node.get_tree()` равен `null`, если узел ещё не «в дереве» (например, геометрия строится в
  `_initialize` headless-теста) → краш корутины. `Engine.get_main_loop()` отдаёт `SceneTree`
  всегда.
- **Контейнер `Generated`.** Вся геометрия живёт под отдельным `Node3D` (`_container`) под
  `_world`. `$world` переживает навигацию (сносятся только его дети), поэтому без контейнера
  старая корутина досыпала бы узлы старой страницы уже в новый мир. После каждого `await`
  корутина проверяет `is_instance_valid(_container)`; при навигации main делает `queue_free`
  контейнера → на следующем кадре он невалиден → достройка молча бросается.
- **main держит сильную ссылку на генератор (`_world_gen`).** `WorldGenerator` — `RefCounted`;
  после возврата из `_rebuild_world` локальная `gen` выходит из области видимости, и
  подвешенная на `await process_frame` корутина-достройка **собирается GC и не возобновляется**
  (симптом: мир строится только до первого `await` — пропадают коридоры, видео не
  привязываются). Поле `_world_gen` держит генератор живым до конца стриминга; следующая
  навигация перезаписывает поле — старый генератор освобождается, его корутина видит снесённый
  контейнер и сама прекращается. (В headless-тесте та же роль у `_gen` — иначе тест бы это не
  поймал.)
- **`video_manager.scan` в [main.gd](../scenes/main.gd) отложен до `build_finished`.** Экраны
  HTML-`<video>` строит `WorldGenerator`, и при стриминге они появляются не сразу — иначе скан
  не привязал бы их. Если `build_complete` уже `true` (маленькая страница) — сканируем сразу.
  В `exclusive`-режиме (генератора нет) — сразу.

### 2. Шеринг ресурсов в `_add_box`

Полы/стены — тысячи боксов, но размеров и цветов среди них мало. `_add_box` берёт
`BoxMesh`/`BoxShape3D`/`StandardMaterial3D` из кэшей (`_box_mesh_cache` по `Vector3`,
`_mat_cache` по `Color`) вместо уникального ресурса на каждый бокс. Ресурсы (`RefCounted`)
безопасно делить между инстансами — после создания мы их не мутируем. Экономит память и время.

### 3. Локальный декод картинок — за кадр

`ImageLoader._start` для локальных схем (`vrweblocal`/`vrwebresource`) раньше читал+декодил
синхронно цепочкой `_deliver→_pump→_start` — на странице с кучей локальных картинок это фриз.
Теперь `_start_local` ждёт кадр перед декодом; слот `_active` занят на время ожидания, поэтому
за кадр декодится не больше `MAX_CONCURRENT` картинок (та же очередь, что у сетевых).

## Где в коде

| Место | Что делает |
|---|---|
| `WorldGenerator._build` | синхронная часть: раскладка + атмосфера + комната спавна + `_compute_spawn` |
| `WorldGenerator._stream_remaining` | достройка остальных комнат порциями по кадрам, `build_finished` |
| `WorldGenerator._add_box` / `_shared_*` | шеринг `BoxMesh`/`BoxShape3D`/материала через кэши |
| `WorldGenerator.build_complete` / `build_finished` | флаг + сигнал «мир достроен» (для main) |
| `main._rebuild_world` / `_world_gen` | держит генератор живым (GC) + откладывает `video_manager.scan` до `build_finished` |
| `ImageLoader._start_local` | локальный декод картинки за кадр |

## Дальше

- **Радиус-стриминг** (LOD/стриминг по расстоянию): строить/сносить комнаты в радиусе ~50 м от
  игрока. Ограничит память на огромных страницах независимо от их размера; позиции комнат
  известны после раскладки. Затронет спавн, видео, мультиплеер, дебаг — отложено.
- Бюджет `BUILD_BUDGET_MS` подобран на глаз (6 мс) — можно адаптировать под FPS.
- Тяжёлая **раскладка** (фазы 1–4, `_apply_collision` это O(n²) по комнатам) пока синхронна;
  если станет узким местом на страницах с сотнями секций — её тоже резать на части.
