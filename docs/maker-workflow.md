# Путь разработчика мира: от пустой сцены до публикации

Это стартовая инструкция для автора VRWeb-мира. Она описывает один рабочий цикл целиком;
справочники по формату и API находятся по ссылкам в конце. Самый короткий путь — начать с
готового проекта Maker Kit, а не подключать addon к пустому проекту вручную.

## 1. Подготовить проект

1. Распакуйте Maker Kit и откройте его корень в Godot 4.6. Корень уже является проектом.
2. Убедитесь, что **VRWeb Tools** включён в Project → Project Settings → Plugins.
3. Сохраните проект в системе контроля версий. Свои сцены, assets и настройки храните отдельно
   от выпуска Maker Kit: при обновлении заменяется только `addons/vrweb_tools`.
4. Откройте `world.tscn` и сразу выполните Scene → Export As… → **VRWML Scene…** → HTML,
   профиль `strict`. Так проверяется установка до начала работы.

Если addon переносится в существующий проект, скопируйте весь каталог `addons/vrweb_tools` с
сохранением пути и включите plugin. Starter `world.tscn` полезен как проверенный минимальный
пример, но не должен перезаписывать существующий проект.

## 2. Создать сцену

Обычная Godot `.tscn` — редактируемый исходник. Корневой `Node3D` является authoring holder:
при HTML export в `<vrwml>` попадают его дети. Для нового мира нужны как минимум:

- визуальная геометрия (`MeshInstance3D`) и свет;
- физическая поверхность (`StaticBody3D` с `CollisionShape3D`), иначе игрок провалится;
- `VRWebSpawner` со вложенным `Marker3D`/`SpawnerPoint`, задающий начальную позицию;
- осмысленные имена узлов; узлам, которые использует скрипт, нужны стабильные queryable id.

Проще дублировать и менять узлы starter-а. В strict-профиле поддерживается проверенный набор
Godot nodes/resources, перечисленный в [Maker Kit](maker-kit.md#локальные-профили-maker-kit).
Полный язык сцены описан в [каталоге VRWML-тегов](space/vrwml-tags.md). `F6` удобен для грубой
проверки геометрии в Godot, но это не проверка VRWeb runtime: spawn, sandboxed-скрипты,
external resources и policy нужно проверять через **Build & Run in Knossos**.

Локальные картинки, аудио и glTF/GLB добавляются из dock **VRWeb**: выберите узел, укажите
property (например, `texture`) и type, затем нажмите **Привязать local asset…**. Пустой property
создаёт `ExtScene`. Maker Kit положит такие файлы в content-addressed `dist/assets/...` и
создаст manifest; `.godot/imported` публиковать не нужно.

## 3. Добавить поведение

Прикреплённый к Godot-узлу GDScript является только локальным editor/runtime-кодом и намеренно
не экспортируется. Переносимое поведение мира пишется на Luau и выполняется Knossos в sandbox
через API `document`.

В dock **VRWeb → Luau-скрипты мира**:

1. выберите **Новый скрипт…**;
2. задайте стабильный id, например `my-world.behavior`;
3. вставьте Luau в поле inline source и нажмите **Сохранить скрипт**;
4. выберите нужный узел, задайте ему **Queryable id**, например `door`, и обращайтесь к нему
   как `document.query("#door")`;
5. сохраните `.tscn`, затем выполните Build & Run.

Минимальный скрипт:

```lua
local door = document.query("#door")
assert(door, "#door is missing")
document.log.info("world script started")
door.on("activate", function(_event)
  door.set("visible", false)
end, "Открыть дверь")
```

Можно указать linked `src` вместо inline source; одновременно задавать оба нельзя. Относительный
linked-файл должен лежать рядом с итоговым HTML на хостинге, а его SRI можно задать в поле
`integrity`. Maker Kit автоматически комплектует локальные media/glTF, но linked Luau v1 пока
нужно копировать в `dist` самому. Для первого мира предпочтителен inline source: он попадает в
HTML и не может потеряться при передаче.

Контракт, lifecycle и модель sandbox: [VRWeb scripting](space/scripting.md). Все доступные
методы, значения, лимиты и ошибки: [Scripting API](space/scripting-api.md). Копируемый большой
пример с input, update и общим состоянием: [state-switch demo](client/state-switch-demo.md).

## 4. Проверять и отлаживать

Рабочий цикл: **сохранить сцену → Build & Run → прочитать review → проверить мир → повторить**.
Build & Run всегда использует strict-профиль, пишет `dist/<scene>.html` и sibling
`<scene>.report.json`, затем запускает настоящий Knossos. Повторное нажатие пересобирает тот же
путь и открывает новую runtime-сессию.

Ошибки делятся на три уровня:

| Где произошла ошибка | Где смотреть | Что делать |
|---|---|---|
| Export/strict policy | export review, dock VRWeb и Godot Output | исправить все Errors; Warning означает известную потерю/риск и требует осознанной проверки |
| Загрузка страницы или asset | лог Knossos, теги `builder`, `resload`, `extres` | проверить URL, регистр пути, manifest, MIME/CORS и наличие файла |
| Luau compile/runtime | лог Knossos, тег `scripts`, сообщения `document.log.*` | искать script id и phase, затем уменьшить до минимального кода и проверить queryable ids/API |

На macOS общий лог Knossos находится в
`~/Library/Application Support/Godot/app_userdata/knossos/logs/`; точная модель файлов и пути
для остальных платформ описаны в [logging.md](logging.md). В сообщении об ошибке скрипта важны
`script id`, фаза (`fetch`, `compile`, `mount`/runtime) и текст. `document.log.debug/info/warning/error`
помогает ставить контрольные точки, не раскрывая Godot objects в sandbox.

Не обходите ошибку переходом на `compatible`, если мир предназначен для публикации. Этот
профиль нужен для диагностики миграции; переносимость подтверждает только `strict`. Если Godot
preview работает, а Knossos нет, доверяйте Knossos: только он исполняет опубликованный формат.

Для воспроизводимой проверки без UI используйте CLI из [Maker Kit](maker-kit.md#установка-и-первый-build).
Код возврата ненулевой при блокирующей ошибке, а JSON report можно приложить к bug report.

## 5. Где искать ответ

Читайте документацию от задачи к деталям:

- этот документ — последовательность действий;
- [Maker Kit](maker-kit.md) — установка, кнопки, strict policy, assets, CLI и hosting;
- [VRWML tags](space/vrwml-tags.md) — что можно поместить в сцену;
- [Scripting API](space/scripting-api.md) — что может Luau;
- [экспорт и импорт](vrweb-export.md) — устройство инструментов и тестовые команды;
- [roadmap](roadmap.md) — известные ограничения и ещё не реализованные возможности.

Кнопка **Открыть руководство разработчика** в dock открывает локальную копию этой инструкции.
Completion для ручного HTML доступен в VS Code из корня Maker Kit.

Если ответа нет, сначала проверьте текущий `roadmap`: отсутствие функции может быть известным
ограничением, а не ошибкой. Затем создайте issue в репозитории проекта. Чтобы вопрос можно было
воспроизвести, приложите:

- версии Maker Kit, Knossos и Godot из `compatibility.json`;
- минимальную `.tscn` или HTML/Luau без приватных assets;
- шаги, ожидаемый и фактический результат;
- `<scene>.report.json` и релевантный фрагмент лога с script id/phase;
- платформу и указание, повторяется ли проблема в starter-проекте.

Если в ходе ответа обнаружился новый контракт или устойчивое ограничение, обновите подходящий
файл в `docs`; новые незавершённые задачи заносятся только в `roadmap.md`.

## 6. Упаковать и поделиться

Для мира с inline Luau и local assets достаточно результата strict build:

1. выполните Build & Run и устраните Errors;
2. возьмите `dist/<scene>.html`, а при наличии локальных assets — также
   `<scene>.assets.json` и каталог `dist/assets/<scene>/`;
3. если используются linked scripts, добавьте их в те относительные пути, которые указаны в
   `src`; внешние HTTP(S)-ресурсы в пакет не входят;
4. передайте эти файлы с сохранением структуры — архивом либо загрузите на статический
   HTTP(S)-хостинг;
5. после загрузки запустите published verifier из
   [Maker Kit](maker-kit.md#проверка-опубликованного-сайта).

Получателю локального архива достаточно распаковать его без переименования вложенных файлов и
открыть HTML в Knossos. Для совместной разработки передавайте не `dist`, а весь authoring-проект
(`.tscn`, assets, Luau и настройки) через source control; каталог `.godot` и сгенерированный
`dist` можно пересоздать. Никогда не заменяйте чужой проект содержимым нового starter-а при
обновлении Maker Kit.
