# Реализация Phase 1 — Singleplayer 3D Browser

Документ описывает **как фактически устроен код** первого вертикального среза:
ввод URL → загрузка HTML → топология → геометрия → прогулка от первого лица.
Логические правила см. в [vrweb-overview.md](vrweb-overview.md) и
[html-to-3d-topology.md](html-to-3d-topology.md) — здесь их воплощение.

## Пайплайн

```
адрес → PageFetcher → HTML → HtmlParser → дерево HtmlNode
      → TopologyBuilder → артефакт Space (Dictionary, без координат)
      → WorldGenerator (+seed) → 3D-узлы в `world`
      → Player ходит, активирует порталы → навигация (swap пространства)
```

Граница топология/геометрия соблюдена: `WorldGenerator` принимает только `Space`
(сериализуемый Dictionary) и `seed`, ни одной 3D-координаты в топологии нет.
Это точка отладки — `tests/test_topology.gd` гоняет пайплайн без 3D и печатает артефакт.

**Визуальная отладка топологии.** Сцена `scenes/topology_debug.tscn` останавливает
пайплайн на артефакте `Space` и рисует его 2D-графом (`TopologyGraphView`): узлы —
комнаты/соединители, рёбра — связи родитель→ребёнок. Две раскладки (кнопка «вид»):
**дерево** — нисходящее «tidy tree» (глубина → Y); **паук** — радиальная (корень в центре,
глубина → радиус, поддеревья расходятся равными секторами на 360°).
ЛКМ-драг — панорама, колесо или кнопки ＋/－/⊡ — зум, клик по узлу — панель деталей (вид,
hints, объекты с функциями переходов и превью контента, якоря, **исходный HTML-кусок**).
Геометрия (`WorldGenerator`) и игрок не задействованы — это проверка результата контракции
`TopologyBuilder` глазами.

Карта «узел → HTML» строится опционально: `TopologyBuilder.build(doc, true)` добавляет в
артефакт ключ `"sources"` (`id` → реконструированная разметка через `HtmlNode.to_html()`).
В проде (`main.gd`) флаг `false` — артефакт остаётся чистым контрактом без разметки.

## Ключевые решения реализации

- **HTML-парсер свой, на GDScript.** Нативного нет, `XMLParser` требует валидный XML.
  Парсер прощает незакрытые теги, void-элементы, комментарии, doctype, raw-text
  (`script`/`style`). Достаточно для «HTML уже готов, без JS».
- **Контракция = рекурсия `_process(node) → {rooms, loose}`.** ≥2 готовых комнаты под
  узлом → узел становится **соединителем**, слияние вверх останавливается; свободные
  листья декорируют ближайшее пространство; цепочки-«леса» с одним ребёнком схлопываются
  (CSS-фон/рамка блокируют схлопывание). Списки → один объект-коллекция.
- **Портал — capability поверх объекта.** `<a href>` классифицируется по содержимому,
  `href` навешивает `function: Transition`. В геометрии любой объект с `function`
  инстанцируется как `Portal`.
- **Абзац = один блок, а не россыпь объектов.** Контейнер, всё содержимое которого —
  phrasing/inline (`<p>` с вложенными `<a>`, `<sup>`, `<span>` и т.п.), сворачивается в
  ОДИН rich-text объект. Его `content.runs` — упорядоченные «прогоны»
  `{text, function?}`: обычный текст склеивается, каждая `<a href>` становится прогоном
  со своей `Transition`. Парсер сохраняет граничные пробелы (`_collapse_ws`), иначе
  inline-ссылки слипаются с текстом. В геометрии такой объект (если есть ссылки)
  рендерится актором `RichPanel`: `RichTextLabel` с BBCode в `SubViewport` → текстура на
  3D-квад; клик игрока мапится в координаты вьюпорта и синтетически прокидывается в
  `RichTextLabel` (`push_input`), тот эмитит `meta_clicked` → переход. Порталы и
  inline-ссылки унифицированы: оба шлют `Transition` в один обработчик
  `main._activate_transition`. Объекты `interact_at(point)` — общий интерфейс для луча игрока.
- **Подсветка прицела.** Кроме `interact_at`, активные объекты реализуют
  `is_active_at(point) -> bool`: `Portal` всегда `true`, `RichPanel` — `true` только когда
  под прицелом реальная inline-ссылка. У `RichTextLabel` нет публичного «meta под точкой», а
  `meta_hover_started/ended` по *синтетическому* движению мыши во вьюпорте не срабатывает (в
  отличие от клика — проверено). Поэтому `is_active_at` «кликает» вхолостую под флагом
  `_probing`: `_on_meta_clicked` тогда не делает переход, а только выставляет `_probe_hit` —
  переиспользуем единственный рабочий канал hit-теста. `Player` каждый кадр опрашивает
  коллайдер луча и при смене состояния шлёт сигнал `aim_target_changed(active, hint)`; `main` по
  нему меняет вид прицела (нейтральный «+» → акцентный кружок «○»). `ImagePanel` тоже реализует
  `is_active_at` (всё полотно — ссылка, если несёт `Transition`).
- **Превью ссылки в строке статуса.** Как в браузере «в углу написано, куда ведёт ссылка»:
  если объект под прицелом куда-то ведёт, в нижнюю строку статуса (`main._status`) пишется
  «куда», а как только прицел уходит «в никуда» — поле очищается. Активные объекты, помимо
  `is_active_at`, отвечают `aim_hint_at(point) -> String` — человекочитаемое назначение через
  общий форматтер `TransitionText.describe(transition)` (`scripts/transition_text.gd`):
  `→ href` / `↪ #anchor` / `↩ назад`. `RichPanel` переиспользует тот же пробный клик: при
  `_probing` ловит не только факт попадания, но и сам `Transition` под точкой (`_probe_transition`),
  поэтому `aim_hint_at` не делает второго hit-теста. У объектов без назначения (например, экран
  видео — интерактивен, но никуда не ведёт) `aim_hint_at` нет, и строка остаётся пустой, а курсор
  всё равно подсвечивается.
- **Ссылки внутри списков и таблиц не теряются.** Элемент `<li>` и ячейка `<td>`/`<th>`
  линеаризуются в `runs` так же, как абзац (см. `_build_runs`). В геометрии, если в списке
  или таблице есть хоть одна ссылка, объект рендерится `RichPanel`'ом (пункты через «•»,
  ячейки через «|», строки — переводом), и каждая `<a href>` остаётся кликабельной; без
  ссылок список/таблица остаются дешёвым `Label3D`. Так любой `<a href>` — будь то часть
  текста в пункте, ячейка таблицы, картинка-ссылка (становится `Portal`) или inline-ссылка
  в абзаце — функционален и попадает в мир.
- **Геометрия — единый генератор `SpaceLayout` + 3D-сборка из клеток.** Раскладку (формы комнат
  из пентамино, их позиции на сетке клеток и коридоры-дорожки) целиком считает `SpaceLayout` —
  тот же генератор, что кормит отладочный вид сверху ([geometry-lab.md](geometry-lab.md)).
  `WorldGenerator` свою раскладку **не дублирует**: он только строит геометрию из готовых
  клеточных футпринтов. Клетка `SpaceLayout` = `GRID` (3 м, = ширине коридора/двери), поэтому
  каждая стена ложится на линию сетки.
  1. **Замер объектов** (`_measure_objects`/`_measure_object`): футпринт `{w, h}` каждой панели
     тем же кодом, что её строит (`_panel_size`, `RichPanel.estimate_height_m`, размеры
     картинки как `ImagePanel`). Обычная `Label3D`-табличка получает динамическую ширину:
     короткие подписи не занимают весь `PANEL_WIDTH_M`, длинные тексты капятся этим потолком.
     В `_object_size` (нужны для упаковки, стен и высоты комнаты).
  2. **Раскладка** (`_consume_layout`): `SpaceLayout.new().build(space, seed)` → клетки комнат,
     связи-коридоры, внутренние маршруты. WorldGenerator переводит локальные клетки в
     **абсолютные** на общей сетке: `_foot/_foot_cells` (футпринт-множество и список клеток),
     `_door_edges` (рёбра-двери — где связь пробивает стену), `_corr_cells` (клетки полов-коридоров),
     `_room_entrance` (проём-вход от родителя), `_positions/_shift` (мировые центры и сдвиг корня к
     началу координат). Высота стен = `max(WALL_HEIGHT, самый_высокий_объект + HEAD_CLEARANCE)`.
  3. **Полы** — по GRID-квадрату на каждую клетку футпринта (и `_build_corridor_floors` — на каждую
     клетку коридора). Пол сплошной по клеткам ⇒ игрок ходит свободно. Коридоры **огорожены
     стенами** (`_add_corr_wall`) по открытым краям клеток — кроме сторон, где улица продолжается
     или примыкает комната (там стена/дверь самой комнаты), так что проходы остаются открытыми.
  4. **Стены по периметру** (`_wall_runs`/`_add_wall_run`): граничное ребро клетки (снаружи нет
     клетки той же комнаты) даёт стену; смежные коллинеарные рёбра склеиваются в прямой **пробег**.
     Рёбра-**двери** (`_door_edges`) из пробегов исключаются → на их месте **проём** (пробег рвётся
     надвое). Стена сдвинута внутрь на полтолщины (внешняя грань на границе клетки) — у вплотную
     стоящих комнат стены стыкуются грань-в-грань, без z-fight.
  5. **Двери/коридоры из связей** (`corridors[*].path`): концы пути лежат в комнатах (там проёмы —
     `_door_edges`/`_room_entrance`), середина — свободные клетки (полы-коридоры). `adjacent`-связь
     = прямая дверь между примкнувшими комнатами (путь из 2 клеток); иначе коридор по улицам.
     Связи `unrouted` (родитель замурован) проёма не дают — комната остаётся изолированной (редко).

  `seed` детерминирует раскладку через `SpaceLayout`; `_rng` в WorldGenerator сидится лишь для
  цветов/атмосферы ⇒ один URL = идентичная геометрия.
- **Объекты — на развёртке виртуальных стен комнаты** (`_compute_object_layouts`,
  `_object_boxes`, `_object_placements`). Рёбра `virtual_walls` из `SpaceLayout` склеиваются в
  непрерывные `wall_boxes`; углы, разрывы и двери режут развёртку. Объекты измеряются до
  раскладки (`_measure_objects`) и жадно пакуются в boxes по реальным метрам: слева направо,
  рядами, с отступом от краёв box и между объектами. Каждый box стартует высотой 1 клетку; если
  места не хватило, генератор увеличивает очередной box на 1 клетку по высоте и полностью
  пересчитывает layout, проходя boxes по кругу, пока все объекты не получат placement.
  Placement задаёт box, координату вдоль стены, вертикальный offset и `pull`; объект притянут к
  стене (`PULL_INSET`) и смотрит назад к дороге. Мировой визуальный низ объекта =
  `OBJECT_FLOOR_OFFSET + placement.y`; сейчас `OBJECT_FLOOR_OFFSET = 1.0`, чтобы все объекты
  висели примерно на метр выше пола и этот baseline можно было менять одной настройкой.
  Клетку-дверь и стену с дверью объект **не занимает никогда** (`_wall_clear` проверяет ребро
  притяжения и ±1 клетку вдоль стены). Без виртуальных стен (редко) — фолбэк по клеткам
  футпринта лицом к центру (`_fallback_slots`) с вертикальной стопкой.
- **Заголовок секции — вывеска над входом в комнату, читаемая с обеих сторон**
  (`_room_title_object`, `_build_room_title`, `_add_title_label`). Первый объект комнаты типа
  `heading` (его кладёт топология, §D) на стену **не вешается** — вместо этого садится
  лентой-перемычкой в верх **входного проёма** (вход от родителя — `_room_entrance`, заполняется
  в `_consume_layout` из конца пути связи на стороне ребёнка). Текст дублируется двумя
  `Label3D` — лицом внутрь и наружу (`inward_yaw` и `+PI`, сдвиг на грань ленты), поэтому
  название видно и из комнаты, и снаружи перед входом. Высота ленты подгоняется под кегль
  уровня (`HEADING_EM`, кап `TITLE_MIN_H..TITLE_MAX_H`), ширина = проём + `TITLE_OVERHANG`.
  У **корня** входа нет (`_room_entrance` пуст) — там заголовок остаётся обычной настенной
  табличкой (фолбэк `_build_room_title → false`).
- **Панели висят на уровне глаз, а не лежат на полу.** Центр каждой текстовой/картиночной
  панели ставится на `EYE_LEVEL` (1.6 м = высота камеры игрока): мелкие таблички висят прямо
  перед лицом, высокие приподняты ровно настолько, чтобы низ не вжимался в пол
  (`maxf(EYE_LEVEL, h/2 + GAP)`; в `_build_panel`, `_build_rich_panel`, `ImagePanel._update_layout`).
  Раньше панели росли от пола вверх — приходилось задирать взгляд.
- **Текст растёт вширь, длинный — скроллится, а не башней вверх.** `RichPanel` шире
  (`PANEL_WIDTH_PX = 960`) — тот же абзац ложится в меньшее число строк. Высота капнута
  `MAX_HEIGHT_PX` (~2.6 м): если текст выше — панель фиксируется на потолке и включает
  внутренний скролл (`scroll_active`), колесо мыши **или прокрутка тачпада** (два пальца)
  над панелью прокручивает её (`Player._do_scroll` → `RichPanel.scroll_by`). Тачпад на macOS шлёт
  прокрутку не колесом, а `InputEventPanGesture` (`delta.y`) — обрабатываются оба источника.
  Луч прокрутки **отдельный и длиннее** луча
  взаимодействия (`SCROLL_REACH = 12 м` против 3.5 м у портала): абзац читают издали, а не
  вплотную, как жмут портал; запрос к `direct_space_state` гасится в `_physics_process` (там
  он безопасен), тики колеса копятся в `_scroll_pending`. Штатный скроллбар `RichTextLabel` в
  `SubViewport` не показывался (казалось,
  что текст просто обрезан) — поэтому его прячем (`_hide_native_scrollbar`) и рисуем **свой
  индикатор** `_thumb` (яркий бегунок справа, `_update_thumb`): высота = доля видимого окна,
  позиция = доля прокрутки; он всегда виден, когда есть что листать. Тот же потолок зашит в
  `estimate_height_px`, чтобы футпринт для раскладки мира
  не раздувался. **Высота — по факту, а не по оценке.** Оценка `estimate_height_*` (по длине
  текста и переносам) систематически занижала высоту → панели обрезались снизу. После реальной
  раскладки текста (`_refit`, отложен на 2 кадра) панель берёт `get_content_height()` и
  пересобирает квад точно под контент (или под потолок со скроллом). Оценка остаётся только для
  предварительной раскладки комнат (футпринт стены), фактическая геометрия её уточняет.
- **Спавн — у первого объекта страницы, лицом к нему** (`_find_first_object_id`,
  `_record_spawn`, `_compute_spawn`; `Player.teleport_to(point, face_target)`). Первый объект
  в порядке чтения — спуск по первым детям до комнаты со своими объектами (наполнение раньше
  подкомнат, заголовок секции — её первый объект). При его расстановке запоминаем мир-позицию,
  направление от стены внутрь комнаты и потолок отступа; игрок встаёт перед объектом (отступ
  3.5 м, клампится, чтобы не вылезти за противоположную стену) и разворачивается лицом к нему
  (только yaw, взгляд по уровню). Делается на каждой перегенерации мира (новая страница = новое
  «первое появление»). Нет объектов — запасной спавн у края корневой комнаты. **Если первый
  объект — заголовок, ушедший вывеской над входом** (не корень), он на стену не ставится, и
  спавн задаётся напрямую (`_record_title_spawn`, флаг `_spawn_done`): игрок встаёт сразу за
  титульным проёмом лицом в комнату — въезд «через заголовок».
- **Размеры объектов — относительно базового текста страницы** (топология §13). Топология
  кладёт в артефакт `typography.base_px` (средневзвешенный по длине текста кегль; заголовки
  не в счёт) и `content.width_px`/`height_px` у картинок (из `width`/`height`/inline-style,
  только абсолютные px). Геометрия выводит ОДИН коэффициент `_m_per_px = M_PER_BASE_LINE /
  base_px` и размеряет им всё: кегль Label3D (`_godot_font`), размер табличек (`_panel_size`),
  кегль `RichPanel` (`setup(runs, font_world_m)`), квад `ImagePanel` (`setup(..., want_w, want_h,
  fallback_w)` из `width_px`/`height_px`, иначе стартовая fallback-ширина `ImagePanel.BASE_WIDTH`
  до загрузки текстуры). Если у картинки без HTML-размера пришла текстура, `ImagePanel`
  уточняет размер по натуральным пикселям с cap'ом и через `size_changed` запускает локальный
  reflow комнаты (`_on_object_size_changed` → `_pack_room_objects` → `_reposition_room_objects`).
  Заголовки масштабируются по уровню (`HEADING_EM`, дефолты браузера h1..h6).
  Итог: картинка во столько раз крупнее строки, во сколько на странице; `h1` заметно больше
  `h6` — иерархия страницы переносится в пространство, объекты перестают быть одного размера.
- **Сквозь объекты комнаты можно проходить.** Кликабельные панели (`RichPanel`, `ImagePanel`)
  держат коллайдер на **слое 2** (`collision_mask = 0`) — только для луча взаимодействия.
  Тело игрока (маска = слой 1) их не задевает, а его `RayCast3D` (`collision_mask = 3`) их
  по-прежнему ловит. Стены/пол/порталы остаются на слое 1 и блокируют движение. Текстовые
  `Label3D`-панели коллайдера не имеют вовсе.
- **Картинки грузятся прогрессивно, как на сайте.** В геометрии объект-картинка (`<img>`
  или `<a><img></a>`) сразу инстанцируется актором `ImagePanel` с заглушкой «🖼 alt» — мир
  собирается мгновенно, не дожидаясь сети. После сборки панель просит свою текстуру у
  `ImageLoader` (живёт ВНУТРИ мира): тот резолвит относительный `src` через
  `PageFetcher.resolve_url`, качает байты пулом `HTTPRequest` (ограниченная параллельность
  `MAX_CONCURRENT`, очередь, кэш по URL), декодирует (png/jpg/webp/gif, опц. svg; тип — строго по
  `Content-Type`, иначе по расширению — БЕЗ перебора кодеков, чтобы не засыпать консоль
  ошибками декодера на каждой картинке), ужимает большие до 1024 px
  и отдаёт `ImageTexture` (анимированный GIF — `AnimatedTexture`, см. docs/gif-support.md).
  Панель заменяет заглушку текстурой и подгоняет пропорции квада
  под пропорции картинки; не удалось — остаётся заглушка (с её размером из HTML
  `width`/`height`, т.е. пропорции сохраняются) и метка «✕».
  - **Ограничение движка: grayscale-JPEG с сэмплингом ≠ 1×1.** Встроенный в Godot декодер
    `jpgd` (и в 4.3, и в 4.4) падает на одно­компонентных (ч/б) JPEG, у которых фактор
    сэмплинга единственной компоненты не 1×1 — такие отдаёт, например, Wikimedia для ч/б фото
    (`Nf=1`, `H2V2`). Вызов `load_jpg_from_buffer` на них возвращает `ERR_PARSE_ERROR` и пишет
    ошибку `_jpegd_mem_loader_func` в консоль. Поэтому `ImageLoader._jpeg_unsupported` заранее
    сканирует маркеры JPEG до `SOF` и, если это `Nf=1` с H/V > 1, отдаёт `null` НЕ зовя декодер
    (картинка получает заглушку, консоль чистая). Цветные (`Nf=3`) и grayscale с `H1V1`
    декодируются нормально. Полноценно показать такие ч/б фото можно только сторонним
    декодером (GDExtension на базе stb_image/libjpeg-turbo) — пока вне Phase 1. Картинка-ссылка несёт `Transition`
  прямо в `ImagePanel` (кликается лучом как `Portal`/`RichPanel`), отдельным порталом её НЕ
  оборачиваем — иначе потеряли бы изображение. Топология для `<a><img></a>` теперь
  сохраняет `src` вложенного `<img>` (см. `_anchor_object`). Лоадер в мире => при навигации
  старый мир сносится вместе с ним и незавершённые загрузки прошлой страницы умирают.
- **Якоря — метки.** `#id` резолвится в позицию комнаты (или комнаты, содержащей объект
  с этим id). Кросс-страничные ссылки = `navigate` → fetch + перегенерация + swap.
- **История навигации — браузерная** (`main._history` + `_history_index`). История это
  список записей `{url, pose}` и индекс текущей; кнопки `back`/`fwd` в navbar двигают
  индекс (`_go_back`/`_go_forward`), новая навигация (`_navigate` с `push_history=true`)
  обрезает ветку «вперёд» и добавляет запись. В записи хранится финальный URL (после
  редиректа, проставляется в `_on_fetched`). Покидая страницу, сохраняем позу игрока
  (`Player.get_pose`: позиция + yaw + наклон камеры) в текущей записи; при возврате
  назад/вперёд `_pending_restore_pose` восстанавливает её через `Player.restore_pose`
  поверх дефолтного спавна — мир детерминирован по URL, поэтому поза остаётся валидной.
  Доступность кнопок — `_update_nav_buttons`. Портальный переход `back` зовёт тот же
  `_go_back`.
- **Атмосфера процедурна и пер-страничная** (`WorldGenerator._build_atmosphere`). Небо и
  солнце генерируются по данным страницы и пересоздаются при каждой навигации (старые
  узлы сносятся вместе с пространством в `_rebuild_world`). Палитра берётся из **визуального
  паспорта документа** (`artifact.document`, см. [world-visualization.md](world-visualization.md)):
  фон-картинка `<body>` → небо-панорама; иначе `background-color` → база неба; `color` (или очень
  тёмный фон) → земля; стены/полы — в тон фона документа. Если у документа нет цвета — откат к
  среднему CSS-фону комнат и производной от `seed` URL (прежнее поведение). «Богатство» контента
  (вес корня) задаёт высоту солнца (пустая страница → низкий закат, насыщенная → высокий
  день); `seed` задаёт азимут теней; чем ниже солнце, тем теплее свет. Детерминировано:
  один сайт = одно и то же небо. В `main._setup_environment` остаётся лишь нейтральный
  свет для стартового экрана до первой загрузки.
- **Падение с краёв** (открытые проёмы) ловится: ниже `fall_limit` игрок респаунится.

## Управление

- **WASD** — движение, **мышь** — взгляд, **Shift** — бег, **Space** — прыжок.
- **ЛКМ / E** — активировать портал под прицелом (центральный «+»). При наведении на
  активный объект (портал или inline-ссылка) прицел подсвечивается кружком «○», а внизу в
  строке статуса показывается, куда ведёт ссылка (`→ href` / `↪ #anchor` / `↩ назад`).
- **Esc** — двухступенчато. В браузинге мира (мышь захвачена) — освобождает курсор (чтобы
  кликнуть адресную строку); клик по 3D — снова захват. Когда мышь уже свободна (возимся с
  UI) — открывает настройки: `Player.settings_requested` → `main._open_settings`. Открытые
  настройки тот же Esc закрывает (их собственный `_unhandled_input`, помечает событие
  обработанным, так что до игрока он не доходит).
- **Enter** (в браузинге, онлайн) — открыть строку чата: мышь освобождается и фокус встаёт
  в поле ввода. Повторный Enter отправляет сообщение (пустой — просто закрывает) и возвращает
  в браузинг. Реализовано сигналом `Player.chat_requested` → `main._on_chat_requested`, обратный
  путь — `main._on_chat_submitted` (всегда `capture_mouse(true)`).
- **Браузинг мира и UI взаимоисключающи.** Пока мышь захвачена (ходим по миру), элементы
  навбара/чата сделаны нефокусируемыми (`focus_mode = FOCUS_NONE`), а текущий фокус снят —
  клавиатура (Tab/Space/Enter) их не достаёт, иначе Space «дожимал» бы `go`, а Tab уводил
  фокус в поле чата. Это централизовано в `Player.capture_mouse`: при `on=true` он шлёт
  `mouse_capture_changed`, на который `main._set_ui_focusable` переключает фокусируемость.
  При освобождении мыши (Esc / клик по адресной строке) UI снова фокусируем.
- **F3** — отладочный инспектор провенанса (см. ниже).
- Адресная строка → `go!` (или Enter): загрузка и сборка пространства.

## Отладочный инспектор провенанса (F3)

Режим «откуда взялся объект»: при наведении прицела на любой узел мира оверлей в правом
верхнем углу показывает его происхождение — тип топологии и исходный HTML.

Цепочка данных:
1. `TopologyBuilder.build(doc, true)` (в `main._on_fetched`) включает запись провенанса:
   артефакт получает ключ `sources` (`id -> реконструированный HTML` узла/секции). В офлайн-
   контракте топология↔геометрия этого ключа нет — он чисто отладочный.
2. `WorldGenerator` при наличии `sources` (`_debug = space.has("sources")`) вешает на каждый
   узел метаданные `vrweb_debug` (константа `DEBUG_META`) — готовую многострочную строку:
   - комнаты/коннекторы (`_room_debug_text`): тип, семантический тег, вес/степень, css, HTML;
     строка садится на холдер `Room_N` (его прямые дети — коллайдеры пола и стен);
   - объекты (`_object_debug_text`): тип, функция перехода, родная комната, текст, HTML;
     строка садится на корневой узел объекта (панель/портал/картинка), который и есть коллайдер.
3. `Player._debug_probe` (каждый кадр в режиме F3) бьёт луч из камеры на `DEBUG_REACH` по слоям
   стен и панелей и поднимается от коллайдера к ближайшему предку с `vrweb_debug`
   (`_find_debug_meta`); текст летит наружу сигналом `debug_probed` (эмит только при смене).
4. `main._on_debug_probed` рисует его в оверлее; `debug_toggled` показывает/прячет панель.

Коридорные полы провенанса не несут (генерируемая инфраструктура, не из топологии) — наведение
на них даёт пустой текст.

## Известные ограничения (кандидаты на следующие итерации)

- **Картинки грузятся** (прогрессивно, см. выше), но: `svg` зависит от наличия кодека в
  сборке Godot; некоторые хосты режут запрос по `User-Agent` (отдают `4xx` — остаётся
  заглушка с «✕»); `data:`-URI и `srcset`/`<picture>` пока не разбираются (берём только `src`).
- **Пороги контракции не настроены** — на «div-soup» возможна фрагментация на много мелких
  комнат. Это и есть главный пункт для калибровки (см. §12 топологии): гонять на реальных
  сайтах и подбирать `threshold`.
- **Каждый `RichPanel` = свой `SubViewport`** (render target). На странице с сотнями
  абзацев это тяжело по памяти/времени старта. Сейчас `UPDATE_ONCE` гасит постоянную
  перерисовку; короткий текст без ссылок остаётся дешёвым `Label3D`. Кандидат на оптимизацию:
  пул вьюпортов, общий атлас или MSDF-шрифты вместо текстуры-на-абзац.
- **Высота `RichPanel` оценивается эвристикой** (длина текста), а не реальным
  `get_content_height` — длинные абзацы могут чуть не совпасть с фактической вёрсткой; тот же
  замер идёт в раскладку (`RichPanel.estimate_height_m`), так что промах сдвигает и футпринт.
- **JS не исполняется** — SPA придут пустыми (ожидаемо для Phase 1).
- **Раскладка комнат считается `SpaceLayout`, не оптимум.** Качество зависит от его параметров
  (`alcove_fill_chance`, `pull_to_parent_chance` и пр.) и сида. `GRID` (=ширине коридора) задаёт
  «ступенчатость» самих комнат и маршрутов, но объекты внутри комнаты пакуются уже по метрическим
  размерам на развёртке виртуальных стен.
- **Коридоры/двери из связей `SpaceLayout`.** Концы пути связи дают проёмы у комнат, середина —
  полы-коридоры. Пересечений комнат коридоры не дают по построению; связь `unrouted` (замурованный
  родитель, редко) оставляет комнату изолированной. Гэп проёма ловится ловушкой падения.
- **Картинка без HTML-размера уточняется после загрузки.** До текстуры она занимает
  fallback-ширину `ImagePanel.BASE_WIDTH`; после загрузки берётся натуральный размер в общей
  метрике, ограниченный `ImagePanel.NATURAL_MAX_WIDTH`/`MAX_HEIGHT`, и комната локально
  перепаковывается. Reflow меняет позиции объектов, но не перестраивает уже созданную геометрию
  стен.
- **`entry` составного поддерева** (§8 топологии) ещё не введён: проход всегда идёт в
  верхний узел поддерева.
