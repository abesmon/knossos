# Мультиплеер (Слой 2 / Phase 2)

> Реализация реалтайма из [vrweb-overview.md](vrweb-overview.md): люди, открывшие
> **одну и ту же страницу**, видят капсулы друг друга, синхронизируют позиции и общаются
> текстом. Транспорт — p2p WebRTC mesh; сигнальный сервер нужен только для handshake.

## Что входит в v1

- **Присутствие + позиции** — капсулы других игроков, синхронизация позиции и поворота (yaw).
- **Аватар с лицом** — на передней грани капсулы квад с текстурой лица (с прозрачностью);
  игрок задаёт картинку файлом в настройках.
- **Текстовый чат** — поверх WebRTC data-канала. Лимит сообщения — 280 символов
  (`NetworkManager.MAX_CHAT_CHARS`), режется и на вводе, и на отправке, и на приёме.
  Над капсулой говорящего всплывает речевой бабл (30 c или до следующего сообщения).
  История чата живёт только в RAM — кольцевой буфер последних 50 записей
  (`CHAT_HISTORY_MAX` в [scenes/main.gd](../scenes/main.gd)), на диск ничего не пишется.
  Буфер сквозной между мирами; переход помечается системной строкой
  «— Присоединились к миру `<url>` —». Вид чата зависит от режима: в UI-режиме (мышь
  свободна) виден лог + поле ввода; в режиме перемещения поле скрыто (ввод открывается
  по Enter), а через 30 c после последнего сообщения лог плавно угасает до 10% и снова
  «просыпается» при новом сообщении или возврате мыши.
- **Онлайн/офлайн режим** — переключается на экране настроек, по умолчанию офлайн.

- **Голосовой чат** — микрофон → VAD → **Opus** (аддон twovoip, libopus) поверх того же mesh,
  пространственное воспроизведение из позиции капсулы говорящего. WebRTC API в Godot даёт только
  data-каналы (медиа-треков нет), поэтому Opus-пакеты идут через отдельный RPC-канал. Включается
  отдельным тумблером в настройках (работает только онлайн). Кодек спрятан за `VoiceCodec`
  (раньше был сырой PCM16). Полностью — в [voice-chat.md](voice-chat.md).

## Комната = страница

Идентификатор комнаты — канонический ключ страницы `PageFetcher.seed_key(url)`
([scripts/page_fetcher.gd](../scripts/page_fetcher.gd)): он нормализует схему, регистр
хоста, хвостовой слеш и фрагмент (`#...`), но различает host + path + query. Один и тот
же URL → одна комната у всех. Уход на другую страницу = выход из комнаты.

## Архитектура

```
[Godot A] --WS--\                    /-- offer/answer/ICE --\
                 [signaling (Python)]  (группирует по room=seed_key)
[Godot B] --WS--/                    \-- offer/answer/ICE --/
      \________ после handshake: WebRTC mesh, p2p — позиции и чат напрямую ________/
```

- **Топология:** mesh (`WebRTCMultiplayerPeer.create_mesh`) — все пиры равны, без
  авторитетного хоста. Подходит для эфемерного «схода» людей на странице. Предел —
  десятки пиров в комнате; дальше нужен SFU (далёкий этап, см. overview).
- **Сигнальный сервер** в обмене данными не участвует: только присваивает peer_id,
  группирует по комнатам и пересылает offer/answer/ICE. См. [../signaling/](../signaling/).

## Клиентские компоненты

| Узел | Файл | Роль |
|---|---|---|
| `Settings` (autoload) | [scripts/settings.gd](../scripts/settings.gd) | online-режим, адрес сигнального сервера, ник; персист в `user://settings.cfg` |
| Экран настроек | [scenes/settings.gd](../scenes/settings.gd) / `.tscn` | overlay поверх UI, правит `Settings` |
| `NetworkManager` (autoload) | [scripts/network_manager.gd](../scripts/network_manager.gd) | WS-сигналинг + WebRTC mesh + RPC; жизненный цикл комнаты |
| `RemotePlayer` | [actors/remote_player/](../actors/remote_player/) | капсула другого игрока (интерполяция, нэйм-плейт) |
| `RemotePlayersView` | [scripts/remote_players_view.gd](../scripts/remote_players_view.gd) | создаёт/удаляет капсулы по сигналам, шлёт позицию локального игрока ~15 Гц, маршрутит голос в капсулы |
| Чат UI | в [scenes/main.gd](../scenes/main.gd) | поле ввода + история, видно только онлайн |
| `VoiceManager` (autoload) | [scripts/voice/voice_manager.gd](../scripts/voice/voice_manager.gd) | захват микрофона, VAD, кодирование и отправка голоса; приём — на капсулах. См. [voice-chat.md](voice-chat.md) |

**Почему `NetworkManager` — autoload, а капсулы — в `world`.** Менеджер должен
переживать перестройку мира при навигации (нужен стабильный путь `/root/NetworkManager`
для RPC). Капсулы, наоборот, живут в `world`: при навигации `world` сносится — капсулы
исчезают сами, что и означает «вышел из комнаты». `RemotePlayersView` пересоздаётся в
`main._rebuild_world` и заново читает текущих пиров.

**RPC.** `multiplayer.multiplayer_peer` назначается mesh-пиром. Состояние —
`@rpc("any_peer","unreliable_ordered") _recv_state(pos, yaw, params)`, чат —
`@rpc("any_peer","reliable") _recv_chat(text)`, карточка —
`@rpc("any_peer","reliable") _recv_identity(nick, face_png, avatar_uri)`. Отправитель
определяется через `multiplayer.get_remote_sender_id()`, поэтому отдельный
`MultiplayerSpawner`/авторитет на капсулу не нужен.

**Видео.** Транспорт видео-плееров синхронизируется тем же приёмом:
`@rpc("any_peer","reliable") _recv_video_event(player_id, action, position)` (play/pause/seek,
last-writer-wins) и `@rpc("any_peer","unreliable_ordered") _recv_video_sync(player_id, position,
playing)` (heartbeat: позиция + состояние). Heartbeat шлёт **таймкипер** —
пир с наименьшим id (`NetworkManager.is_timekeeper()`), так что late-join и autoplay
синхронизируются автоматически. Таймкипер — частный случай **авторитета комнаты**
(детерминированный «первому», без переговоров; полностью — в [authority.md](authority.md)). Привязка по `player_id` (страница одна = id одни), применяет
`VrwebVideoManager`. Подробно — в [video-player.md](video-player.md).

**Голос.** Идёт отдельным RPC `@rpc("any_peer","unreliable_ordered","call_remote", 1)`
`_recv_voice(payload)` по **каналу 1** — чтобы голос не делал head-of-line blocking с
состоянием/чатом (канал 0), а ретрансмиты не ломали реалтайм. Под этот канал меш создаётся с
доп. data-каналом: `create_mesh(id, [TRANSFER_MODE_UNRELIABLE_ORDERED])` — иначе `put_packet`
падает с «max channels: 3» (по умолчанию каналов только 3). `NetworkManager` остаётся чистым
транспортом (`send_voice`/`voice_received`); захват и кодек — в `VoiceManager`,
воспроизведение — на капсуле. Подробно — в [voice-chat.md](voice-chat.md).

`params` — словарь параметров аватара (наклон взгляда, Grounded, скорости и т.д.): любой
новый сигнал расширяет систему **без правки сигнатуры RPC**. Контракт и полное описание —
в [avatars.md](avatars.md).

## Аватар: ник, лицо и выбор модели

Лицо хранится в `user://face.png` всегда как **256×256 PNG с альфой** (при первом запуске
туда копируется `res://resources/default_face.png`). В настройках можно выбрать свой файл:
`Settings.set_face_from_file()` конвертирует в RGBA8, ресайзит до 256×256 и сохраняет —
это и есть «всегда ресайз перед сохранением на устройство».

**Передача.** Как только p2p-канал к пиру открывается (`multiplayer.peer_connected`),
ему уходит «карточка» — ник, PNG-байты лица и идентификатор аватара (`Settings.avatar_uri`)
одним reliable-RPC `_recv_identity` (`NetworkManager._on_mp_peer_connected`). Приёмник
декодирует PNG в `ImageTexture` и эмитит `identity_received(id, nick, face, avatar_uri)`;
`RemotePlayersView` накладывает лицо через маркер-текстуру и резолвит `avatar_uri` в модель
(см. [avatars.md](avatars.md) → «Выбор аватара»). До прихода карточки капсула показывает
дефолтный аватар и лицо. Смена ника/лица/аватара в настройках рассылается уже подключённым
пирам через `NetworkManager.broadcast_identity()`.

`_recv_state` синхронизирует позицию, yaw и словарь параметров аватара. Yaw разворачивает
всю капсулу (тело `RemotePlayer`); остальные сигналы (наклон взгляда `LookPitch`, скорости,
Grounded…) уходят в шину аватара, и **аватар сам решает**, как их отобразить. Дефолтный
аватар (`DefaultAvatar`) наклоняет к небу/земле только квад лица с редуцирующим множителем
`FACE_PITCH_FACTOR` (0.35) — прежнее поведение. Идентичность (ник/лицо) теперь идёт в аватар
через `Avatar.apply_identity`. Подробности системы аватаров — в [avatars.md](avatars.md).

## Предусловие: нативный WebRTC-аддон

Официальные **десктоп**-сборки Godot не включают WebRTC. Нужен GDExtension-аддон
`webrtc-native` (репозиторий `godotengine/webrtc-native`, релиз под Godot 4.4):

1. Скачать архив релиза, распаковать в `addons/webrtc/` (там `.gdextension` + бинарники
   под платформы).
2. Перезапустить проект — должны резолвиться классы `WebRTCPeerConnection` и
   `WebRTCMultiplayerPeer`.

Без аддона приложение **работает в офлайне** как раньше: `NetworkManager` не использует
статическую типизацию WebRTC-классов и при попытке выйти в онлайн без аддона выдаёт
предупреждение (`NetworkManager.webrtc_available()`), не ломая запуск. Для web-экспорта
WebRTC встроен в браузер — аддон там не нужен.

## STUN / TURN

ICE настроен на публичный STUN `stun.l.google.com:19302` (в `NetworkManager.ICE_SERVERS`).
Для p2p сквозь симметричный NAT понадобится **TURN-сервер** — пока вне v1, выносится в
настройки позже.

## Протокол сигналинга

JSON поверх WebSocket. Полное описание и таблицы — в [../signaling/README.md](../signaling/README.md).
Кратко: сервер шлёт `welcome` (peer_id), `peers`/`peer_join`/`peer_leave` (состав
комнаты) и релеит `offer`/`answer`/`candidate`. **Антиглар:** offer инициирует пир с
меньшим id — ровно одна сторона начинает соединение.

## Федерация (точка расширения, не в v1)

`room` — произвольная строка, комнаты изолированы. Чтобы пиры с разных сигнальных
серверов попадали в одну комнату (как в Matrix), сервер можно научить пересылать `join`
и сигналы дружественным серверам из конфигурации, а peer_id сделать глобально уникальными
(префикс сервера). Это даст возможность хостить свои сигнальные серверы и обходить
блокировки. Клиент уже умеет менять адрес сигнального сервера в настройках.

## Проверка (end-to-end)

1. Поднять сервер: `cd signaling && pip install -r requirements.txt && python server.py`
   (или Docker — см. README сервера).
2. Установить аддон `webrtc-native` в `addons/webrtc/`.
3. Запустить два экземпляра клиента. В обоих: настройки → включить «Онлайн», адрес
   локального сервера `ws://localhost:8080`, разные ники → сохранить. Поля можно очистить
   кнопкой «✕»/«Сбросить» — тогда подставятся дефолты: адрес
   `https://signaling.vrweb.home.syrupmg.ru` (схема приводится к `wss://`), ник
   `Guest-XXXX`, лицо `resources/default_face.png`.

   > **Два инстанса на одной машине делят `user://`** (общий `settings.cfg`/`face.png`), из-за
   > чего смена лица/аватара «протекает» на оба. Чтобы изолировать — задать **dev-песочницу**
   > `Sandbox` (см. [scripts/sandbox.gd](../scripts/sandbox.gd)): запускать экземпляры с разным
   > id, тогда все их `user://`-файлы уходят под `user://<id>/`. Способы: переменная окружения
   > `VRWEB_SANDBOX=A` / `VRWEB_SANDBOX=B`, либо аргумент `-- --sandbox=A`. Без id — обычный
   > общий `user://`. Песочница покрывает **все** `user://`-файлы (настройки, лицо, кэш аватаров).
4. Ввести в обоих **один и тот же URL** → видны капсулы друг друга, движение
   синхронизируется; чат доставляется.
5. Один уходит на другой URL → его капсула исчезает у второго; возврат — снова виден.
6. Выключить «Онлайн» → капсулы и чат пропадают, одиночный режим работает как прежде.
