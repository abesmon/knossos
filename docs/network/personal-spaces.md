# Персональные пространства (модуль домашнего сервера)

> Домашний сервер может предоставлять своим пользователям **персональные пространства** —
> хостимые им страницы, которыми по умолчанию управляет владелец. Пространство — обычная
> vrweb-страница: заходят по URL, комната по URL, редактируется через
> [page-persistence.md](page-persistence.md) (для которого этот модуль — **reference
> implementation**).
>
> Статус: **реализовано** (v1). Сервер — [../homeserver/spaces.py](../../homeserver/spaces.py) +
> [../homeserver/spaces_api.py](../../homeserver/spaces_api.py) (feature `personal-spaces.v1`),
> управление — веб-морда `/space`; клиент — вход по кнопке «Моё пространство» (вкладка
> «Аккаунт») и фолбэком из «Вернуться домой» при пустой домашней странице.

## Персональное пространство ≠ домашняя страница

Две **разные, но связанные** сущности — не путать:

| | **Домашняя страница** (`Settings.home_page`) | **Персональное пространство** |
|---|---|---|
| Что это | произвольная закладка «стартовать отсюда» | хостируемый домашним сервером ТВОЙ дом |
| Где живёт | локальная настройка клиента (URL-строка) | на домашнем сервере, feature `personal-spaces.v1` |
| Каким может быть | ЛЮБОЙ URL (чужой сайт, своё пространство, что угодно) | всегда `https://<домашний сервер>/s/<slug>` |
| Кто владеет/правит | никто — это просто адрес | ты (+ выданные редакторы), правится и персистится |
| Адрес | хранится буквально в настройках | НЕ хранится клиентом — спрашивается у сервера каждый раз (переживает ротацию) |
| Вход в клиенте | «Вернуться домой» (вкладка «Мир») | «Моё пространство» (вкладка «Аккаунт») |

Они **независимы**: можно иметь домашней страницей новостной сайт и **отдельно** ходить в
своё пространство кнопкой на «Аккаунте». Связь только одна, для удобства: если домашняя
страница **не задана**, «Вернуться домой» фолбэком открывает персональное пространство —
пустой старт логично вести в свой дом. Сделать пространство домашней страницей буквально
(записать его URL в `home_page`) не рекомендуется: URL ротируется, а «Моё пространство»
резолвит актуальный адрес сам.

Реализация входов: [../scenes/settings.gd](../../scenes/settings.gd) (`space_requested` /
`home_requested`), [../scenes/main.gd](../../scenes/main.gd) (`_go_to_personal_space`).

## Место в архитектуре: почти всё уже специфицировано

Ключевое свойство модуля — он **не добавляет новых сущностей** в мир. Пространство — это
страница; всё взаимодействие с ней идёт по уже описанным контрактам, и уровень каждого
вопроса определён:

| Вопрос | Уровень | Контракт |
|---|---|---|
| зайти в пространство | обычная навигация по URL | [multiplayer.md](multiplayer.md), комната = `seed_key(url)` |
| увидеть других, голос, эфемерный слой | комната | [ephemeral-changes.md](ephemeral-changes.md) |
| сохранить правки в пространство | страница ↔ её сервер | [page-persistence.md](page-persistence.md): `persist`-атрибут, `rev`, флаш |
| кто имеет право сохранять | внутренняя логика сервера | политика `allowlist`, управляется владельцем через веб-морду |
| узнать адрес СВОЕГО пространства | клиент ↔ домашний сервер | **единственный новый контракт — этот документ** |

Права редактирования (выдать другу доступ, забрать) владелец настраивает на сайте сервера
или ином админ-устройстве — клиенту про это знать не нужно: он просто получает от
capability-запроса персистенции `writable: true/false` и показывает/прячет кнопку сохранения.

## Контракт клиент ↔ домашний сервер (v1)

Сервер анонсирует feature **`personal-spaces.v1`** в `/.well-known/vrweb`. Клиент, увидев
её у своего сервера (и залогинившись), получает ровно две возможности:

| Метод и путь | Auth | Что делает |
|---|---|---|
| `GET /api/v1/spaces/home` | Bearer | `{ "url": "https://example.com/s/x7Qk…", "name": "…" }` — актуальный адрес своего пространства (создаётся лениво при первом запросе) |
| `POST /api/v1/spaces/home/rotate` | Bearer | сменить slug адреса; ответ — как у GET, с новым `url` |

Принципиальное правило: **клиент не хранит URL своего пространства** — «кнопка домой» каждый
раз спрашивает `GET /api/v1/spaces/home` и навигируется на ответ. Поэтому ротация адреса для
владельца незаметна (его «домой» всегда ведёт куда надо), а ломаются только розданные ссылки —
что и является целью ротации.

Вход в клиенте — кнопка **«Моё пространство»** (вкладка «Аккаунт», видна когда залогинен и
сервер анонсирует `personal-spaces.v1`); адрес резолвится тем же GET в момент нажатия,
литеральный URL нигде не оседает. Вход **не зависит** от домашней страницы (см. раздел
«Персональное пространство ≠ домашняя страница»): «Вернуться домой» открывает пространство
лишь фолбэком, когда `home_page` пуст. Всё остальное (редактирование, права, приватность) —
не вопрос связи клиент ↔ домашний сервер.

## Reference implementation персистенции

Страница пространства обязана быть образцовым страничным сервером по
[page-persistence.md](page-persistence.md):

- блок `<vrwml persist="https://example.com/api/v1/spaces/flush" rev="…">`;
- capability-запрос отвечает `policy: "allowlist"`, `writable` по `address`;
- владелец в allowlist всегда; выданные права — записи в том же allowlist;
- поскольку flusher'ы могут быть с **других** домашних серверов, проверка идентичности —
  федеративная (сертификат + proof), как в спецификации. Для локальных пользователей
  сервер может дополнительно принимать Bearer — оптимизация, не замена.

Так модуль закрывает исходную проблему спецификации персистенции «homeserver пока не хостит
страницы»: появляется первый сервер, на котором флаш можно реализовать и обкатать целиком.

## Приватность

### Угрозы

1. **Угадываемый адрес.** `example.com/home/<nickname>` вычислим по нику — любой может
   заходить когда угодно, в том числе когда владельца нет.
2. **Визиты в отсутствие владельца.** Даже неугадываемый, но однажды розданный адрес
   работает вечно: гость возвращается когда захочет.
3. **Комната без страницы.** Комната = `seed_key(url)`, сигналинг пускает в любую строку
   `room`. Даже если сам **фетч страницы** закрыть, знающий URL может войти в комнату и
   слушать голос/чат тех, кто внутри. Любая защита только на уровне страницы — дырявая.

### Механизмы (ортогональные, комбинируются)

- **Неугадываемый slug** — адрес вида `/s/<случайный токен>` вместо `/home/<nickname>`.
  Закрывает угрозу 1. Адрес становится capability: знаешь — можешь попробовать войти.
  Красивый `/home/<nick>` — опция для сознательно публичных пространств.
- **Ротация** (`POST …/rotate`) — ручной сброс capability: все розданные ссылки умирают.
  Владелец не страдает (см. правило «клиент не хранит URL»). Это механизм **отзыва утёкшей
  ссылки**, а не повседневная приватность: сбрасывать при каждом уходе неудобно (гости
  теряют адрес, которым пользовались легитимно) — и не нужно, см. следующий пункт.
- **Дверь: открыто, пока хозяин дома (presence-gate)** — элегантное решение угрозы 2,
  возможное именно потому, что домашний сервер и сигналинг — **один процесс**
  ([home-server.md](home-server.md)): хостер пространства сам видит, кто в его комнате.
  Пространство отвечает на фетч (и пускает в комнату) только пока внутри владелец; ушёл —
  сервер отдаёт `403 {"error": {"code": "space_closed"}}`, и новые гости не входят. Это
  автоматизация идеи «сбросить адрес, уходя», но без поломки ссылок: адрес постоянный,
  доступ временный. Гостям UX понятен из жизни: «хозяина нет дома».
- **Закрытый фетч (allowlist на вход)** — пространство отдаётся только аутентифицированным
  из списка. Для угрозы 3 требует той же проверки и на сигналинге. Самый строгий и самый
  дорогой режим: для федеративных гостей нужен аутентифицированный фетч страницы
  (подписанный GET по образцу proof из персистенции) — **точка расширения, не v1**.

### Политика пространства (выбирает владелец, веб-морда)

| Политика | Фетч страницы | Вход в комнату | По умолчанию? |
|---|---|---|---|
| `public` | всем | всем | нет |
| `when-home` | пока владелец внутри (или это сам владелец) | так же | **да** |
| `private` | только allowlist (v2: федеративный auth фетча) | так же | нет |

Рекомендация v1: **случайный slug + `when-home` по умолчанию**, ротация — кнопка «сменить
адрес» на веб-морде и в клиенте. Этого хватает для здравой приватности без новых
криптографических контрактов.

### Как владелец входит в закрытый дом (и почему это дёшево)

Хвост presence-gate: владельца нет внутри → пространство закрыто → но его собственный фетч
обязан пройти, иначе никто никогда не войдёт. Решение тривиально, потому что владелец —
**локальный пользователь этого же сервера**: клиент прикладывает свой Bearer-токен к
HTTP-запросам страниц своего домашнего сервера (`Authorization` при фетче, поле
`access_token` в `join` сигналинга — та самая «привязка сессий — дешёвая точка расширения»
из [home-server.md](home-server.md)). Сервер видит владельца — открывает. Никакой федерации:
свой сервер, свой токен.

То же покрывает локальных гостей из allowlist в режиме `private`. Федеративные гости в
`private` — за расширением (подписанный фетч); в `when-home` федеративным гостям ничего не
нужно — их пускает присутствие владельца.

### Присутствие: чему сервер верит

«Владелец внутри» сервер определяет по **своему** состоянию: WS-сессия сигналинга,
аутентифицированная токеном владельца, находится в комнате `seed_key(url пространства)`.
Не по самозаявленному нику и не по свидетельствам пиров. Соответствие URL → room key сервер
считает сам (та же детерминированная функция `seed_key`).

### Известные хвосты (принятые в v1)

- **Гости внутри после ухода владельца.** Меш — p2p: уже вошедшие продолжают общаться без
  сигналинга. «Дверь» закрывает только **новые** входы; выгнать оставшихся сервер не может
  (может лишь рвать свои WS, что не рушит открытые data-channel'ы). Приемлемо: контент
  страницы у них уже есть, а комната без владельца доживает и умирает сама.
- **Кэш страницы у гостя.** Побывавший в `when-home`-пространстве видел его HTML; закрытие
  двери не отбирает копию. Приватность здесь про **присутствие и свежесть**, не про тайну
  однажды показанного.
- **`space_closed` раскрывает существование пространства.** Отличим от честного 404 — по
  ответу видно, что за адресом что-то есть. Принято ради внятного UX («хозяина нет дома»
  вместо «страница не найдена»); параноидальный режим «закрыто = 404» — дешёвая опция позже.

Нюанс конфигурации: `base_url` в `homeserver.cfg` (или `VRWEB_BASE_URL`) — публичный
базовый URL, из которого собираются адреса `/s/<slug>` и persist-endpoint; по умолчанию
`https://{domain}`. Для локальной разработки — `http://127.0.0.1:8080`, иначе клиент не
попадёт по сгенерированным ссылкам, а ключ комнаты не совпадёт с `seed_key` клиента.

## Границы v1

Текущая реализация ограничена одним пространством `home`, действующей privacy policy и
описанными выше способами авторизации. Все расширения personal spaces ведутся в
[едином roadmap](../roadmap.md#p2--personal-spaces-и-persistence-vnext).
