# Presence — «где люди» (`presence.v1`)

> Фича домашнего сервера (Слой 3, [home-server.md](home-server.md)): сервер отдаёт сводку
> «на какой странице сколько людей». Reference implementation — модуль
> [../../homeserver/presence.py](../../homeserver/presence.py); клиент —
> `HomeServer.fetch_presence()` и раздел «Кто где сейчас» на вкладке «Аккаунт».

## Идея

Мир VRWeb — это обычные страницы, и их онлайн не известен снаружи: пока не зайдёшь по URL,
не узнаешь, есть ли там кто-то. Presence решает это на уровне домашнего сервера: сервер,
поддерживающий `presence.v1`, по запросу отдаёт список «URL → сколько людей», опционально —
теги страниц. Клиент показывает список как «витрину» сервера: видно, где собрались люди,
и можно перейти одним кликом.

## Контракт и его границы

Контракт — **только формат запроса/ответа**. Всё остальное — детали конкретной реализации,
и это осознанно:

- **Откуда данные — не предписано.** Сам домашний сервер не знает, кто на каких страницах:
  этим владеет сигналинг. Предполагается, что сервер как-то связывается с одним или
  несколькими сигналингами и агрегирует их данные — как именно, контракт не диктует.
  В нашем reference implementation домашний сервер и сигналинг — монолит
  ([home-server.md](home-server.md) → «разные сущности, один процесс»), поэтому источник —
  собственный `SignalingHub` (`room_counts()`).
- **Что попадает в выдачу — решает сервер.** Контракт не требует фильтрации, но
  добросовестный сервер обязан думать о приватности (см. ниже). Выдача даже **может быть
  разной для разных пользователей**: например, сервер может анонсировать конкретно вам
  комнату друга (друг в друзьях + не скрывается), которую анониму не покажет. Контракту это
  не противоречит — он описывает формат, а не наполнение.
- **Только количество, не список людей.** Контракт оперирует числом людей на странице.
  Списки участников намеренно не входят в v1: кто-то хочет быть «невидимым», и анонс имён —
  чужая приватность (см. «Модель доверия»).
- **Теги — свободные строки.** Откуда они берутся, контракт не говорит: сервер может
  размечать страницы сам, брать из конфига, из голосования пользователей — что угодно.
  В reference implementation теги проставляет админ сервера в конфиге.

## Протокол

Сервер анонсирует `presence.v1` в `features` discovery (`/.well-known/vrweb`) и отвечает на:

```
GET /api/v1/presence[?url=...][&limit=N][&offset=M]
```

```json
{
  "rooms": [
    {"url": "example.com/lobby", "count": 3, "tags": ["hub", "official"]},
    {"url": "other.org/gallery?page=2", "count": 1, "tags": []}
  ],
  "total": 2,
  "generated_at": 1780000000
}
```

Параметры запроса (все опциональны):

- **`url`** — точечный запрос: «сколько людей вот на этой странице?» Выдача — только по
  этой странице (сервер нормализует адрес тем же `seed_key`, клиент может слать URL «как
  есть»). 
- **`limit` / `offset`** — пагинация: не больше `limit` записей, пропустив первые `offset`
  (`limit` <= 0 или отсутствует — без ограничения). Страницы режутся по **живым** данным:
  между запросами выдача может сдвинуться (комнаты появляются и пустеют), дубли и пропуски
  на границах страниц — норма, а не ошибка. Стабильные курсоры — кандидат в `presence.v2`.

Поля ответа:

- **`url`** записи — канонический ключ страницы (= ключ комнаты, `seed_key` из
  [multiplayer.md](multiplayer.md): без схемы и фрагмента, хост в lowercase, query значим).
  Клиент открывает его как обычный адрес — резолвер подставляет `https://`.
- **`count`** — сколько людей на странице. **Приблизительное значение**: контракт не
  обещает точности — сервер вправе округлять или занижать (например, ради k-анонимности:
  точный `count: 1` на чьей-то странице деанонимизирует). Гарантируется только «люди там
  есть» (записи с нулём в выдачу не попадают).
- **`tags`** — свободные метки страницы (может быть пустым).
- **`total`** — размер выдачи **до** пагинации. `rooms.length < total` — признак, что
  клиент видит срез, а не всё.
- **`generated_at`** — unix-время формирования сводки.
- Сортировка reference implementation — по убыванию людности; клиенту на неё полагаться
  не обязательно.

**Выдача — не глобальная истина.** Presence отражает то, что видно **этому** серверу
(его сигналингу/сигналингам), в тот момент и для того, кто спросил. Она может быть неполной,
обрезанной и разной для разных пользователей — всё это в рамках контракта. Клиент общается
об этом с одним сервером — своим домашним; как серверы договариваются между собой о чьём-то
count — вопрос другого порядка (server-to-server, не этот контракт).

Авторизация: сервер сам решает, кому отвечать. Reference implementation настраивается
конфигом (`presence_access`): `public` — любому GET; `authenticated` — только пользователям
этого сервера (Bearer, иначе 401). Клиент шлёт Bearer всегда, когда залогинен на этом же
сервере, — публичному серверу он не мешает, а авторизуемому необходим (и открывает дорогу
персонализированной выдаче). Доступ **чужих** пользователей федерации (server-to-server
или гость с сертификатом другого домена) — отдельная будущая история, не этот механизм.

Выключение: `presence_enabled = false` убирает `presence.v1` из features, эндпоинт отвечает
404 (`presence_disabled`), веб-страница и ссылка на неё исчезают.

## Эволюция контракта

- **Неизвестное игнорируется.** Клиенту стоит игнорировать незнакомые поля в ответе
  и незнакомые значения; серверу — незнакомые query-параметры. Ни то, ни другое — не ошибка.
- **Расширение — только добавлением.** Новые возможности `presence.v1` появляются как новые
  опциональные поля/параметры; существующие поля не меняют тип и семантику. Пример
  будущего расширения: метаданные страницы (title/thumbnail) в записи.
- **Несовместимое —> в новую версию.** Радикальное изменение контракта должно привести к появлению `presence.v2` как отдельной feature-строки в discovery, живущей рядом с `presence.v1`
  (переходный период)

## Приватность: что фильтрует добросовестный сервер

Сигналинг видит **все** комнаты, включая приватные, — публиковать всё подряд не стоит.
Reference implementation выфильтровывает из выдачи:

1. **Приватные инстансы** — URL с query-параметром `vrweb-instance`
   ([multiplayer.md](multiplayer.md) → «Приватные инстансы»). Такая ссылка — **ключ** к
   комнате: она создаётся, чтобы отдать её лично друзьям, и анонсировать её в публичной
   выдаче = слить ключ. Прочие query-параметры не считаются приватными (это обычные
   страницы сайтов), хотя технически любой query даёт отдельную комнату.
2. **Персональные пространства** этого сервера (`/s/<slug>`,
   [personal-spaces.md](personal-spaces.md)): slug — секретный адрес, его знание =
   приглашение. Даже у пространства с политикой `public` анонс адреса — дело владельца,
   не сервера.

Фильтрация действует и для **точечного запроса** (`?url=`): вопрос о приватном инстансе
или пространстве получает пустую выдачу, даже если спрашивающий знает полный URL с ключом.
Иначе presence был бы инструментом слежки: зная ключ комнаты, можно было бы дистанционно
мониторить её занятость, не заходя внутрь (и не показываясь её участникам).

Ещё раз: это политика **нашей** реализации, а не требование контракта. Другой сервер может
фильтровать иначе (или не фильтровать — см. ниже, почему выбор сервера вообще вопрос
доверия к нему).

## Модель доверия

Любой сигналинг и любой домашний сервер **технически видят** присутствие клиента: комнаты и
есть их данные. Поэтому:

- Пользуясь сервером, клиент **уже доверяете ему** свои presence-данные. Presence-фича ничего
  нового серверу не открывает — она лишь формализует, что из этого он публикует.
- Недобросовестный сервер **может** слить присутствие клиента (в т.ч. в приватном инстансе,
  в т.ч. пометив «кто именно»), даже если пользователь просил этого не делать, — никакой
  протокол этому не помешает. Защита здесь — выбор сервера, которому доверяешь, и
  добросовестность реализации, а не криптография. Это то же самое допущение, что и во всей
  архитектуре ([security.md](../security.md), [ranks.md](ranks.md)): стек не навязывает
  защиту, пользователь выбирает, кому верить.
- «Невидимость» пользователя (не показывать меня в presence, даже в количестве) — будущее
  расширение на стороне сервера; в v1 её нет, и честный сервер должен говорить об этом
  прямо, а не обещать невозможное.

Теги в конфиге (`homeserver.cfg`):

```ini
[presence.tags]
example.com/lobby = hub, official
```

Ключ-URL нормализуется при выдаче (`seed_key`), так что схему/регистр хоста/хвостовой слеш
админ может писать как удобно.

Веб-версия `/presence` подчиняется тому же `presence_access`: при `authenticated` без логина
уводит на `/login`.

## Дорожная карта

- **Персонализированная выдача**: друзья (`friends.v1`) видят комнаты друг друга даже там,
  где аноним не видит ничего, — с согласия обеих сторон.
- **Невидимость**: пользовательский флаг «не учитывать меня в presence» (сервер честно
  вычитает из count).
- **Федеративный доступ**: сводка для чужих клиентов/серверов (server-to-server), когда
  федерация появится у остальных фич.
- **Стабильные курсоры пагинации** (`presence.v2`): замороженный снапшот на сервере вместо
  среза живых данных — когда появятся выдачи, для которых `limit`/`offset` уже честно
  не работает.
