# Эфемерные изменения сцены

> Транспортный слой поверх [авторитета](authority.md): любой пир **запрашивает** изменение
> сцены **действием** (action), авторитет — единственная точка коммита — валидирует и рассылает
> **события** (event). Контракт движок-агностичен: это протокол данных, а не Godot-специфика,
> чтобы его могла реализовать не-Godot сторона. Структура сериализуема — журнал событий можно
> будет выгрузить на сервер, чтобы тот обновил HTML-дерево (персистенция; пока не делается).

## Зачем

Сцена в VRWeb детерминирована по URL: открывшие одну страницу видят одинаковый мир
([multiplayer.md](multiplayer.md)). Эфемерные изменения добавляют в этот общий мир объекты,
которых нет в исходном HTML, **на лету**: временные порталы, метки, рисунки, расставленные
предметы. Первый инструмент — **[пузыри](#пузыри-первый-инструмент)**; дальше на тот же слой
лягут другие инструменты, **не меняя протокол**.

## Почему action/event, а не рассылка журнала

Наивный вариант — авторитет рассылает весь журнал на каждое изменение — плох:

- **Трафик** O(N объектов) на каждую мелкую правку.
- **Привязка к движку**: рассылать «состояние сцены» — значит зашивать форму состояния в
  транспорт. Нам нужен транспортный **контракт**, который реализует и не-Godot сторона.
- **Валидация при рассинхроне**: если инициатор шлёт «вот моё новое состояние», авторитету
  трудно понять, легально ли оно. Инициатор должен описывать **только нужную мутацию**.

Поэтому — модель **команда/событие** (command/event sourcing):

```
инициатор X        авторитет A (один)                 followers B,C
   | ACTION            |                                    |
   | {op,id,kind,…}    |  валидация против СВОЕГО состояния  |
   |----------------->|   + прав (владение/ранг)            |
   |                  |  мутация, штамп (epoch,seq)         |
   |                  |  EVENT ------------------------------>|  apply по порядку
   |                  |                                      |  (sender_is_authority)
```

- **Инициатор** шлёт авторитету **действие** — намерение, без претензий на состояние.
- **Авторитет** — единственная точка сериализации: валидирует против **своего** состояния и
  прав, присваивает порядковый `(epoch, seq)`, мутирует состояние и рассылает **событие**.
- **Followers** применяют события строго по порядку. Рассинхрон у инициатора не ломает
  валидацию — авторитет проверяет своё состояние, а не заявленное инициатором.

## Объект состояния

Основная часть состояния — `id → object`, где object — плоский JSON-сериализуемый словарь:

| Поле | Тип | Смысл |
|---|---|---|
| `id` | String | адрес объекта. Для `add` генерит **инициатор** (`<user8>.<seq>`) — чтобы потом адресовать СВОЙ объект |
| `kind` | String | тип объекта (`"bubble"`, будущие). По нему вьюха решает, что строить |
| `parent` | String | **куда**: `""` — корень мира \| `"<id>"` — другой объект \| `"page:<nodeId>"` — узел HTML-дерева страницы |
| `author` | String | `user_id` владельца — **штампует авторитет** из привязки `peer→user_id` (инициатор не задаёт) |
| `ts` | float | время создания по часам авторитета (отсчёт TTL) |
| `ttl` | float | секунд жизни; `>0` — эфемерный (истекает), `0` — постоянный (кандидат на выгрузку) |
| `props` | Dictionary | JSON-сериализуемая нагрузка под `kind` (позиции — как `[x,y,z]`, не `Vector3`) |

Формат `id` — `<user8>.<seq>` (первые 8 hex `user_id` инициатора + счётчик его объектов).
Разбор, свойства и почему префикс недоверенный — в отдельном документе
[object-id.md](object-id.md).

Рядом, но не внутри object map, живёт единая конфигурация инстанса:
`config = {attrs: {mode?}, by: user_id}`. Сейчас allowlist содержит только корневой атрибут
`<vrwml mode="combine|exclusive">`. У config нет `id`, `author`, TTL или object ownership;
это состояние комнаты, а `by` — только диагностика. Оно не входит в persistence object delta.

## Протокол

### Действие (инициатор → авторитет)

Плоское намерение. Поля по op:

```jsonc
{ "op": "add",      "id": "u1.3", "kind": "bubble", "parent": "",
  "ttl": 30, "props": { "url": "…", "position": [1,1.6,5], "label": "Вася" } }
{ "op": "update",   "id": "u1.3", "props": { "label": "новый" } }   // props — ПАТЧ (мердж; null удаляет ключ)
{ "op": "remove",   "id": "u1.3" }
{ "op": "reparent", "id": "u1.3", "parent": "u1.2" }
{ "op": "update-config", "set": { "mode": "exclusive" } }
```

Инициатор НЕ задаёт `author`/`ts`/`epoch`/`seq` — это привилегия авторитета.

### Событие (авторитет → все)

То же + порядковый штамп. Для `add` несёт полный объект; для `update` — **патч** (дельта);
для `remove`/`reparent` — минимум.

```jsonc
{ "epoch": 1, "seq": 7, "op": "add", "id": "u1.3", "kind": "bubble",
  "parent": "", "author": "a1b2…", "ts": 1719772800.0, "ttl": 30,
  "props": { … } }
{ "epoch": 1, "seq": 8, "op": "update-config",
  "set": { "mode": "exclusive" }, "by": "a1b2…" }
```

`update-config` принимает только allowlisted ключи и валидные значения; `null` снимает
override и возвращает базовый режим страницы. Некорректный config event не материализуется,
но занимает свой `seq`, чтобы один испорченный пакет не превращал дальнейший ordered stream
в постоянный GAP.

### Порядок: `(epoch, seq)`

- `seq` — монотонный в пределах эпохи; авторитет — единственный источник, reliable-ordered канал
  сохраняет порядок.
- `epoch` растёт при **смене авторитета** (`begin_authority()` ставит `epoch = max(виденное)+1`),
  чтобы события нового авторитета были заведомо новее экс-авторитетских. Запоздавший пакет
  старого авторитета (меньший epoch) отбрасывается сам.
- **Применение у follower** (`apply_event`): `epoch<наш` → дроп (устаревшее); `epoch>наш` →
  **GAP** (новая эпоха раньше снимка — ресинк); `seq<=наш` → дубликат; `seq>наш+1` → **GAP**
  (пропуск); `seq==наш+1` → применить. GAP → запросить снимок (один раз, флаг `_scene_resync`).

### Подтверждение действия (ack, опционально)

Отказ авторитета в базовом протоколе «тихий» (действие не породило событий). Для UI,
которому нужен явный исход (консоль пространства — [space-console.md](../client/space-console.md)),
транспорт расширен подтверждением:

- `request_scene_action_tracked(action) -> token` — то же действие, но с транспортной меткой
  `token` (не поле протокола: авторитет вынимает её до коммита, в состояние она не попадает);
- авторитет отвечает инициатору `ack {token, accepted}` **после** рассылки событий тем же
  reliable-каналом — к приходу ответа инициатор уже применил результат;
- у инициатора — сигнал `scene_action_acked(token, accepted)`. Доверие как у событий:
  ack принимается только от авторитета по собственному расчёту получателя.
- Таймаут ожидания — забота вызывающего (авторитет мог уйти, не ответив); если мы сами
  авторитет или авторитета нет (офлайн) — исход известен сразу, сигнал эмитится отложенно.

### Снимок (ресинк)

`{ epoch, seq, objects, config }` — полное состояние. Config snapshot повторно проходит
allowlist/schema-валидацию; невалидная часть сбрасывается. Снимок нужен в двух случаях:

- **push новичку**: авторитет шлёт снимок при открытии p2p-канала (`_on_mp_peer_connected`);
- **pull при смене авторитета / GAP**: пир запрашивает снимок (`_request_scene_snapshot`).

Свежий follower (epoch 0) до снимка **не угадывает** эпоху и на любое событие отвечает GAP →
тянет снимок. Это правильно: он мог пропустить сотни событий до входа.

## Права (владение + фундамент под систему прав)

Базовое правило: **править/удалять/перемещать можно только СВОИ объекты**. Проверка — в
`SceneChanges` (чистая, тестируемая), решение принимает авторитет:

- `update`/`remove`/`reparent` объекта `X` — только если `X.author == sender` **или** sender
  админ.
- `add` ребёнка в `parent`: корень и узлы страницы (`page:…`) открыты; вложение в **чужой**
  эфемерный объект — только владельцу того объекта или админу.
- `add` с занятым `id` — отказ (анти-хайджек: нельзя «занять» чужой адрес).
- защита от цикла при `reparent` (новый родитель не может быть потомком объекта).

**Кто такой админ** определяет транспорт: `NetworkManager` передаёт в `authority_commit`
`sender_is_admin = rank_of_peer(sender) <= EPHEMERAL_ADMIN_RANK` (ранги — [ranks.md](ranks.md);
0 ≈ админ/авторитет). `SceneChanges` про ранги не знает — получает готовый булев. Это **точка
расширения**: сюда ляжет полноценная система прав (capabilities, политика по `kind`, пороги
рангов), не меняя транспорт.

`author` авторитет берёт из своей привязки `peer_id → user_id` (карточка идентичности), инициатор
его не задаёт. Якорь доверия — тот же, что у рангов: самозаявленный `user_id`, который стек
**намеренно не защищает** от подделки. Это не «компромисс до верификации», а часть модели;
насколько доверять неподтверждённому автору — политика клиента, см. [ranks.md](ranks.md).

Корневая конфигурация имеет отдельную ACL: транспорт передаёт
`sender_can_config = rank_of_peer(sender) <= INSTANCE_CONFIG_RANK`, сейчас порог равен 0.
Первый участник/authority автоматически имеет rank 0 и может делегировать его другим. После
делегирования любой rank-0 пир может слать `update-config`; authority остаётся сериализатором,
но не единственным пользователем с правом. Ранги комнаты никак не дают право на flush.

## Лимиты (защита)

В `SceneChanges`: `MAX_OBJECTS` (потолок числа объектов), `MAX_TTL` (потолок TTL),
`MAX_PROPS_BYTES` (размер `props`, заодно гарантия сериализуемости через `JSON.stringify`).
Превышение — отказ в коммите.

## TTL

Истечение — обязанность **авторитета** (единый владелец состояния и часов): в `_process`
(throttle ~4 Гц) он зовёт `expire(now)`, та снимает объекты с `ts+ttl<now` **каскадно** с
потомками и возвращает события `remove`, которые рассылаются. Followers TTL не применяют — ждут
событий. Преемник продолжает истекать со следующего кадра (`has_authority()` проверяется каждый
кадр). `ts` — настенные часы авторитета; для визуального таймера у followers рассинхрон NTP
(<1 c) на фоне 30-секундного TTL пренебрежим, а каноничное удаление приходит событием.

## Вложенность и каскад

`parent` даёт дерево объектов. `EphemeralView` монтирует объект с `parent=<id>` как **ребёнка
узла родителя** (наследование трансформа); при `reparent` — перемонтирует. Удаление родителя
(явное или по TTL) **каскадно** снимает всех потомков: авторитет шлёт `remove` на каждого
(листья раньше родителя). У вьюхи Godot-узел родителя уносит детей с собой — поэтому последующие
`remove` детей идемпотентны (`is_instance_valid`).

Якорь `page:<nodeId>` (привязка к узлу HTML-дерева): для узлов **VRWML-слоя** вьюха резолвит
его в реальный построенный узел (реестр целей из `main`, id — детерминированные, см.
[space-console.md](../client/space-console.md)); неизвестный/HTML-узел пока падает в корень. Именно этот
якорь — мост к персистенции (эфемерный ребёнок узла страницы → реальный дочерний элемент в
HTML-дереве при выгрузке).

## API (`NetworkManager`)

| Член | Назначение |
|---|---|
| `request_scene_action(action)` | запросить изменение (если мы авторитет — коммит сразу) |
| `request_scene_action_tracked(action) -> token` | то же с обратной связью: исход придёт сигналом ниже |
| сигнал `scene_action_acked(token, accepted)` | ответ авторитета на отслеживаемое действие |
| `new_object_id() -> String` | сгенерить адрес нового объекта (для `op=add`) |
| `scene_objects() -> Dictionary` | снимок состояния (вьюха/отладка/выгрузка) |
| `scene_object(id) -> Dictionary` | один объект |
| `scene_config_attrs() -> Dictionary` | allowlisted root overrides текущего инстанса |
| `scene_config() -> Dictionary` | config вместе с диагностическим `by` |
| сигнал `scene_object_added(id, object)` | объект добавлен |
| сигнал `scene_object_updated(id, object)` | объект изменён/перемещён (полное состояние) |
| сигнал `scene_object_removed(id)` | объект снят |
| сигнал `scene_config_changed(config)` | изменилась корневая конфигурация инстанса |
| сигнал `scene_reset()` | состояние перезагружено снимком — пересобрать всё |
| `const EPHEMERAL_ADMIN_RANK` | порог ранга для админ-обхода владения |
| `const INSTANCE_CONFIG_RANK` | порог ранга для `update-config` (сейчас 0) |

Сигналы несут **плоские данные**; консьюмер материализует по `kind`, не зная транспорта.

## Пузыри (первый инструмент)

**Пузырь** — временный портал «пользователь ушёл сюда». При навигации на **другую** страницу
уходящий шлёт действие `add` `kind="bubble"` в **покидаемой** комнате: объект с TTL 30 c в точке
игрока, указывающий на URL назначения. Остальные видят пузырь и, кликнув, идут вслед.

- **Создаёт** [main](../../scenes/main.gd) в начале `_navigate` (через системный `BubbleTool`
  [системы инструментов](../client/tools.md)), непосредственно перед входом в нейтральный хаб.
  Это последний момент, когда игрок, старый мир и p2p-меш покидаемой комнаты ещё существуют.
  Цель — запрошенный URL, разрешённый относительно текущей базы; последующий HTTP-редирект
  пузырь уже не учитывает. Условия: онлайн, мы в комнате, целевая комната (`seed_key`)
  отличается от текущей.
- После отправки клиент сразу удаляет старый мир и разрывает сигнальное соединение. Поэтому
  пузырь — best effort: локальный коммит выполняется до выхода, но доставка удалённым пирам не
  задерживает переход. Приоритет здесь у строгой границы пространств: покинутая страница не
  должна жить и оставаться подключённой во время загрузки или security-preflight.
- **Внешний вид** — полупрозрачная сфера-орб с подписью и обратным отсчётом, затухает к концу TTL
  ([actors/bubble/](../../actors/bubble/)). Реализует тот же интерфейс луча, что и `Portal`
  (`interact_at`/`is_active_at`/`aim_hint_at`), плюс контракт `setup_object(object)`.
- **Не мешает передвижению**: пузырь висит на слое коллизий 2 (`collision_layer = 2`,
  `collision_mask = 0`) — его ловит только клик-луч игрока (маска 1+2), а тело игрока (маска 1)
  проходит сквозь. Тот же приём, что у `ImagePanel`/`RichPanel`.

`props` пузыря: `{ "url": String, "position": [x,y,z], "label": String }` — всё JSON-сериализуемо.

## Штрихи (второй инструмент)

**Штрих** — линия, нарисованная карандашом в пространстве. Один непрерывный штрих (нажатие →
ведение → отпускание ЛКМ) = **один** эфемерный объект `kind="vrweb-node"`, `parent=""`
с `props.tag="VRWebStroke"` (корень мира),
`ttl=0` (постоянный в пределах сессии — рисунок не должен таять; кандидат на выгрузку, см.
[персистенцию](#персистенция-точка-расширения-не-сейчас)). Инструмент и его слои — в
[pencil-tool.md](../client/pencil-tool.md); здесь — только контракт данных на эфемерном слое.

Канонический VRWML-тег и его attrs (точки — плоская последовательность ради компактности):
```html
<VRWebStroke points="[x0,y0,z0,x1,y1,z1]" color="Color(r,g,b,1)" width="0.02" />
```

В объекте слоя он хранится как:
```jsonc
{ "tag": "VRWebStroke",
  "attrs": {
    "points": "[x0,y0,z0,x1,y1,z1,…]", // мировые координаты
    "color": "Color(r,g,b,1)",          // 0..1
    "width": "0.02"                   // метры
  } }
```

- **Финализация при отпускании.** Во время ведения штрих рисуется локальным превью из CSG-точек;
  в слой уходит **один** `add` только при отпускании ЛКМ. Никаких `update` по ходу — минимум
  трафика, ровно один объект. Карандаш прореживает путь шагом 3 см и ограничивает его 200 точками;
  общий `MAX_PROPS_BYTES` остаётся последней границей.
- **Ластик** = `remove` своего штриха целиком: при пересечении (тест точка↔полилиния с учётом
  толщины) шлётся `remove`. По текущим правам править можно только **свои** объекты — чужие штрихи
  ластик отсеивает заранее (и авторитет всё равно отклонил бы). Граффити-режим (стирать любые) —
  точка расширения политики прав по тегу внутри `vrweb-node`.
- **Офлайн** (вне комнаты): слой коммитит `add` в локальную standalone-машину, поэтому
  рисование/стирание работает без сети; при входе в комнату локальное состояние заменяется её
  авторитетным снимком.
- **Протокол не меняется**: штрих использует готовый `vrweb-node` и общий путь
  `EphemeralView → VrwebBuilder`; отдельной ветки `kind` в клиенте нет.

## HTML-представление слоя (консоль пространства)

Состояние слоя имеет каноническое HTML-представление, которое сериализует/разбирает чистый
модуль [scripts/ephemeral/scene_html.gd](../../scripts/ephemeral/scene_html.gd): дифф правки
пересчитывается обратно в действия протокола (`add`/`update`/`remove`/`reparent`) — так
работает интерактивный редактор пространства по `~`. Консоль показывает **слитый** блок
`<vrwml>` (узлы страницы + оверлей), пользователь правит сцену как единую сущность, наружу
уходит только дельта. Подробно — [space-console.md](../client/space-console.md).

## Оверлей VRWML-слоя (редактирование страницы через дельты)

Узлы VRWML-слоя страницы не лежат в эфемерном состоянии — они детерминированно строятся из
HTML. Их правка выражается **дельтами** на том же протоколе (новые kind, протокол не менялся):

- `kind="vrweb-patch"` — правка/удаление узла страницы: props
  `{ "set": {атрибут: raw-строка}, "removed": bool }`. id **детерминированный**
  `vpatch:<id узла>` — анти-хайджек `add` даром даёт «узел правит один», владение — что
  чужой патч не переписать. Материализация: применение к живому узлу с запоминанием
  оригиналов (снятие патча откатывает страницу как была).
- `kind="vrweb-node"` — добавленный узел сцены: props `{ "tag", "attrs" }` (сырые строки,
  как в HTML). Строится тем же VrwebBuilder; `parent="page:<id узла>"` теперь монтируется
  под **реальный** узел страницы (реестр целей у EphemeralView).

Эти объекты (`ttl=0`) — и есть накопленная дельта для будущего «флаша» в страницу
(см. «Персистенция» и [space-console.md](../client/space-console.md)).

## Переносимые предметы (kind="vrweb-item")

Объект с `props: { "src": "<url item-документа>" }` — **переносимый предмет** (item):
каждый клиент сам скачивает документ (VRWML-фрагмент + Luau + ассеты), строит сцену тем же
конвейером, что страницы, и запускает скрипты предмета в собственном realm со стандартным
capability pool ([VrwebItemHost](../../scripts/vrweb_item_host.gd), обоснование —
[portable-tools.md](../space/portable-tools.md)). Удаление объекта (remove/TTL/каскад)
снимает и сцену, и realm. `src` иммутабелен: смена предмета = remove + add. Скриптовые и
grabbable-адреса внутри item автоматически получают namespace по id объекта — два экземпляра
одного предмета не сталкиваются. Протокол слоя снова не менялся.

## Доступ скриптов к слою

Page/item скрипты получают add/update/remove через capability `vrweb/scene-objects/1`
(`document.scene`, см. [scripting-api.md](../space/scripting-api.md#эфемерный-слой-сцены-vrwebscene-objects1)).
Действия идут этим же протоколом от имени локального пользователя — авторитет и права слоя
не отличают их от действий инструментов клиента.

**Вне комнаты слой — полноценная локальная машина.** Пока комнаты нет, авторитета не
существует вовсе и участник ровно один, поэтому действия коммитятся локально с правами
админа (RPC не шлётся). Это шире, чем offline mode: иначе у пользователя с включённым
онлайном, но не в комнате (локальная или одиночная страница), действия молча терялись бы —
именно так пропадали штрихи и размещённые картинки. Локальные объекты сессионные: при входе
в комнату состояние сбрасывается и заменяется снимком её авторитета.

## Изображения (третий инструмент)

**Инструмент размещения изображений** (клавиша 3): сначала режим прицеливания (луч ≤ 1.8 м с
полупрозрачным превью-квадратом, разворот по нормали поверхности или лицом к игроку; ЛКМ —
подтвердить, ПКМ/3 — отмена), затем выбор картинки. Файл импортируется в **realtime-ресурс**
(компактный блоб с адресом по хэшу содержимого, [realtime-resources.md](realtime-resources.md)),
а в слой уходит обычный `add` `kind="vrweb-node"` с тегом `<VRWebImage>` и КОРОТКОЙ ссылкой
в атрибутах:

```jsonc
{ "op": "add", "id": "u1.7", "kind": "vrweb-node", "parent": "", "ttl": 0,
  "props": { "tag": "VRWebImage",
             "attrs": { "src": "vrwebblob://sha256/…", "alt": "кот.png",
                        "transform": "Transform3D(…)" } } }
```

Протокол слоя снова не менялся: базы64 в `props` нет (лимит `MAX_PROPS_BYTES` не страдает),
байты картинки пиры докачивают отдельным p2p-протоколом по требованию материализации, а
поскольку это `vrweb-node` — консоль пространства, права владения и персистенция работают
готовыми путями. Офлайн — локальный узел напрямую, как офлайн-штрихи.

## Персистенция (точка расширения, не сейчас)

Журнал событий — естественный лог для выгрузки на сервер: `add`/`update`/`remove`/`reparent`
ложатся прямо на мутации HTML-дерева (вставить узел под parent, поправить атрибуты, удалить).
`ttl=0` — постоянные объекты, кандидаты на флаш; `ttl>0` (пузыри) обычно не персистятся. Поля
(`author`, `ts`, `parent`, `props`) уже несут всё нужное для реплея.

Протокол выгрузки специфицирован (и реализован) в [page-persistence.md](page-persistence.md):
discovery поддержки сервером, федеративная авторизация flusher'а, per-object ответ
(applied/rejected/deferred) и дедупликация «запечённых» объектов против новой базы. Отсюда
два следа в этом слое: `SceneChanges.reserved_ids` (id узлов страницы; `add` с таким id
отклоняется — анти-коллизия дедупа, заполняет `main` из индекса) и пометка сохранённых
объектов обычным `update` `props: { "persisted_rev": … }` — протокол слоя не менялся.

## Границы

- **Split-brain**: при разрыве меша на компоненты у каждой свой авторитет; каждый поднимет свою
  эпоху. При слиянии `min`-правило выберет одного, остальные ресинкнутся снимком (эпохи могут
  столкнуться — для эфемерных объектов приемлемо). См. [authority.md](authority.md).
- **Новые инструменты**: новый `kind` + ветка в `EphemeralView._make_node` + (опционально) политика
  прав. Протокол (`action`/`event`/`snapshot`) не меняется. Клиентская половина рецепта (класс
  инструмента, слоты, роутинг ввода) — в [tools.md](../client/tools.md).
- **Оптимистичное применение** у инициатора пока не делается — объект появляется по событию от
  авторитета (одна RTT). Для пузыря это неважно (инициатор уходит). При необходимости добавляется
  поверх без смены протокола.
