# Адаптивные представления одного VRWeb-инстанса

> Статус: архитектурное предложение, не реализовано. Имя HTTP header и словарь его значений ещё
> не стандартизированы; в примерах используется рабочее имя `VRWeb-Capability-Code`.

## Задача

Разные клиенты VRWeb не обязаны поддерживать одинаковый набор движковых классов, shader formats,
исполняемого кода, графических возможностей или способов ввода. Как браузер может не реализовать
часть Web API, отключить её локальной policy или отобразить документ иначе, VRWeb-клиент вправе:

- пропустить неизвестный или запрещённый класс/свойство;
- заменить элемент своей совместимой реализацией;
- отключить конкретный shader format, scripting capability или иные возможности;
- ввести собственные ограничения исполнения, включая циклы и runtime budgets;
- представить один семантический объект более простой геометрией, материалом или UI;
- вообще не строить визуальный слой, оставаясь voice/text участником комнаты.

Словарь стандарта и реально включённый capability pool клиента — разные сущности. Желательно,
чтобы клиент сообщал unsupported capability и diagnostics, но стандарт не требует от каждой
реализации одинакового набора возможностей или пиксельно одинакового результата.

Одной клиентской адаптации недостаточно во всех случаях. Автор мира может заранее знать, что для
мобильного, VR, text-only или иного класса клиента лучше отдать другой документ: иные assets,
геометрию, UI, скрипты либо совсем другое локальное представление. При этом все пользователи
должны остаться в одной WebRTC-комнате.

## Две стороны адаптации

### Клиент адаптирует универсальную декларацию

VRWML декларативен: клиент может сопоставить стандартный тег своей реализации, упростить material,
выбрать LOD, проигнорировать неподдерживаемое свойство либо сохранить только семантически важную
часть. Это основной путь, когда один документ достаточно хорошо описывает все варианты.

Преимущество — автор поддерживает один документ. Недостаток — клиент не всегда может вывести
хорошее упрощённое представление из данных, созданных для совершенно другой платформы.

### Сервер отдаёт разные документы по request header

Клиент запрашивает тот же top-level URL, добавляя простой строковый hint:

```http
GET /world HTTP/1.1
Host: example.com
VRWeb-Capability-Code: android
```

Другой клиент может отправить другое значение или не отправлять header:

```http
VRWeb-Capability-Code: some-future-profile
```

HTTP-сервер видит hint и при желании возвращает другой HTML/VRWML document. URL не меняется,
поэтому существующий `PageFetcher.seed_key(url)` без каких-либо исключений приводит всех клиентов
в одну WebRTC-комнату. History, bookmarks, presence, приватные instance query, persistence и
relative URL также сохраняют текущую семантику.

`VRWeb-Capability-Code` — representation hint, а не security capability, доказательство платформы
или право доступа. Значение полностью контролируется клиентом/пользователем и легко подделывается.
Сервер не должен принимать решения авторизации на его основе.

## Почему header лучше query-параметра

Query-вариант потребовал бы искусственно разделить полный document URL и WebRTC room key, удалить
из room key один специальный параметр и одинаково реализовать новую URL canonicalization во всех
клиентах. Он также сделал бы platform-вариант видимым в history, bookmarks, ссылках, presence и
persistence.

Request header оставляет две независимые оси на естественных местах:

```text
URL                         = идентичность документа и realtime-инстанса
VRWeb-Capability-Code       = HTTP negotiation локального представления
```

Это соответствует обычной browser-модели content negotiation: одна ссылка открывается на разных
клиентах, каждый сообщает свой hint, а сервер может вернуть подходящее представление. WebRTC и
остальные слои вообще не знают, что HTTP responses различались.

## HTTP-кэширование

Если response действительно зависит от hint, сервер обязан объявить это кэшу:

```http
Vary: VRWeb-Capability-Code
```

Если `Vary` забыть, shared cache или CDN может сохранить ответ одного профиля и вернуть его другому
клиенту. Это главный инфраструктурный нюанс header-подхода. Сервер должен добавить поле к уже
существующему `Vary`, а не затереть значения вроде `Accept-Encoding`.

При отсутствии header сервер возвращает baseline document для обычного браузера или клиента без
этого механизма. Незнакомый code также можно проигнорировать и обслужить baseline response.

Hint предлагается отправлять только при загрузке top-level world document. Assets получают обычные
URL, выбранные этим документом; это не дробит кэш каждого изображения или скрипта. Если практике
понадобится negotiation отдельных ресурсов, она должна быть спроектирована отдельно.

Поведение на redirect chain ещё нужно зафиксировать. Рабочее ожидание — клиент повторяет header
для top-level document request после redirect, а серверы каждого hop корректно используют `Vary`.
Это не влияет на room identity: как и сейчас, комната определяется final URL.

## Что обязан согласовать автор мира

Одна URL/WebRTC-комната не делает разные responses автоматически семантически совместимыми. Автор
вариантов отвечает за общий network ABI и пространственное соответствие там, где оно требуется:

- единая система координат, масштаб и смысл зон, в которых синхронизируются avatars/positions;
- стабильные ids общих объектов;
- одинаковые schema ids, versions и типы полей replicated state;
- совместимые semantics команд и authority rules;
- стабильные script ids и remote endpoint names/versions для взаимодействующих вариантов;
- корректное поведение при отсутствии визуального объекта или endpoint в упрощённом варианте;
- согласованные spawn/teleport anchors и collision semantics;
- fallback для realtime-объектов, чей визуальный класс вариант не отображает.
- согласованный persistence mapping, если разные responses редактируют и сохраняют один document
  URL.

Варианты не обязаны иметь одинаковую геометрию. Например, сложный концертный зал может стать на
мобильном клиенте несколькими простыми зонами или плоской картой. Но если оба клиента трактуют
одну сетевую позицию как принципиально разные места, spatial voice и взаимодействия станут
непонятны. Это authoring-задача, не то, что transport может исправить автоматически.

Полезный принцип: синхронизировать семантическое состояние, а производное представление строить
локально. Не следует синхронизировать engine resources или предполагать наличие конкретного
rendered node у каждого участника.

## Поведение клиента и сервера

Клиент:

- выбирает собственный code или не отправляет header;
- может позволить пользователю/dev mode переопределить code;
- не считает code полномочием или подтверждённым свойством устройства;
- при неизвестном/неподдержанном ответе применяет обычную частичную материализацию;
- использует неизменный final URL для входа в WebRTC, history и document lifecycle.

Сервер:

- может игнорировать незнакомый code и вернуть baseline document;
- может вернуть любое представление, не обещая, что code является точной моделью устройства;
- отправляет `Vary: VRWeb-Capability-Code`, если response зависит от header;
- не использует code для authorization;
- по возможности сохраняет общий network ABI между вариантами.

Страница также может вообще не иметь variant responses. Тогда разные клиенты адаптируют одну
декларацию локально — механизм является opt-in автора, а не обязательным branching каждого сайта.

## Почему пока нет словаря codes

На этом этапе значение header — opaque string. Стандарт не фиксирует `android`, `mobile`, `low`,
названия клиентов или renderer tiers и не утверждает, какие измерения нужно кодировать вместе.
Практика должна показать, нужны ли авторам преимущественно platform families, input modes,
renderer profiles, accessibility representations или составные варианты.

Ранний registry почти наверняка закрепил бы неверные категории. Сначала реализации и миры могут
использовать собственные договорённости; позже повторяющиеся значения и требования станут основой
versioned registry или более структурированного negotiation mechanism. Неизвестное значение всегда
остаётся корректным hint, который сервер вправе проигнорировать.

## Связь с graceful degradation

Адаптивные документы добавляют ещё одну ось к уже принятой модели:

```text
одна URL и realtime-встреча
  ├─ один document, разные клиентские materializers
  ├─ разные server-provided responses по request header
  ├─ voice-only представление без полного document
  └─ text/headless представление
```

Это не fork мира: общей остаётся встреча и её сетевые контракты, различается локальное
представление. Механизм продолжает принцип VRWeb «один стек — много реализаций», не заставляя
самый слабый и самый богатый клиент строить одинаковую сцену.
