# Авторитет комнаты

> Для некоторых действий в инстансе (комнате = странице) нужно, чтобы их выполнял **ровно
> один** пир — «авторитет». Авторитет даётся тому, кто **раньше всех вошёл в комнату**, а при
> его уходе автоматически переходит к следующему старейшему. Всё это в p2p WebRTC mesh — без
> сервера-арбитра и без переговоров между пирами.
>
> Базовая модель сети — в [multiplayer.md](multiplayer.md). Это её развитие.

## Главная идея: авторитет не выбирают, его вычисляют

Сигнальный сервер на **каждый вход в комнату** штампует монотонный `seq`
([signaling/server.py](../../signaling/server.py), `_join_seqs = count(1)`; приходит в ответе
`peers` — свой, и в `peers`/`peer_join` — по каждому пиру). Поэтому:

- **Наименьший `seq` среди членов комнаты = тот, кто вошёл в НЕЁ раньше всех.** Это и есть
  «авторитет первому».
- **Авторитет — чистая функция от состава связанной p2p-компоненты:**
  `authority = min по seq(мы + все пиры с открытым p2p data-channel)`. В полном меше каждый
  пир знает весь состав (он соединён со всеми), поэтому **каждый считает авторитета сам,
  локально, и приходит к одному и тому же ответу**. Пир, который виден только через сигналинг,
  ещё не участвует в авторитете: по нему нельзя принять RPC-действие.

**Почему `seq` входа, а не `peer_id`.** `peer_id` выдаётся при **подключении к серверу**
(до входа в какую-либо комнату) и живёт, пока жив WebSocket. Старшинство по id означало бы
«кто раньше запустил приложение», а не «кто раньше вошёл в комнату»: давно подключённый
клиент имел бы меньший id и перехватывал бы авторитет в **любой** комнате, куда бы ни зашёл
позже остальных, — а его пустой снимок затирал бы накопленное состояние (реальный баг,
июль 2026). `peer_id` остаётся **адресом** (меш/RPC/антиглар «offer шлёт меньший id») и
fallback-правилом старшинства, если сервер не прислал `seq` (старый сигналинг).

Отсюда — нет «протокола передачи авторитета», нет голосований и нет состояния, о котором
пирам надо «договариваться». Договариваться не о чем: при одинаковом составе результат
`min` одинаков у всех.

## Передача — только при уходе, и автоматически

Монотонные seq дают важное свойство: **новичок всегда получает seq больше**, чем у всех
присутствующих. Значит:

- Новый пользователь **не может перехватить** авторитет — он видит, что есть кто-то с меньшим
  seq, и считает авторитетом его. Роль «липнет» к старожилу.
- Авторитет сдвигается **только** когда текущий авторитет уходит или p2p-канал к нему
  закрывается: `peer_leave`/`p2p_peer_disconnected` убирает его из связанной компоненты →
  все пересчитывают `min` → авторитетом становится следующий по старшинству. Без выборов,
  без окна «бесхозности» с точки зрения правила.
- **Ушёл из комнаты — потерял старшинство**: повторный вход (в т.ч. навигация туда-обратно
  или реконнект после обрыва) даёт свежий, больший seq.

```
комната: seq={3,7,12}       авторитет = seq 3
  ← входит seq=20           авторитет = seq 3  (новичок НЕ перехватывает)
  ← уходит seq=3            авторитет = seq 7  (передалось само, всем сразу)
```

## Доверие: получатель проверяет сам, а не верит на слово

Новому участнику **не нужно верить** чьему-либо заявлению «я авторитет». Авторитет не
*заявляется*, а *проверяется получателем привилегированного действия*:

> Действие принимается, только если `get_remote_sender_id() == authority_id()` по
> **собственному** расчёту получателя (`NetworkManager.sender_is_authority()`).

- Никто не может «сказать, что он авторитет», и быть принятым — каждый вычисляет авторитета
  сам.
- `id` отправителя берётся из меша (`multiplayer.get_remote_sender_id()`) — он привязан
  сервером/мешем, прикладной слой его не задаёт, поэтому модифицированный клиент не может
  выдать себя за чужой/меньший id (ср. с тем, как чат режется и на приёме — клиенту в
  принципе не доверяют). **Якорь доверия — тот же сигнальный сервер**, которому уже доверяют
  состав комнаты; ничего нового доверять не нужно.
- Запоздавшие пакеты от **бывшего** авторитета во время передачи роли отбрасываются сами: к
  моменту их прихода `authority_id()` у получателя уже указывает на нового авторитета, и
  `sender != authority`. Поэтому отдельная «эпоха/term» для корректности выбора *кто* не
  нужна.

## Состояние при передаче

Авторитет — это **роль**, а не данные. Чтобы преемник не начинал с нуля:

- **Эфемерное состояние** (пример — таймкод видео): новый авторитет просто продолжает слать
  heartbeat. Наследовать нечего. Так уже работает синхронизация видео.
- **Накопленное состояние**: авторитет периодически вещает снимок, остальные держат «тёплую»
  копию → преемник продолжает с последнего, что слышал. Поздно зашедший получает снимок либо
  следующим heartbeat, либо запросив его у авторитета при подключении (pull в
  `multiplayer.peer_connected`). Реализованные примеры такого состояния — **таблица рангов**
  (кто на что имеет право): см. [ranks.md](ranks.md); и **журнал эфемерных изменений сцены**
  (объекты, добавленные в общий мир на лету — пузыри и будущие инструменты): см.
  [ephemeral-changes.md](ephemeral-changes.md). Оба наследуются преемником тем же приёмом
  (push новичку + pull при смене авторитета).

## API (`NetworkManager`)

| Член | Назначение |
|---|---|
| `authority_id() -> int` | id текущего авторитета (`min` по нам + p2p-пирам); `0` вне комнаты |
| `has_authority() -> bool` | авторитет ли мы (true ровно у одного в связной компоненте меша) |
| `sender_is_authority() -> bool` | для **приёмника** @rpc: пришло ли действие от авторитета |
| `is_timekeeper() -> bool` | алиас `has_authority()` — частный случай для синхронизации видео |
| сигнал `authority_changed(new_authority: int, is_me: bool)` | состав изменил `min`; консьюмеры начинают/прекращают привилегированные действия |

`authority_changed` эмитится при подъёме/сносе меша и при открытии/закрытии p2p-каналов —
но только когда реально меняется результат `min` (`_refresh_authority`).

### Как добавить новое действие, требующее авторитета

1. **Отправка:** выполняй/рассылай действие только если `NetworkManager.has_authority()`
   (или по сигналу `authority_changed` начинай это делать, когда `is_me`).
2. **Приём:** в теле `@rpc`-метода в начале проверь `NetworkManager.sender_is_authority()` —
   иначе игнорируй пакет.
3. **Состояние:** если действие копит состояние, периодически вещай снимок (heartbeat),
   чтобы преемник и новички подхватывали его без явной передачи.

Пример «таймкипера» видео — образец паттерна: один пир (`is_timekeeper()`) шлёт heartbeat,
при его уходе роль уходит следующему сама, поздно зашедшие синхронизируются следующим
пакетом. Подробно — в [video-player.md](../client/video-player.md).

## Границы и риски

- **Split-brain** возможен только при реальном разрыве сети меша на компоненты: в каждой
  компоненте появится свой `min`/авторитет. Внутри связной компоненты все согласованы. При
  слиянии компонент `min` пересчитается; для действий, копящих состояние и чувствительных к
  слиянию, добавляй на прикладном уровне монотонный счётчик (term) и правило разрешения
  конфликтов — но для большинства эфемерных действий это избыточно.
- **Легитимный авторитет может злоупотреблять** своими полномочиями — это неизбежное
  следствие модели «первому». Ограничивается только тем, какие полномочия вообще даны
  авторитету. Доверие к серверу как к источнику состава/id уже принято в текущей модели.
- **Идемпотентность подключения всё ещё важна.** `NetworkManager.connect_to_server()` не рвёт
  уже открытый или ещё подключающийся сокет: переоткрытие в окне рукопожатия выдало бы новый
  `peer_id` и порвало бы p2p-адресацию. Единственное исключение — сменился сам эффективный
  адрес сигналинга (пользователь переопределил, домашний сервер анонсировал другой, см.
  [home-server.md](home-server.md)): старый сокет ведёт не на тот сервер, поэтому рвём и
  подключаемся заново. Для старшинства авторитета сама нестабильность id
  больше не фатальна (старшинство считается по `seq` входа в комнату), но реконнект по-прежнему
  означает выход из комнаты → потерю старшинства → свежий `seq` при возврате — это ожидаемое
  поведение, а не баг.
- **Федерация** (см. [multiplayer.md](multiplayer.md)): `seq` — счётчик ОДНОГО сервера; если
  комнату начнут собирать несколько серверов, seq разных серверов несравнимы. Тогда авторитета
  выбирать по (timestamp присоединения, tie-break) или держать порядок входа на уровне
  межсерверного протокола.
