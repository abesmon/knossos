# Персистенция страницы (Слой 4): флаш эфемерного слоя

> Протокол, по которому клиент сохраняет накопленную дельту [эфемерного слоя](ephemeral-changes.md)
> **в саму страницу** на её сервере. Это «Слой 4» из [README.md](../README.md):
> клиент шлёт дельту → сервер валидирует и применяет к своей копии страницы → страница
> переиздаётся → следующая загрузка видит изменения уже в базе. Протокол — открытый контракт:
> любой сервер может реализовать его самостоятельно, клиент узнаёт о поддержке через discovery.
>
> Статус: **реализовано** (v1). Сервер — [персональные пространства](personal-spaces.md)
> homeserver'а ([../homeserver/spaces_api.py](../../homeserver/spaces_api.py) +
> [../homeserver/vrweb_scene.py](../../homeserver/vrweb_scene.py) — порт индексации/слияния);
> клиент — кнопка «В страницу» консоли пространства
> ([../scripts/space_console.gd](../../scripts/space_console.gd)).

## Роли и границы доверия

| Сущность | Роль в протоколе | Кому доверяет |
|---|---|---|
| **Страничный сервер** (origin страницы, напр. `example.org`) | принимает дельту, сам применяет её к странице, переиздаёт | подписи домашних серверов (через их `/.well-known/vrweb`), своей политике прав |
| **Клиент-инициатор** («flusher») | собирает дельту из эфемерного слоя, подписывает запрос своим ключом идентичности | своей копии состояния слоя |
| **Домашний сервер** пользователя | в запросе не участвует; его подпись на сертификате — якорь идентичности | — |
| **Комната** (p2p-меш) | о флаше не знает; узнаёт постфактум через пометку объектов (см. «Дедупликация») | авторитету, как обычно |

Ключевые принципы (продолжают решения [space-console.md](../client/space-console.md)):

- **Server-authoritative merge.** Клиент шлёт **объекты дельты**, а не слитый HTML. Сервер
  сам применяет их к своей копии страницы — присланному «готовому» документу он не доверяет
  (иначе любой писатель переписывал бы страницу целиком, а не свою дельту).
- **Идентичность — федеративная**, та же, что в комнатах ([home-server.md](home-server.md)):
  сертификат `nick@domain` + доказательство владения ключом. Страничному серверу не нужна
  своя база пользователей — он проверяет подпись через домен из адреса.
- **«Принять ≠ применить».** Ответ сервера различает «применил», «отклонил» и «принял на
  рассмотрение» (модерация/премодерация — внутренняя логика сервера, протокол её не диктует).

## Персистенция — свойство СТРАНИЦЫ, не домашнего сервера

Ключевая рамка: **флашат конкретную страницу на её origin**, а не «домашний сервер». Поэтому
persistence — это **капабилити страницы (или сайта)**, и обнаруживается она **по самой
странице**, а не через discovery домашнего сервера. `persistence` НЕ входит в `features`
домашнего сервера ([home-server.md](home-server.md) → «Что вообще имеет право быть в
`features`»): фичи домашнего сервера — про связку клиент↔домашний-сервер (идентичность,
сигналинг, персональные пространства), а флаш адресуется origin'у страницы. Наши персональные
пространства выдают страницы с `persist`, но это свойство **тех страниц**, не анонс сервера.

## Discovery: умеет ли страница принимать дельты

Три уровня, от точного к общему; клиенту достаточно любого:

1. **Атрибут на блоке `<vrwml>`** (основной, per-page):

   ```html
   <vrwml persist="https://example.org/vrweb/flush" rev="a41f...">
   ```

   Страница сама говорит, куда флашить, и несёт идентификатор своей версии (`rev`).
   Нет атрибута — страница read-only. **Это единственный механизм, который использует наша
   реализация** (страницы пространств несут `persist`); уровни 2–3 — для сторонних серверов.

2. **HTTP-заголовок ответа страницы** (когда `<vrwml>` генерится и трогать его неудобно):

   ```
   VRWeb-Persist: https://example.org/vrweb/flush
   ```

3. **`/.well-known/vrweb` origin'а страницы, feature `site-persistence.v1`** — общесайтовый
   дефолтный endpoint в `config.persist_url`, чтобы каждой странице сайта не носить атрибут.
   Это капабилити **страничного origin'а** (роль «сервер, хостящий страницы»), а НЕ домашнего
   сервера: формат well-known тот же, но роль другая. Один хост может играть обе роли (наш
   монолит — и домашний сервер, и хост пространств), но фичи ролей не смешиваются: `identity.v1`/
   `signaling.v1`/`personal-spaces.v1` — про домашний сервер, `site-persistence.v1` — про
   origin страниц. Наша реализация уровень 3 не использует (обходится per-page атрибутом).

Приоритет: атрибут > заголовок > well-known. Endpoint — абсолютный URL **того же origin**,
что и страница (клиент отклоняет чужой origin — иначе страница-злоумышленник направляла бы
подписанные дельты на чужой сервер).

**Идентификатор версии базы (`rev`) обязателен** для сервера, поддерживающего персистенцию:
атрибут `rev` на `<vrwml>` или `ETag` ответа страницы (атрибут приоритетнее). Клиент
запоминает его при фетче — это `base_rev` будущего флаша и опора дедупликации.

## Кого сервер готов слушать: capability-запрос

`GET <endpoint>?url=<канонический URL страницы>[&address=<nick@domain>]` — без авторизации:

```jsonc
{
  "v": 1,
  "rev": "a41f...",                     // текущая версия базы
  "identity_required": true,            // false — принимает и анонимные дельты
  "policy": "allowlist",                // "open" | "verified" | "allowlist" | "owner"
  "writable": true,                     // ответ для ?address=...; без address опускается
  "accepts_kinds": ["vrweb-patch", "vrweb-node"],
  "max_objects": 256, "max_bytes": 262144
}
```

- `policy` — грубая подсказка: `open` (кто угодно), `verified` (любой с подтверждённым
  адресом), `allowlist`/`owner` (список/владелец — сам список не публикуется).
- `writable` — **необязывающая** подсказка для UI (показывать ли кнопку «сохранить в
  страницу»): ответ по `address` не аутентифицирован, настоящее решение сервер принимает
  только на POST. Врать здесь серверу незачем, но полагаться на это нельзя.
- `accepts_kinds` — какие kind'ы оверлея сервер умеет вливать. v1-минимум — `vrweb-patch` и
  `vrweb-node`; мировые объекты (`stroke`, `bubble`) в v1 не флашатся (у них нет
  HTML-представления в базе).

## Запрос флаша

`POST <endpoint>`, `Content-Type: application/json`. POST, а не PATCH: семантически это
и есть patch, но endpoint — отдельный ресурс, а не URL документа; версию несёт поле `v`.

```jsonc
{
  "v": 1,
  // ПОДПИСЫВАЕМАЯ строка (уходит вербатим — серверу не нужно воспроизводить чужую
  // сериализацию; тот же приём, что certificate_json). Внутри — JSON:
  //   { "v": 1, "url": "…", "base_rev": "a41f…", "ts": 1780000000,
  //     "nonce": "…base64 16+ байт…", "objects": [ … ] }
  "payload": "{\"v\":1,\"url\":\"https://example.org/example-page\",…}",
  "identity": {                                  // сертификат flusher'а (как в меше)
    "certificate_json": "…каноническая строка…",
    "signature": "…base64, подпись домашнего сервера…"
  },
  "proof_signature": "…base64…"                  // sign(PRIV_клиента, "vrweb-flush.v1:" + payload)
}
```

`objects` внутри payload — плоские объекты слоя как есть:

```jsonc
[ { "id": "vpatch:n0-2", "kind": "vrweb-patch", "parent": "",
    "props": { "set": { "size": "Vector3(2,1,1)" }, "removed": false } },
  { "id": "u1.3", "kind": "vrweb-node", "parent": "page:n0",
    "props": { "tag": "OmniLight3D", "attrs": { "position": "Vector3(0,2,0)" } } } ]
```

- **Дельта** — `id`/`kind`/`parent`/`props`; `ts`/`ttl`/`epoch`/`author` не передаются:
  права сервер проверяет по **подписанту запроса**, состояние комнаты его не касается.
  Порядок — родители раньше детей (вложенный `vrweb-node` монтируется под уже вставленного).
- **`id` — непрозрачный адрес** (`<user8>.<seq>`, разбор в [object-id.md](object-id.md)). Он
  служит ключом дедупликации и HTML-`id` (см. ниже), но **из префикса нельзя выводить авторство
  или права** — префикс штампует инициатор, а не авторитет. Кто вправе флашить — решается только
  по подписанту запроса, а не по тому, чей `user8` в `id`. Иначе флашер мог бы протащить объект
  под чужим адресом.
- **Подпись одним round-trip'ом** вместо challenge–response: `proof_signature` —
  `rsa-sha256` над байтами `"vrweb-flush.v1:" + payload`. Свежесть даёт `ts` (окно ±5 минут
  по часам сервера), одноразовость — `nonce` (сервер помнит виденные внутри окна). Подпись
  покрывает `url` и `base_rev` — перенаправить чужую дельту на другую страницу/версию нельзя.
- **Проверка идентичности сервером** — ровно два шага из [home-server.md](home-server.md):
  подпись домашнего сервера на сертификате (ключи с `https://<домен из address>/.well-known/vrweb`,
  кэшируются) + `proof_signature` ключом из сертификата. Алгоритм — `rsa-sha256`, как везде.
- **Локальный fast-path**: пользователю, чей аккаунт живёт на этом же сервере, достаточно
  `Authorization: Bearer <token>` вместо `identity`/`proof_signature` (так флашит владелец
  персонального пространства) — оптимизация, не замена федеративного пути.
- **Анонимная дельта** (без Bearer и `identity`) допустима только при
  `identity_required: false` — целиком на усмотрение сервера (аналог анонимных правок вики).

## Ответ сервера

`200 OK` — сервер обработал запрос (даже если всё отклонил по правам на уровне объектов):

```jsonc
{
  "v": 1,
  "status": "partial",                  // "applied" | "partial" | "rejected" | "deferred"
  "rev": "b7c2...",                     // новая версия базы (если хоть что-то применено)
  "results": {
    "vpatch:n0-2": { "outcome": "applied" },
    "u1.3":        { "outcome": "rejected", "reason": "tag not allowed" },
    "u1.9":        { "outcome": "deferred" }   // принято на рассмотрение (модерация)
  }
}
```

- Гранулярность — **объект**: внутри объекта применение атомарно, между объектами — нет
  (ровно как «частичный отказ батча» в консоли пространства). `status` — агрегат по `results`.
- `deferred` — «принял, не значит применил»: сервер взял дельту на премодерацию. `rev` при
  этом не меняется; клиент трактует как «не применено» (пометки не ставит — см. ниже), а
  сервер применит позже сам. Уведомлений о результате в v1 нет (точка расширения).
- Ошибки уровня запроса — HTTP-статусы, тело `{"error": {"code", "message"}}`:

  | Код | Смысл | Реакция клиента |
  |---|---|---|
  | `401` | сертификат/подпись/nonce не прошли | перевыпустить сертификат, повторить |
  | `403` | политика не разрешает этому адресу | показать отказ |
  | `409` | version skew: `base_rev` устарел; тело несёт `current_rev` | перезагрузить страницу, пересчитать дельту против новой базы (дифф уже так умеет), повторить |
  | `413` | превышены `max_objects`/`max_bytes` | разбить/сократить |
  | `415` | незнакомая версия media type | договориться о старшей общей версии |

`409` — главный рабочий конфликт: страница могла измениться (в т.ч. чужим флашем) между
загрузкой и сохранением. Пересчёт дельты против свежей базы — та же механика, что дифф
консоли (`SceneHtml.diff_scene`): оверрайды, совпавшие с новой базой, уйдут сами, реальные
конфликты всплывут пользователю.

## Семантика применения на сервере

Маппинг kind → мутация HTML-дерева страницы (внутри её блока `<vrwml>`):

| Kind | Мутация |
|---|---|
| `vrweb-patch` (`vpatch:<nodeId>`) | merge `set` в атрибуты узла `<nodeId>`; `removed: true` — удалить узел с поддеревом |
| `vrweb-node` | вставить элемент `tag` c `attrs` под родителя (`parent="page:<id>"` → под узел `<id>`; `""` → в корень блока) |

Требования к серверу:

- **Сохранять id объекта как HTML-`id` вставляемого элемента** (`<OmniLight3D id="u1.3" …>`).
  Это не косметика, а опора дедупликации (см. ниже): объект оверлея и его «запечённая» копия
  в базе обязаны совпадать по адресу.
- Значения атрибутов — сырые строки, сервер их **не интерпретирует** (валидировать по белому
  списку тегов/атрибутов — может и должен, исполнять — нет).
- Применение дельты и переиздание страницы с новым `rev` — атомарно на стороне сервера.
- Идемпотентность повторов: повторный `vrweb-patch` с теми же значениями — но-оп; повторный
  `vrweb-node` с id, уже существующим в базе, — отказ объекта (`"already persisted"`), не дубль.
- **Пришпиливание структурных id**: при первом изменяющем флаше сервер записывает структурные
  id (`n0-2`) явными атрибутами `id` всем узлам без авторского. Иначе удаление/вставка узла
  сдвигала бы адреса соседей, и патчи следующих флашей целились бы мимо. Идемпотентно,
  детерминизм id у клиентов не ломает (авторский id побеждает при индексации).

## Дедупликация: проблема «А сохранил, С зашёл»

Сценарий из задачи. А и Б в комнате, база `rev=5`, в оверлее `u1.3` (vrweb-node) и
`vpatch:n0-2`. А флашит, сервер применил, база стала `rev=6`. Заходит С: он фетчит **новую**
базу (узел `u1.3` уже в ней) и получает от авторитета снапшот слоя с теми же объектами —
наивно `u1.3` построится у него дважды.

Решение — три правила, каждое самодостаточно (пометка нужна для GC/UI, не для корректности):

1. **Дедуп по id против базы (корректность).** `EphemeralView` не строит `vrweb-node`,
   чей `id` уже есть в индексе страницы (`SceneHtml.build_page_index`) — сервер обязан
   сохранять id при флаше именно ради этого. У С (`rev=6`) — id в базе → скип; у А и Б
   (`rev=5`) — id в базе нет → строится, как раньше. Никто не видит дубля, при этом состояние
   слоя у всех **одинаковое** — расходится только материализация, ровно как задумано
   (материализация и так функция от «база + оверлей», а базы у пиров уже разные).
   `vrweb-patch` дедупа не требует: применение патча к базе, куда он уже влит, — но-оп по
   построению (оверрайды совпадают со значениями базы).

2. **Анти-коллизия с другой стороны.** Чтобы правило 1 не «съело» честный новый объект,
   валидация `add` для `vrweb-node` отклоняет id, уже занятый в индексе базы (расширение
   готового анти-хайджека «add с занятым id — отказ»). Тогда объект оверлея с id из базы
   мог появиться единственным путём — это его же запечённая копия.

3. **Пометка `persisted` (сервис, не корректность).** После ответа сервера flusher шлёт
   обычные `update` по каждому объекту с `outcome: "applied"`:
   `props: { "persisted_rev": "b7c2..." }`. Протокол слоя не меняется. Зачем:
   консоль/UI показывают «запечено в страницу»; GC пока не снимает помеченные объекты после
   достижения нужной rev всеми пирами. Эта работа ведётся в [едином roadmap](../roadmap.md).
   Если flusher упал между флашем и
   пометкой — ничего не ломается: дедуп держится на правиле 1.

Сходимость: А и Б продолжают видеть контент из оверлея, С — из базы; содержимое идентично
(дельта и есть то, что влито). При перезагрузке страницы А и Б получают `rev=6` и их
материализация тоже переключается на базу. Оверлей-объекты доживают до конца комнаты как
обычные эфемерные — это дешевле и надёжнее, чем гонять `remove` наперегонки с фетчем базы
(преждевременный `remove` стёр бы контент у всех, кто ещё на старой базе).

## Расширение: блобы (kind `vrweb-blob`)

Объекты дельты могут ссылаться на **realtime-ресурсы** — байты, созданные в сессии
(картинки инструмента размещения): в атрибутах стоит `vrwebblob://sha256/<hex>`, а сами
байты на сервере страницы отсутствуют. Чтобы флаш-пакет был самодостаточным (без привязки
к p2p), клиент кладёт такие блобы **документной формой** первыми в `objects`:

```jsonc
{ "id": "blob:ab12…", "kind": "vrweb-blob", "parent": "",
  "props": { "algo": "sha256", "hash": "ab12…", "data": "<base64>" } }
```

Сервер, анонсировавший `vrweb-blob` в `accepts_kinds`, обязан сверить `sha256(data) == hash`
и волен: сохранить байты файлом и переписать `vrwebblob://`-ссылки применяемых объектов на
URL у себя; либо вставить в блок `<vrwml>` элемент `<VRWebBlob hash data/>` (ссылки работают
у клиентов и без p2p); либо отклонить по размеру/типу/квоте.

Референсная реализация ([homeserver/vrweb_scene.py](../../homeserver/vrweb_scene.py)) выбирает
инлайн-путь (страницы — в простой БД, без файлового хранилища): верифицирует хэш, дедуплицирует
по содержимому, печёт `<VRWebBlob>` в разметку. Обход верификации закрыт — `<VRWebBlob>` нельзя
подделать через `vrweb-node` или патч; лимиты `MAX_BLOB_BYTES`/`MAX_BLOBS`, тело до `MAX_BYTES`
(4 МиБ). Детали — «Персистенция» в [realtime-resources.md](realtime-resources.md).

**Совместимость**: консоль шлёт блобы только после capability-запроса — старый сервер
(без `vrweb-blob` в `accepts_kinds`) получает пакет прежнего формата, а объекты со ссылками
на блобы пропускаются, чтобы не запечь в страницу мёртвые ссылки. Заодно клиент уважает
анонсированный `max_bytes` (пакет с файлами легко его превышает). Подробно — «Персистенция»
в [realtime-resources.md](realtime-resources.md).

## Границы v1

Протокол v1 сохраняет только поддерживаемые документные kinds, не рассылает обязательный
reload после flush, не возвращает поздний исход `deferred` и не определяет rollback истории
страницы. Все расширения persistence ведутся в [едином roadmap](../roadmap.md).
