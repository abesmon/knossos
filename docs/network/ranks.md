# Ранги (таблица прав комнаты)

> Авторитет комнаты (см. [authority.md](authority.md)) ведёт **таблицу рангов** — кто на что
> имеет право. Таблица рассылается всем; ей доверяют, потому что её источник — авторитет.
> При смене авторитета таблица наследуется и продолжает жить. Это надстройка над
> [авторитетом](authority.md) и базовой [сетью](multiplayer.md).

## Модель

- **Таблица:** `user_id (String) → rank (int)`. Хранится у каждого пира, **владелец —
  авторитет**: только он её правит и рассылает.
- **Чем МЕНЬШЕ ранг — тем больше прав.** `0` ≈ админ. Так удобнее планировать «сколько прав
  дать»: описываешь от нуля, а «почти без прав» — это просто большое число (дефолт для тех,
  кого нет в таблице, — `DEFAULT_RANK`, заведомо далеко от нуля).
- **Все ранги явные**, включая собственный ранг авторитета `0`. Это сделано нарочно: запись
  привязана к `user_id`, поэтому переживает уход пира и его перезаход (см. ниже).

## Почему это не требует переговоров и (почти) не эксплойтится

Ранги ложатся ровно на «накопленное состояние авторитета» из [authority.md](authority.md):

- **Доверие к содержимому.** Таблицу принимают только от авторитета: получатель сам вычисляет
  `authority_id()` и проверяет `sender_is_authority()` в обработчике `_recv_ranks`. Подсунуть
  свою таблицу, не будучи авторитетом, нельзя.
- **Кто авторитет** — детерминированная функция состава (минимальный `join_seq` среди
  подключённых участников комнаты), её каждый считает локально. Никаких выборов.
- **Наследование при смене авторитета.** Новый авторитет уже держит «тёплую» копию таблицы
  (получал её рассылки) → продолжает рассылать её + может править. Становясь авторитетом, пир
  явно прописывает себе `rank = 0` (`_claim_authority_rank`).

## Идентичность: `user_id`, а не `peer_id`

Ключ таблицы — **постоянный `user_id`** ([Settings.user_id](../../scripts/settings.gd):
16 криптослучайных байт в hex, персистится в `user://`), а НЕ эфемерный `peer_id` (он новый
при каждом подключении). Иначе ранг терялся бы при перезаходе.

Свой `user_id` видно на экране настроек (вкладка «Сеть»): поле только для чтения, кнопки
«Копировать» и «Переиздать». Переиздание (`Settings.regenerate_user_id`) генерит новый id и
сразу персистит — выданные прежнему id ранги при этом теряются (новый id никто не знает).

Связь «кто есть кто» в рантайме: пир сообщает свой `user_id` в карточке идентичности
(`_recv_identity`), отсюда `NetworkManager` держит `peer_id → user_id`. Карточка идёт уже
по p2p/RPC, не через сигналинг: ник пира может быть виден раньше по `peer_joined`, но
`user_id` появится только после `p2p_peer_connected`. Чтобы узнать ранг пира:
`peer_id → user_id → rank`.

**Сценарий ради которого всё это:** A — авторитет, ранг 0. A уходит → авторитетом становится
B, продолжает рассылать таблицу (в ней всё ещё `A → 0`). A возвращается (уже НЕ авторитет, у
него новый `peer_id`) → по своему стабильному `user_id` находит себя в таблице и сохраняет
ранг 0. Доверие унаследовалось от пира к пиру, без участия A.

### Доверие: самозаявленный id и подтверждённая идентичность — обе постоянны

`user_id` **самозаявлен и не подписан**: скопировав чужой `user_id`, пир может выдать себя за
того пользователя и унаследовать его ранг. Важно понимать — это **не временный компромисс на
пути к «настоящей» верификации, а сознательная часть модели.** Стек VRWeb **намеренно не
защищает** от выдачи себя за другого на уровне транспорта/технологий и не планирует этого делать.

Рядом уже работает вторая система — **подтверждённая идентичность** от домашних серверов
([home-server.md](home-server.md)): подписанный сертификат «`nick@domain` ↔ публичный ключ
клиента», доказательство владения ключом в меше (challenge–response,
`NetworkManager.identity_verified`), галочка `✓ nick@domain` в списке «Пользователи». **Обе
системы постоянны и живут параллельно:** подтверждённая идентичность не заменяет самозаявленную,
а дополняет её как опциональный, более сильный якорь доверия. Так и останется.

Текущее ограничение политики доверия:

- **Сейчас клиент доверяет всем одинаково** — и подтверждённым, и «голым» `user_id`. Авторитет
  принимает ранги/действия/сообщения любого пира так, будто он тот, кем представился.
- Целевая модель не запрещает неподтверждённых пиров на уровне транспорта: решение принимать,
  предупреждать или фильтровать их действия остаётся пользовательской policy клиента. Реализация
  такой policy ведётся в [едином roadmap](../roadmap.md#p2--space-console-и-live-editing).

Техническая опора для строгих политик уже есть: `peer_id` привязан мешем и неподделываем
([authority.md](authority.md)), а поверх него сертификат доказывает `nick@domain`. Клиент, которому
нужна строгая политика доверия, может на это опереться; клиент, которому достаточно открытости, —
нет. Выбор за реализацией и пользователем, а не за транспортом.

## Согласованность при смене авторитета

`peer_leave` старого авторитета приходит по сигналингу (WebSocket), а рассылка таблицы новым
авторитетом идёт по мешу — разные транспорты, порядок не гарантирован. Чтобы push новой
таблицы не был отвергнут раньше времени, добавлен **pull**: как только пир пересчитывает
авторитета на другого (`authority_changed`), он запрашивает таблицу у него (`_request_ranks`
→ авторитет отвечает `_recv_ranks`). Плюс авторитет сам пушит таблицу новичкам при
подключении и при каждом изменении. Итог — согласованность без периодического heartbeat.

## API (`NetworkManager`)

| Член | Назначение |
|---|---|
| `set_rank(user_id, rank)` | назначить ранг (только авторитет; рассылает таблицу) |
| `clear_rank(user_id)` | убрать запись → `DEFAULT_RANK` (только авторитет) |
| `rank_of_user(user_id) -> int` | ранг по стабильному id; нет в таблице → `DEFAULT_RANK` |
| `rank_of_peer(peer_id) -> int` | ранг пира; авторитет всегда `0` |
| `my_rank() -> int` | наш ранг (если мы авторитет — `0`) |
| `user_id_of(peer_id) -> String` | `user_id` пира (из карточки); `""` если ещё не пришла |
| `ranks_snapshot() -> Dictionary` | копия таблицы (для UI/отладки) |
| сигнал `ranks_changed()` | таблица или привязка `peer_id→user_id` изменилась |
| `const DEFAULT_RANK` | ранг по умолчанию (`1<<30`) — практически без прав |

## UI: раздел «Пользователи»

В настройках есть вкладка **«Пользователи»**, доступная только **онлайн и находясь в
инстансе** (иначе скрыта — [scenes/settings.gd](../../scenes/settings.gd),
`_update_users_availability`). Список живой: перестраивается по сигналам `peer_joined`/
`peer_left`/`identity_received`/`ranks_changed`/`authority_changed`/`connection_changed`.

Показывает всех — себя (первой строкой, с пометкой `(вы)`), всех онлайн-пиров и записи о
рангах для офлайн-юзеров:

- **Онлайн-пир** (`● ник`): если у него есть ранг — он виден; если нет — явная инфа о ранге не
  выводится (просто «—»). Если пир известен только через сигналинг, строка помечается
  «P2P подключается» — значит WebRTC/RPC-канал ещё не открыт и `user_id` технически не мог
  прийти. Если p2p уже открыт, но карточка ещё не пришла, строка помечается
  «ID ещё не получен». В обоих случаях рангом управлять нельзя, пока неизвестен `user_id`.
- **Офлайн-запись** (`○ ID xxxxxxxx… (офлайн)`): ранг есть, а пира онлайн нет. Тоже бывает.
- **Авторитет** отмечается символом `★` после имени (`NetworkManager.authority_user_id`).

**Для авторитета** у каждой *чужой* строки с известным `user_id` появляются контролы:
`SpinBox` + «Задать» (`set_rank`) и «Удалить ранг» (`clear_rank`, активна только если ранг
есть). Своя строка и для не-авторитета весь раздел — только просмотр.

### Как проверять право на действие

```gdscript
# «действие требует ранга не хуже 10» (меньше = больше прав):
if NetworkManager.my_rank() <= 10:
    do_privileged_thing()
```

Для действий, рассылаемых по сети, проверяй ранг **отправителя на приёме** —
`rank_of_peer(get_remote_sender_id())` — а не только у себя: получатель не должен верить, что
отправитель «имел право», на слово (тот же принцип, что и `sender_is_authority`).

Сейчас ранг не входит в public Scripting API. Возможное versioned capability и связанная модель
доверия ведутся в [едином roadmap](../roadmap.md#p2--space-console-и-live-editing).
