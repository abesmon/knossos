# Realtime-ресурсы: контент-адресуемые блобы

> Четвёртый вид ресурсов сцены — рядом со ссылками (http), локальными
> (`vrweblocal://`/`vrwebresource://`, [local-resources.md](../client/local-resources.md)) и
> ресурсами страницы. Realtime-ресурс создаётся пользователем **во время сессии** (первый
> случай — картинка инструмента размещения) и не существует ни на каком сервере: его байты
> живут у пиров комнаты и передаются p2p. В [эфемерный слой](ephemeral-changes.md) и по сети
> уходит только короткая **ссылка по хэшу содержимого**.

## Зачем и почему не base64 в props

Наивный способ дать эфемерному объекту картинку — вшить base64 в `props`. Он плох всем:
+33% к размеру, повторная пересылка одних и тех же байтов в каждом снимке/событии,
`MAX_PROPS_BYTES` (8 КиБ) не оставляет места, а состояние слоя перестаёт быть «плоскими
лёгкими данными». Вместо этого блоб разделён на **адрес** и **байты**:

```
vrwebblob://sha256/<64 hex>          ← ссылка, ~75 байт, в props/атрибутах
```

- **Самоверифицируемость**: адрес — sha256 содержимого. Приёмник сверяет хэш — качать
  можно у ЛЮБОГО пира без доверия, авторитет в передаче не участвует. Подменить байты
  по ссылке невозможно.
- **Иммутабельность**: блоб не меняется никогда — не нужен протокол инвалидации; «другая
  картинка» = другой блоб и `update` объекта с новой ссылкой.
- **Дедупликация даром**: одна картинка, размещённая десять раз, хранится и качается один раз.

## Три формы одного блоба

| Форма | Где живёт | Формат |
|---|---|---|
| **Ссылка** | props объектов слоя, атрибуты vrweb-тегов | строка `vrwebblob://sha256/<hex>` |
| **Бинарная** | p2p WebRTC между пирами комнаты | чанки по 16 КиБ, без base64 (RPC несёт `PackedByteArray` нативно) |
| **Документная** | флаш-пакет персистенции и (по желанию сервера) сама страница | base64 — допустим в одноразовом HTTP/HTML, не в горячем пути |

Документная форма делает пакет/страницу **самодостаточными**: сервер персистенции получает
байты без всякой привязки к p2p и сам решает, что с ними делать (см. «Персистенция» ниже).

## Лимиты

| Константа | Значение | Смысл |
|---|---|---|
| `BlobProtocol.MAX_BLOB_BYTES` | 2 МиБ | потолок размера блоба — общий для импорта, p2p и рекомендованный серверам |
| `BlobProtocol.CHUNK_BYTES` | 16 КиБ | чанк p2p-передачи (безопасен для WebRTC-сообщения) |
| `BlobStore.MAX_MEM_BYTES` / `MAX_DISK_BYTES` | 64 / 256 МиБ | капы хранилища (см. «Безопасность») |
| `BlobStore.IMPORT_KEEP_BYTES` | 512 КиБ | мельче — оригинал как есть; крупнее — пережим |

## p2p-передача

```
requester                                  пиры комнаты
   |── _recv_blob_query(hex) ─────────────▶|  у кого есть локально —
   |◀─ _recv_blob_have(hex) ───────────────|  отвечают have
   |── _recv_blob_pull(hex) ──▶ ПЕРВЫЙ ответивший
   |◀─ _recv_blob_chunk(hex, i, size, bytes) ── пейсинг ~8 чанков/кадр
   └─ сборка BlobProtocol.Rx → сверка sha256 → BlobStore.ingest
```

- **Запрос ленивый**: блоб качается только когда материализованный объект попросил текстуру
  (`ImageLoader` → `BlobStore.request`), а не «всё, на что ссылается снимок».
- **Один активный источник** на блоб; чанки от прочих отправителей — дроп.
- **Мусорный провайдер** (хэш/размер не сошлись, замолчал на `BLOB_STALL_SEC`) попадает в
  bad-список этого блоба, передача сбрасывается, ретрай — у другого пира.
- **Ретраи**: перезапрос неотвеченных ~раз в `BLOB_RETRY_SEC`; новому p2p-пиру недостающие
  блобы задаются адресно (он мог принести байты, которых нет у старожилов — например, автор
  переподключился). Таймаута «навсегда» нет: ожидание живёт, пока жив консьюмер
  (умершие колбэки отсеиваются, мир снесён навигацией → ожидание само исчезает).
- **Отдача**: лимит одновременных отдач (`MAX_BLOB_UPLOADS`, per-peer
  `MAX_BLOB_UPLOADS_PER_PEER`), лишние pull молча дропаются — запросчик ретраит.
- **Смена комнаты**: активные передачи сбрасываются, сами ожидания живут — докачаются у
  пиров новой комнаты, если ссылки встретятся и там.

Доступность = как у самих эфемерных объектов: пока в комнате есть хоть один пир с байтами.
Ушли все — ссылка без блоба, картинка деградирует в заглушку «🖼 ✕» ([graceful
degradation](../graceful-degradation.md)); байты вернутся — доедет при следующем запросе.

## Импорт картинок (`BlobStore.import_image`)

Файл пользователя → компактный платформонезависимый блоб:

- формат распознаётся **по сигнатуре** (png/jpg/webp/gif), MIME/расширению не доверяем;
- оригинал ≤ 512 КиБ — как есть (родной формат, GIF-анимация живёт);
- крупнее — декод, сторона ≤ 1280, пережим в WebP (lossy 0.85), при необходимости
  половиним разрешение до попадания в `MAX_BLOB_BYTES`. Крупный GIF — статичный первый кадр.

## Теги vrweb-слоя

### `<VRWebImage>` — картинка, размещённая в мире

```html
<VRWebImage src="vrwebblob://sha256/ab12…" alt="кот.png"
            position="Vector3(1, 1.6, -2)" rotation="Vector3(0, 1.57, 0)" width="2"/>
```

Кастомный тег ([vrweb-tags.md](../space/vrweb-tags.md)): квад с текстурой
(`PlacedImage extends ImagePanel`), якорь **центром** в своей точке. `src` — блоб-ссылка
или обычный URL (блоб-ссылки не резолвятся против base — они абсолютны); `width`/`height` —
метры (нет — натуральный размер текстуры с капом 3 м); прочие атрибуты — свойства Node3D.
Текстуру грузит сам через `ImageLoader` мира (группа `image_loader`).

### `<VRWebBlob>` — документная форма (байты инлайн)

```html
<VRWebBlob hash="ab12…64hex…" data="<base64>"/>
```

Мета-тег (не узел): при сборке страницы/оверлея байты сверяются с `hash` и уходят в
`BlobStore.ingest` — страница может нести блобы **инлайн**, и все `vrwebblob://`-ссылки
резолвятся вообще без p2p. Байты, не совпадающие с хэшем, молча отбрасываются.

## Инструмент размещения (клавиша 3)

Двухфазный, как курсор-«призрак» в редакторах: сначала **прицеливание**, потом выбор файла.

1. **`3`** — включается режим прицеливания. Из камеры бьёт луч (макс. 1.8 м); на его конце —
   полупрозрачный квадрат 0.3×0.3, показывающий место и разворот будущей картинки.
   Попал в поверхность (стена/пол, слой 1) — квадрат прижат к ней и развёрнут **по нормали**;
   мимо — висит на конце луча **лицом к игроку**. Обновляется каждый физ-кадр (луч —
   `direct_space_state`, только в `_physics_process`).
2. **ЛКМ** — подтвердить точку: открывается системный диалог выбора картинки. Трансформ
   прицеливания зафиксирован в момент клика (камера дальше может двигаться).
3. **ПКМ** или повторное **`3`** (а также Esc / потеря фокуса окна) — отмена режима.

Выбранный файл → `BlobStore.import_image` → **обычный** `add` эфемерного слоя (протокол слоя
не менялся, как обещал раздел «Новые инструменты» в [ephemeral-changes.md](ephemeral-changes.md)):

```jsonc
{ "op": "add", "id": "u1.7", "kind": "vrweb-node", "parent": "", "ttl": 0,
  "props": { "tag": "VRWebImage",
             "attrs": { "src": "vrwebblob://sha256/…", "alt": "кот.png",
                        "transform": "Transform3D(…)" } } }
```

Разворот несёт полный `transform` (позиция + ориентация по нормали/лицом к игроку) одним
атрибутом. Поскольку это `vrweb-node`, материализация (EphemeralView → VrwebBuilder), консоль
пространства, права владения и дедуп персистенции работают готовыми путями. Офлайн (вне
комнаты) — локальный узел напрямую в мир, как офлайн-штрихи карандаша. Код: прицеливание и
превью — `Player` (`image_placement_active`/`image_placement_confirmed`,
[actors/player/player.gd](../../actors/player/player.gd)), диалог и создание объекта —
`main._on_image_placement_confirmed` ([scenes/main.gd](../../scenes/main.gd)).

## Персистенция: kind `vrweb-blob` во флаш-пакете

Расширение протокола [page-persistence.md](page-persistence.md). Если дельта ссылается на
блобы, консоль кладёт их **документной формой** первыми в `objects`:

```jsonc
{ "id": "blob:ab12…", "kind": "vrweb-blob", "parent": "",
  "props": { "algo": "sha256", "hash": "ab12…", "data": "<base64>" } }
```

Пакет самодостаточен — p2p серверу не нужен. Что сервер делает с блобом — **его политика**:

1. **Файлом**: проверить `sha256(data) == hash`, сохранить, в применяемых атрибутах
   переписать `vrwebblob://sha256/<hex>` на URL файла у себя. Страница дальше живёт на
   обычных ссылках.
2. **Инлайн**: вставить в блок `<vrweb>` элемент `<VRWebBlob hash data/>` — ссылки
   продолжают работать у любых клиентов без p2p (и без файлового хранилища на сервере).
3. **Отклонить** kind целиком (или отдельные блобы — по размеру/типу/квоте).

Требования к серверу, принимающему блобы: анонсировать `vrweb-blob` в `accepts_kinds`
capability-ответа; **обязательно** сверять sha256; ограничивать размер/количество; убедиться
sniff'ом, что контент ожидаемого типа (для картинок — сигнатуры png/jpg/webp/gif).

**Референсная реализация** ([homeserver/vrweb_scene.py](../../homeserver/vrweb_scene.py),
`_apply_blob`) выбирает путь **инлайн**: страницы лежат в простой БД, файлового хранилища
нет, поэтому блоб печётся в разметку как `<VRWebBlob id="blob:<hex>" algo hash data/>`.
Гарантии: `algo == sha256`, `hash` — 64 hex, `id == "blob:"+hash`, `sha256(decode(data)) == hash`,
размер ≤ `MAX_BLOB_BYTES` (2 МиБ), не более `MAX_BLOBS` (4) блобов за флаш; data перекодируется
канонически. Дедуп — по `hash` уже вшитых `<VRWebBlob>` (повторный флаш → `already persisted`).
Обход верификации закрыт: `<VRWebBlob>` **нельзя** создать через `kind=vrweb-node` (tag заблокирован)
и нельзя мутировать патчем (иммутабелен — иначе порвётся контент-адресация). `MAX_BYTES` тела
поднят с 256 КиБ до 4 МиБ под документную форму (и анонсируется в `max_bytes`).

**Совместимость со старым сервером** — гейт по capability на клиенте
([scripts/space_console.gd](../../scripts/space_console.gd)): перед флашем с блобами консоль
зовёт `GET <endpoint>?url=…`; нет `vrweb-blob` в `accepts_kinds` — блобы не отправляются, а
объекты со ссылками на них **пропускаются** (иначе в страницу запеклись бы мёртвые ссылки),
пользователю — честный статус. Пакет для старого сервера байт-в-байт совпадает с прежним
форматом. Заодно уважается анонсированный `max_bytes` (иначе флаш с файлами отлетал бы 413).

## Безопасность

Модель доверия: **контент проверяется хэшем, не отправителем** — сверка на p2p-приёме,
при чтении дискового кэша, при ingest из документа и (по контракту) на сервере при флаше.
Остаются атаки на ресурсы:

| Угроза | Защита |
|---|---|
| Флуд хранилища через слой (256 объектов × 2 МиБ ссылок) | капы BlobStore (память 64 МиБ LRU, диск 256 МиБ по mtime) + ленивый запрос (только по требованию материализации) |
| Мусорный провайдер (have + мусор/зависание) | один источник на блоб, финальная сверка хэша, bad-список источника, ретрай у другого |
| Чанк-спам без запроса | чанки без pending-записи или от чужого отправителя — молчаливый дроп |
| DoS раздачей | лимиты одновременных отдач (глобальный и per-peer), лишние pull — дроп |
| Файловый спам через флаш | флаш аутентифицирован (Bearer/сертификат) — атрибутируемо; капы и квоты — политика сервера |
| Вредоносные байты картинки | тот же декод-путь, что у любых картинок недоверенного веба (ImageLoader); лимиты импорта |
| Неприемлемый контент | уровень доверия штрихов/чата; точка расширения — политика прав по kind в SceneChanges ([ranks.md](ranks.md), [security.md](../security.md)) |

## Границы v1

- **Доступность** — сессионная: ушли все держатели байтов → заглушка (см. выше). Флаш с
  блобами — путь к настоящей персистентности.
- **GIF-анимация** сохраняется только у оригиналов ≤ 512 КиБ.
- **Один источник на блоб** — без параллельной догрузки кусками от нескольких пиров
  (торрент-стайл — возможное расширение, контракт чанков уже позволяет).
- **Права на блобы не нужны**: они иммутабельны и анонимны; права остаются на уровне
  объектов слоя (владение `vrweb-node`).
- **Типы**: сейчас блобы — картинки; контракт типа не знает (байты + хэш), будущие типы
  (аудио-сэмплы, мелкие меши) лягут на тот же слой новыми потребителями.
