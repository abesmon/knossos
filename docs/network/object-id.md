# Формат `id` объекта (`<user8>.<seq>`)

Единый источник правды по адресам объектов эфемерного слоя. Ссылаются сюда:
[ephemeral-changes.md](ephemeral-changes.md) (генерация), [page-persistence.md](page-persistence.md)
(как сервер относится к `id` при флаше).

## Структура

`id` вида `57e86255.1` — это **не** одно большое число, а `<user8>.<seq>`:

- **`user8`** — первые 8 hex-символов `Settings.user_id` инициатора (сам `user_id` — 16 случайных
  байт в hex, 32 символа; см. `Settings._new_user_id`). Отсюда «хаотичный» вид — это обрезок
  случайного hex, а не осмысленное число.
- **`seq`** — счётчик объектов пира (`NetworkManager._obj_seq`), инкрементится на каждый вызов
  `new_object_id()`.

Fallback: если `user_id` короче 8 символов (не должно случаться при нормальной генерации),
префикс — `p<peer_id>`.

Генератор — `NetworkManager.new_object_id()`; вызывается инициатором для `op=add`.

## Зачем именно так

Дешёвый **адрес без координации между пирами**:

- **уникальность** — 48 бит энтропии в префиксе делают коллизии между разными пирами
  практически невозможными, а счётчик разводит объекты одного пира. Договариваться заранее не
  нужно.
- **адресуемость** — по своему `id` пир потом правит/удаляет свой объект.

## Префикс недоверенный — важное свойство

Префикс штампует **инициатор**, а не авторитет, поэтому из него **нельзя выводить авторство или
права**:

- в realtime-комнате доступ проверяется по `bindings.creator`, который authority ставит из
  transport actor, а не по префиксу `id`;
- на сервере при флаше `id` — **непрозрачный адрес**: он служит ключом дедупликации и
  HTML-`id` вставляемого узла, но право флашить решается только по подписанту запроса, а не по
  тому, чей `user8` в `id`. Иначе флашер мог бы протащить объект под чужим адресом.

Короче: `id` — это адрес, а не удостоверение.
