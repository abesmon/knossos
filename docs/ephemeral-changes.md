# Эфемерные изменения сцены

> Транспортный слой поверх [авторитета](authority.md): любой пир **запрашивает** изменение
> сцены **действием** (action), авторитет — единственная точка коммита — валидирует и рассылает
> **события** (event). Контракт движок-агностичен: это протокол данных, а не Godot-специфика,
> чтобы его могла реализовать не-Godot сторона. Структура сериализуема — журнал событий можно
> будет выгрузить на сервер, чтобы тот обновил HTML-дерево (персистенция; пока не делается).

## Зачем

Сцена в VRWeb детерминирована по URL: открывшие одну страницу видят одинаковый мир
([multiplayer.md](multiplayer.md)). Эфемерные изменения добавляют в этот общий мир объекты,
которых нет в исходном HTML, **на лету**: временные порталы, метки, рисунки, расставленные
предметы. Первый инструмент — **[пузыри](#пузыри-первый-инструмент)**; дальше на тот же слой
лягут другие инструменты, **не меняя протокол**.

## Почему action/event, а не рассылка журнала

Наивный вариант — авторитет рассылает весь журнал на каждое изменение — плох:

- **Трафик** O(N объектов) на каждую мелкую правку.
- **Привязка к движку**: рассылать «состояние сцены» — значит зашивать форму состояния в
  транспорт. Нам нужен транспортный **контракт**, который реализует и не-Godot сторона.
- **Валидация при рассинхроне**: если инициатор шлёт «вот моё новое состояние», авторитету
  трудно понять, легально ли оно. Инициатор должен описывать **только нужную мутацию**.

Поэтому — модель **команда/событие** (command/event sourcing):

```
инициатор X        авторитет A (один)                 followers B,C
   | ACTION            |                                    |
   | {op,id,kind,…}    |  валидация против СВОЕГО состояния  |
   |----------------->|   + прав (владение/ранг)            |
   |                  |  мутация, штамп (epoch,seq)         |
   |                  |  EVENT ------------------------------>|  apply по порядку
   |                  |                                      |  (sender_is_authority)
```

- **Инициатор** шлёт авторитету **действие** — намерение, без претензий на состояние.
- **Авторитет** — единственная точка сериализации: валидирует против **своего** состояния и
  прав, присваивает порядковый `(epoch, seq)`, мутирует состояние и рассылает **событие**.
- **Followers** применяют события строго по порядку. Рассинхрон у инициатора не ломает
  валидацию — авторитет проверяет своё состояние, а не заявленное инициатором.

## Объект состояния

Состояние — `id → object`, где object — плоский JSON-сериализуемый словарь:

| Поле | Тип | Смысл |
|---|---|---|
| `id` | String | адрес объекта. Для `add` генерит **инициатор** (`<user8>.<seq>`) — чтобы потом адресовать СВОЙ объект |
| `kind` | String | тип объекта (`"bubble"`, будущие). По нему вьюха решает, что строить |
| `parent` | String | **куда**: `""` — корень мира \| `"<id>"` — другой объект \| `"page:<nodeId>"` — узел HTML-дерева страницы |
| `author` | String | `user_id` владельца — **штампует авторитет** из привязки `peer→user_id` (инициатор не задаёт) |
| `ts` | float | время создания по часам авторитета (отсчёт TTL) |
| `ttl` | float | секунд жизни; `>0` — эфемерный (истекает), `0` — постоянный (кандидат на выгрузку) |
| `props` | Dictionary | JSON-сериализуемая нагрузка под `kind` (позиции — как `[x,y,z]`, не `Vector3`) |

## Протокол

### Действие (инициатор → авторитет)

Плоское намерение. Поля по op:

```jsonc
{ "op": "add",      "id": "u1.3", "kind": "bubble", "parent": "",
  "ttl": 30, "props": { "url": "…", "position": [1,1.6,5], "label": "Вася" } }
{ "op": "update",   "id": "u1.3", "props": { "label": "новый" } }   // props — ПАТЧ (мердж; null удаляет ключ)
{ "op": "remove",   "id": "u1.3" }
{ "op": "reparent", "id": "u1.3", "parent": "u1.2" }
```

Инициатор НЕ задаёт `author`/`ts`/`epoch`/`seq` — это привилегия авторитета.

### Событие (авторитет → все)

То же + порядковый штамп. Для `add` несёт полный объект; для `update` — **патч** (дельта);
для `remove`/`reparent` — минимум.

```jsonc
{ "epoch": 1, "seq": 7, "op": "add", "id": "u1.3", "kind": "bubble",
  "parent": "", "author": "a1b2…", "ts": 1719772800.0, "ttl": 30,
  "props": { … } }
```

### Порядок: `(epoch, seq)`

- `seq` — монотонный в пределах эпохи; авторитет — единственный источник, reliable-ordered канал
  сохраняет порядок.
- `epoch` растёт при **смене авторитета** (`begin_authority()` ставит `epoch = max(виденное)+1`),
  чтобы события нового авторитета были заведомо новее экс-авторитетских. Запоздавший пакет
  старого авторитета (меньший epoch) отбрасывается сам.
- **Применение у follower** (`apply_event`): `epoch<наш` → дроп (устаревшее); `epoch>наш` →
  **GAP** (новая эпоха раньше снимка — ресинк); `seq<=наш` → дубликат; `seq>наш+1` → **GAP**
  (пропуск); `seq==наш+1` → применить. GAP → запросить снимок (один раз, флаг `_scene_resync`).

### Снимок (ресинк)

`{ epoch, seq, objects }` — полное состояние. Нужен в двух случаях:

- **push новичку**: авторитет шлёт снимок при открытии p2p-канала (`_on_mp_peer_connected`);
- **pull при смене авторитета / GAP**: пир запрашивает снимок (`_request_scene_snapshot`).

Свежий follower (epoch 0) до снимка **не угадывает** эпоху и на любое событие отвечает GAP →
тянет снимок. Это правильно: он мог пропустить сотни событий до входа.

## Права (владение + фундамент под систему прав)

Базовое правило: **править/удалять/перемещать можно только СВОИ объекты**. Проверка — в
`SceneChanges` (чистая, тестируемая), решение принимает авторитет:

- `update`/`remove`/`reparent` объекта `X` — только если `X.author == sender` **или** sender
  админ.
- `add` ребёнка в `parent`: корень и узлы страницы (`page:…`) открыты; вложение в **чужой**
  эфемерный объект — только владельцу того объекта или админу.
- `add` с занятым `id` — отказ (анти-хайджек: нельзя «занять» чужой адрес).
- защита от цикла при `reparent` (новый родитель не может быть потомком объекта).

**Кто такой админ** определяет транспорт: `NetworkManager` передаёт в `authority_commit`
`sender_is_admin = rank_of_peer(sender) <= EPHEMERAL_ADMIN_RANK` (ранги — [ranks.md](ranks.md);
0 ≈ админ/авторитет). `SceneChanges` про ранги не знает — получает готовый булев. Это **точка
расширения**: сюда ляжет полноценная система прав (capabilities, политика по `kind`, пороги
рангов), не меняя транспорт.

`author` авторитет берёт из своей привязки `peer_id → user_id` (карточка идентичности), инициатор
его не задаёт. Якорь доверия — тот же, что у рангов; самозаявленность `user_id` — тот же принятый
компромисс до центров авторизации (см. [ranks.md](ranks.md)).

## Лимиты (защита)

В `SceneChanges`: `MAX_OBJECTS` (потолок числа объектов), `MAX_TTL` (потолок TTL),
`MAX_PROPS_BYTES` (размер `props`, заодно гарантия сериализуемости через `JSON.stringify`).
Превышение — отказ в коммите.

## TTL

Истечение — обязанность **авторитета** (единый владелец состояния и часов): в `_process`
(throttle ~4 Гц) он зовёт `expire(now)`, та снимает объекты с `ts+ttl<now` **каскадно** с
потомками и возвращает события `remove`, которые рассылаются. Followers TTL не применяют — ждут
событий. Преемник продолжает истекать со следующего кадра (`has_authority()` проверяется каждый
кадр). `ts` — настенные часы авторитета; для визуального таймера у followers рассинхрон NTP
(<1 c) на фоне 30-секундного TTL пренебрежим, а каноничное удаление приходит событием.

## Вложенность и каскад

`parent` даёт дерево объектов. `EphemeralView` монтирует объект с `parent=<id>` как **ребёнка
узла родителя** (наследование трансформа); при `reparent` — перемонтирует. Удаление родителя
(явное или по TTL) **каскадно** снимает всех потомков: авторитет шлёт `remove` на каждого
(листья раньше родителя). У вьюхи Godot-узел родителя уносит детей с собой — поэтому последующие
`remove` детей идемпотентны (`is_instance_valid`).

Якорь `page:<nodeId>` (привязка к узлу HTML-дерева) — точка расширения: у проекта уже есть
провенанс `id → HTML` (`TopologyBuilder.build(doc, true)`). Пока вьюха резолвит `page:` в корень;
именно этот якорь станет мостом к персистенции (эфемерный ребёнок узла страницы → реальный
дочерний элемент в HTML-дереве при выгрузке).

## API (`NetworkManager`)

| Член | Назначение |
|---|---|
| `request_scene_action(action)` | запросить изменение (если мы авторитет — коммит сразу) |
| `new_object_id() -> String` | сгенерить адрес нового объекта (для `op=add`) |
| `scene_objects() -> Dictionary` | снимок состояния (вьюха/отладка/выгрузка) |
| `scene_object(id) -> Dictionary` | один объект |
| сигнал `scene_object_added(id, object)` | объект добавлен |
| сигнал `scene_object_updated(id, object)` | объект изменён/перемещён (полное состояние) |
| сигнал `scene_object_removed(id)` | объект снят |
| сигнал `scene_reset()` | состояние перезагружено снимком — пересобрать всё |
| `const EPHEMERAL_ADMIN_RANK` | порог ранга для админ-обхода владения |

Сигналы несут **плоские данные**; консьюмер материализует по `kind`, не зная транспорта.

## Пузыри (первый инструмент)

**Пузырь** — временный портал «пользователь ушёл сюда». При навигации на **другую** страницу
уходящий шлёт действие `add` `kind="bubble"` в **покидаемой** комнате: объект с TTL 30 c в точке
игрока, указывающий на URL назначения. Остальные видят пузырь и, кликнув, идут вслед.

- **Создаёт** [main](../scenes/main.gd) в начале `_navigate`, пока меш покидаемой комнаты ещё жив
  и поллится (между действием и сносом меша идёт асинхронный фетч — reliable-пакет успеет уйти).
  Условия: онлайн, мы в комнате, целевая комната (`seed_key`) отличается от текущей.
- **Уходящий-авторитет** не теряет пузырь: коммитит и рассылает событие, и только потом уходит —
  followers уже применили, преемник наследует состояние снимком и истекает TTL.
- **Внешний вид** — полупрозрачная сфера-орб с подписью и обратным отсчётом, затухает к концу TTL
  ([actors/bubble/](../actors/bubble/)). Реализует тот же интерфейс луча, что и `Portal`
  (`interact_at`/`is_active_at`/`aim_hint_at`), плюс контракт `setup_object(object)`.

`props` пузыря: `{ "url": String, "position": [x,y,z], "label": String }` — всё JSON-сериализуемо.

## Персистенция (точка расширения, не сейчас)

Журнал событий — естественный лог для выгрузки на сервер: `add`/`update`/`remove`/`reparent`
ложатся прямо на мутации HTML-дерева (вставить узел под parent, поправить атрибуты, удалить).
`ttl=0` — постоянные объекты, кандидаты на флаш; `ttl>0` (пузыри) обычно не персистятся. Поля
(`author`, `ts`, `parent`, `props`) уже несут всё нужное для реплея.

## Границы

- **Split-brain**: при разрыве меша на компоненты у каждой свой авторитет; каждый поднимет свою
  эпоху. При слиянии `min`-правило выберет одного, остальные ресинкнутся снимком (эпохи могут
  столкнуться — для эфемерных объектов приемлемо). См. [authority.md](authority.md).
- **Новые инструменты**: новый `kind` + ветка в `EphemeralView._make_node` + (опционально) политика
  прав. Протокол (`action`/`event`/`snapshot`) не меняется.
- **Оптимистичное применение** у инициатора пока не делается — объект появляется по событию от
  авторитета (одна RTT). Для пузыря это неважно (инициатор уходит). При необходимости добавляется
  поверх без смены протокола.
