# Упаковка объектов по стенам (развёртка стен)

**Статус: реализовано** в [world_generator.gd](../scripts/world_generator.gd) (`_plan_objects` →
`_wall_boxes`/`_wall_walk`/`_group_faces`/`_make_box` → `_try_pack`/`_finalize_pack`, зовётся из
`_build_room`). Заменяет раскладку «1 объект = 1 клетка» упаковкой настенных объектов по их
**реальным метрическим габаритам** на непрерывные участки стен. Инварианты — в
[tests/test_space_geometry.gd](../tests/test_space_geometry.gd) (грани боксов в футпринте и не на
дверях; все объекты размещены; центры в футпринте).

## Зачем

Сейчас геометрия ([world_generator.gd](../scripts/world_generator.gd), `_object_slots` /
`_place_objects`) ставит **один объект в одну клетку** маршрута вне зависимости от его размера:
для каждой клетки дорожки берётся клетка «по левую руку», туда садится объект. Лишние объекты
уходят стопкой вверх. Это расточительно: маленький текст занимает столько же места, сколько
большая картинка, а большая панель не может «растечься» по длинной стене.

Размер каждого объекта **уже считается** — `WorldGenerator._measure_objects` →
`_object_size[id] = Vector2(w, h)` в метрах (rich-панель, картинка, видео, портал, текстовая
панель). Эта фаза остаётся и переиспользуется как «шаг 1» алгоритма ниже.

Идея новой раскладки — родная эволюция уже существующего «обхода по левую руку»: слот превращается
из *клетки* в *грань стены*, а грани сшиваются в непрерывные участки (боксы), по которым объекты
пакуются с учётом их ширины/высоты.

## Алгоритм

### 1. Замер объектов

Как сейчас: `_measure_object` для каждого настенного объекта (тексты, rich-панели, картинки,
видео, прочее) → его футпринт `Vector2(w, h)` в метрах. Без изменений.

### 2. Обход маршрутов → «виртуальные стены»

Идём по внутренним маршрутам комнаты (`SpaceLayout.rooms[id].routes`, фаза 7
[geometry-lab.md](geometry-lab.md)) **по часовой стрелке, отмечая грань стены по левую руку**.
«Виртуальная» — потому что неважно, стоит ли там реальная стена; запоминаем сам факт грани, она
понадобится как место под объект. Грань = пара `(клетка, направление)`.

Два типа комнат:

- **Тупик (один выход).** Маршрут — петля «зайти, обойти, вернуться». Идём по петле по часовой
  стрелке, на каждом шаге маркируем грань по левую руку (внешний контур петли). Пример:

  ```
  ++++++        +6789+
  +rrrr+        5rrrrA
  +r++r+   →    4r++rB
  +rrrr+        3rrrrC
  +++d++        +21dD+
  ```
  `r` — путь, `d` — единственная дверь. Цифры/буквы 1..D — порядок виртуальных стен по обходу.

- **Проходная (несколько выходов).** Обход выводится **прямо из `routes`** (звезда из путей
  вход→каждый выход). Зайдя в комнату, идём по первому маршруту, ведущему **налево относительно
  входа**, до его выхода; разворачиваемся, идём обратно **по тому же маршруту** до перекрёстка;
  от перекрёстка снова берём первый маршрут налево — и так «по звёздочке», пока не вернёмся ко
  входу. **Всё время, пока идём** (туда и обратно), маркируем грань по левую руку. Пример:

  ```
  +++++x++        +++++x++
  +++++r++        +45+7r8+
  xrr+rr++   →    xrr6rr9+
  ++rrr+++        +3rrrA++
  ++++d+++        ++21d+++
  ```

**Углы считаются дважды.** Двигаясь по клеткам и маркируя стену по левую руку, на углу мы как бы
поворачиваемся и маркируем стену в новом направлении тоже — поэтому одна клетка может дать
**больше одной полезной грани** (на углу — две). Это и даёт развёртке корректную длину на
поворотах.

### 3. Развёртка → боксы (разрез на углах)

Из упорядоченной последовательности граней строим **развёртку**, «порезанную на углах». Участок
между двумя разрезами — **непрерывная прямая стена** (бокс): максимальный пробег коллинеарных
соседних граней (одно направление, смежные клетки). Внутри одного бокса объект может занимать
несколько клеток в ширину → **на длинный бокс влезают большие объекты**, чего нынешняя
поклеточная раскладка не умеет.

Бокс имеет ширину `w_i` (длина пробега в метрах = клетки × `GRID`) и высоту `h_i` (в клетках,
изначально **1**). Высота — это «сколько рядов объектов стопкой» помещается на участке стены;
реальная высота стены потом подгоняется под контент (`_wall_height` уже так делает). Стены/боксы
могут расти вверх **неограниченно** — это отдельная от площади комнаты метрика (см. ниже).

**Объект сам расширяет бокс по высоте.** Старт `h_i = 1` клетка — это лишь начальный бюджет. Если
объекту нужна высота больше текущего бюджета (высокая панель, картинка), он **автоматически
раздвигает бокс до своей высоты** — объект никогда не обрезается. То есть высота ряда = высота
самого высокого объекта в нём, а рост `h_i` (шаг 4) добавляет место под **дополнительные ряды**.

### 4. Упаковка (shelf packing с ростом высоты)

Пакуем объекты **в порядке чтения** (приоритет — как нынешний «первый объект» для спавна),
жадно **слева направо, сверху вниз**, по боксам, с учётом `padding` (поля внутри бокса) и
`margin` (зазор между объектами):

1. кладём объект в текущий бокс, если влезает по ширине в текущий ряд (бокс при необходимости
   раздвигается под высоту объекта); иначе — новый ряд в этом же боксе, если влезает по высоте `h_i`;
2. бокс заполнен (по ширине и высоте) → переходим к следующему боксу;
3. дошли до конца боксов, а объекты остались → **возвращаемся к боксу 1, увеличиваем его `h` на
   1** и **полностью переразбираем** раскладку; не хватило снова → `+1` боксу 2; и так по кругу.
   Прошли все боксы (`+1` каждому) и всё ещё не влезло → новый круг с боксов 1.

Растём, пока не уместим всё.

#### Замечания к шагу 4

- **Порядок инкремента — это предпочтение, не только поиск вместимости.** Возврат к боксу 1
  первым означает, что **ранние стены (по чтению) набирают высоту раньше** поздних. Это осознанный
  выбор раскладки, а не побочный эффект.
- **Сложность** наивно O(N²) (N — объектов в комнате, полный repack на каждый `+1`). Для комнаты
  это всегда мало → отдельная оптимизация цикла не нужна. Дешёвый разгон: стартовать `h` не с 1, а
  с оценки по площади `ceil(Σ высот объектов / Σ ширин боксов)`, чтобы пропустить большинство
  кругов.
- **Предел высоты — мягкий.** Физического потолка у «стопки рядов» нет (стена тянется под
  контент), но из соображений читаемости стоит задать кап и поведение при переполнении (ужать /
  скролл / явная стопка, как сейчас лишние объекты).

## Размер комнаты и площадь стен — РАЗВЯЗАНЫ

Это **две независимые метрики, их не схлопывать**:

- `need` ([space_layout.gd](../scripts/space_layout.gd) `_estimate_need`) — просто число, по
  которому считается **площадь комнаты** (пол, форма). Остаётся как есть; развёртка его **не
  питает**.
- **Площадь виртуальных стен** оторвана от `need` и на него никак не рассчитывает: боксы могут
  расти вверх **бесконечно**. Поэтому упаковка всегда выполнима по высоте независимо от формы
  комнаты — нехватки стены «по площади» не бывает, не хватает только горизонтали (см. широкий
  объект ниже).

## Решения по краевым случаям

- **Объект шире самого длинного бокса.** Кладём его в **самый большой (длинный) бокс** как есть.
  (Никакого ужатия/сайзинга комнаты — горизонталь не наращиваем, вертикаль и так бесконечна.)
- **Высокий объект.** Бокс авто-раздвигается под его высоту (шаг 3/4) — обрезания нет.
- **Переполнение по количеству.** Боксы растут вверх до тех пор, пока всё не уместится (шаг 4);
  жёсткого капа нет.

## Ограничения реализации (v1)

- **Заполнение боксов последовательное, без возврата.** Если объект не лезет по ширине в текущий
  бокс даже отдельным рядом, переходим к следующему боксу и в текущий больше не возвращаемся —
  последующие (более узкие) объекты тоже идут в боксы дальше, оставляя ряды текущего пустыми.
  Для типичной комнаты (единицы объектов) потеря места мала; можно улучшить поиском подходящего
  бокса под каждый объект.
- **Комнаты без маршрутов** (нет боксов) используют **запасную поклеточную раскладку**
  (`_place_objects_fallback`, как раньше) с базовой высотой стен.
